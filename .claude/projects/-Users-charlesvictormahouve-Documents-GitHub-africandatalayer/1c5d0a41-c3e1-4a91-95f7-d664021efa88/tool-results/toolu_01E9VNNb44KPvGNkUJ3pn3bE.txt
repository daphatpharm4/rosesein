    23→## 1. Universal Data Model
    24→
    25→### 1.1 Entity Hierarchy
    26→
    27→The data model extends ADL's existing event-sourced architecture (`point_events` -> projected state -> snapshots -> deltas) with formalized entity layers.
    28→
    29→```
    30→                         +------------------+
    31→                         |      Zone        |
    32→                         | (Bonamoussadi)   |
    33→                         +--------+---------+
    34→                                  |
    35→                          1       |      *
    36→                         +--------v---------+
    37→                         |    Location      |
    38→                         | (physical place) |
    39→                         +--------+---------+
    40→                                  |
    41→                          1       |      *
    42→                         +--------v---------+
    43→                         |     Entity       |
    44→                         | (business/asset) |
    45→                         +--------+---------+
    46→                                  |
    47→                          1       |      *
    48→                         +--------v---------+         +----------+
    49→                         |   Observation    |-------->|  Media   |
    50→                         | (field visit)    |  1   *  | (photos) |
    51→                         +--------+---------+         +----------+
    52→                                  |
    53→                          2       |      *    (requires pair)
    54→                         +--------v---------+
    55→                         |     Delta        |
    56→                         | (computed diff)  |
    57→                         +---------+--------+
    58→                                   |
    59→                          *        |      1
    60→                         +---------v--------+
    61→                         |     Agent        |
    62→                         | (field worker)   |
    63→                         +------------------+
    64→```
    65→
    66→**Key relationships:**
    67→- A Zone contains many Locations (a geofenced area like Bonamoussadi)
    68→- A Location hosts one or more Entities (a GPS coordinate can have a pharmacy AND a mobile money kiosk)
    69→- An Entity has many Observations over time (each field visit produces one Observation)
    70→- An Observation captures one or more Media items (photos, videos)
    71→- A Delta is computed from exactly two Observations of the same Entity (before/after)
    72→- An Agent (field worker) produces many Observations
    73→
    74→**Alignment with existing code:**
    75→- `point_events` table = Observation events (CREATE_EVENT, ENRICH_EVENT)
    76→- `ProjectedPoint` = the current materialized state of an Entity
    77→- `snapshots` table = frozen Entity state at a point in time
    78→- `snapshot_deltas` table = computed Deltas
    79→- `user_profiles` table = Agent profiles
    80→
    81→### 1.2 Core Entity Schemas
    82→
    83→#### 1.2.1 Zone
    84→
    85→Zones are geofenced operational areas. Bonamoussadi is the pilot zone.
    86→
    87→```sql
    88→CREATE TABLE IF NOT EXISTS zones (
    89→  id text PRIMARY KEY,                         -- e.g., 'bonamoussadi'
    90→  name_en text NOT NULL,                       -- 'Bonamoussadi'
    91→  name_fr text NOT NULL,                       -- 'Bonamoussadi'
    92→  country text NOT NULL DEFAULT 'CM',          -- ISO 3166-1 alpha-2
    93→  city text NOT NULL,                          -- 'Douala'
    94→  arrondissement text,                         -- 'Douala V'
    95→  bounds_south double precision NOT NULL,      -- 4.0755
    96→  bounds_west double precision NOT NULL,       -- 9.7185
    97→  bounds_north double precision NOT NULL,      -- 4.0999
    98→  bounds_east double precision NOT NULL,       -- 9.7602
    99→  center_latitude double precision NOT NULL,   -- 4.0877
   100→  center_longitude double precision NOT NULL,  -- 9.7394
   101→  timezone text NOT NULL DEFAULT 'Africa/Douala', -- IANA timezone
   102→  status text NOT NULL DEFAULT 'active'
   103→    CHECK (status IN ('active', 'planned', 'archived')),
   104→  created_at timestamptz NOT NULL DEFAULT now(),
   105→  updated_at timestamptz NOT NULL DEFAULT now()
   106→);
   107→```
   108→
   109→**Note:** This formalizes what already exists in `shared/geofence.ts` as `BONAMOUSSADI_BOUNDS` and `BONAMOUSSADI_CENTER`. The existing constants remain as code-level shortcuts for the pilot zone.
   110→
   111→#### 1.2.2 Location
   112→
   113→A Location is a physical place on the ground, identified by GPS coordinates. Multiple entities can share one location (e.g., a pharmacy next door to a mobile money kiosk in the same building).
   114→
   115→```sql
   116→CREATE TABLE IF NOT EXISTS locations (
   117→  id text PRIMARY KEY,                          -- 'loc_0192a3b4c5d6' (see ID system below)
   118→  zone_id text NOT NULL REFERENCES zones(id),
   119→  latitude double precision NOT NULL,
   120→  longitude double precision NOT NULL,
   121→  geohash text NOT NULL,                        -- geohash-6 for proximity queries (~1.2km precision)
   122→  geohash_precise text NOT NULL,                -- geohash-9 for dedup (~4.8m precision)
   123→  altitude_m double precision,                  -- meters above sea level (if available)
   124→  gps_accuracy_m double precision,              -- horizontal accuracy in meters from device
   125→  landmark_description text,                    -- 'Behind Total station, facing the pharmacy'
   126→  landmark_description_fr text,                 -- French equivalent
   127→  road_name text,                               -- informal or formal road name
   128→  quarter text,                                 -- sub-neighborhood: 'Denver', 'Tradex side'
   129→  created_at timestamptz NOT NULL DEFAULT now(),
   130→  updated_at timestamptz NOT NULL DEFAULT now(),
   131→  created_by_agent text                         -- agent who first created this location
   132→);
   133→
   134→CREATE INDEX idx_locations_zone ON locations (zone_id);
   135→CREATE INDEX idx_locations_geohash ON locations (geohash);
   136→CREATE INDEX idx_locations_geohash_precise ON locations (geohash_precise);
   137→CREATE INDEX idx_locations_coords ON locations
   138→  USING gist (point(longitude, latitude));
   139→```
   140→
   141→**Example row:**
   142→```json
   143→{
   144→  "id": "loc_0192a3b4c5d6",
   145→  "zone_id": "bonamoussadi",
   146→  "latitude": 4.0891,
   147→  "longitude": 9.7412,
   148→  "geohash": "s10tqk",
   149→  "geohash_precise": "s10tqk4w7",
   150→  "gps_accuracy_m": 8.5,
   151→  "landmark_description": "Next to Tradex fuel station, opposite the pharmacy de garde",
   152→  "landmark_description_fr": "A cote de la station Tradex, en face de la pharmacie de garde",
   153→  "road_name": "Rue de Denver",
   154→  "quarter": "Denver"
   155→}
   156→```

   158→#### 1.2.3 Entity
   159→
   160→An Entity is a business, asset, or infrastructure point at a Location. This maps directly to the existing `ProjectedPoint` concept and the `point_events.point_id` grouping.
   161→
   162→```sql
   163→CREATE TABLE IF NOT EXISTS entities (
   164→  id text PRIMARY KEY,                          -- 'ent_pharmacy_0192a3b4c5d6'
   165→  location_id text NOT NULL REFERENCES locations(id),
   166→  zone_id text NOT NULL REFERENCES zones(id),
   167→  vertical text NOT NULL,                       -- 'pharmacy', 'fuel_station', etc.
   168→  name text,                                    -- 'Pharmacie du Bonheur'
   169→  formal_status text DEFAULT 'unknown'
   170→    CHECK (formal_status IN ('formal', 'informal', 'unknown')),
   171→  operational_status text DEFAULT 'active'
   172→    CHECK (operational_status IN ('active', 'closed_temporary', 'closed_permanent', 'unknown')),
   173→  first_observed_at timestamptz NOT NULL,
   174→  last_observed_at timestamptz NOT NULL,
   175→  observation_count integer NOT NULL DEFAULT 1,
   176→  details jsonb NOT NULL DEFAULT '{}',          -- vertical-specific fields (see extensions)
   177→  gaps text[] DEFAULT '{}',                     -- missing enrichable fields
   178→  confidence_score numeric(3,2) DEFAULT 0.50,   -- 0.00 to 1.00
   179→  photo_url text,
   180→  source text,                                  -- 'field_agent', 'osm_import', 'seed_data'
   181→  external_id text,                             -- ID in external system
   182→  canonical_entity_id text,                     -- points to merged entity if deduplicated
   183→  is_canonical boolean NOT NULL DEFAULT true,   -- false if merged into another entity
   184→  created_at timestamptz NOT NULL DEFAULT now(),
   185→  updated_at timestamptz NOT NULL DEFAULT now(),
   186→  CONSTRAINT valid_vertical CHECK (vertical IN (
   187→    'pharmacy', 'fuel_station', 'mobile_money',
   188→    'alcohol_outlet', 'billboard', 'transport_road',
   189→    'census_ground_truth'
   190→  ))
   191→);
   192→
   193→CREATE INDEX idx_entities_zone_vertical ON entities (zone_id, vertical);
   194→CREATE INDEX idx_entities_location ON entities (location_id);
   195→CREATE INDEX idx_entities_vertical_status ON entities (vertical, operational_status);
   196→CREATE INDEX idx_entities_canonical ON entities (canonical_entity_id) WHERE canonical_entity_id IS NOT NULL;
   197→CREATE INDEX idx_entities_external ON entities (external_id) WHERE external_id IS NOT NULL;
   198→```
   199→
   200→**Alignment with existing code:** This formalizes what `projectPointsFromEvents()` in `/lib/server/pointProjection.ts` computes at runtime. The existing `point_events` table remains the source-of-truth event log; `entities` is the materialized projection (currently computed on-the-fly as `ProjectedPoint`).
   201→
   202→#### 1.2.4 Observation
   203→
   204→An Observation is a single field visit or data capture event. This maps directly to the existing `point_events` table with additional quality metadata.
   205→
   206→```sql
   207→-- The existing point_events table IS the observation store.
   208→-- Below documents the extended schema for the data acquisition system.
   209→
   210→CREATE TABLE IF NOT EXISTS observations (
   211→  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
   212→  entity_id text NOT NULL REFERENCES entities(id),
   213→  agent_id text NOT NULL REFERENCES agents(id),
   214→  zone_id text NOT NULL REFERENCES zones(id),
   215→  vertical text NOT NULL,
   216→
   217→  -- Event classification
   218→  event_type text NOT NULL
   219→    CHECK (event_type IN ('CREATE_EVENT', 'ENRICH_EVENT', 'VERIFY_EVENT', 'CLOSE_EVENT')),
   220→  observation_type text NOT NULL DEFAULT 'field_visit'
   221→    CHECK (observation_type IN ('field_visit', 'remote_survey', 'import', 'automated')),
   222→
   223→  -- Location at time of observation (agent's GPS, may differ from entity location)
   224→  agent_latitude double precision NOT NULL,
   225→  agent_longitude double precision NOT NULL,
   226→  agent_gps_accuracy_m double precision,
   227→  agent_altitude_m double precision,
   228→
   229→  -- Captured data
   230→  details jsonb NOT NULL DEFAULT '{}',
   231→  photo_urls text[] DEFAULT '{}',
   232→  photo_count integer NOT NULL DEFAULT 0,
   233→
   234→  -- Temporal
   235→  observed_at timestamptz NOT NULL,             -- when the agent actually observed (from device clock)
   236→  device_clock_offset_ms integer,               -- estimated offset from server time
   237→  received_at timestamptz NOT NULL DEFAULT now(), -- when server received this observation
   238→  synced_at timestamptz,                        -- when offline observation was synced
   239→
   240→  -- Quality signals
   241→  gps_accuracy_m double precision,
   242→  observation_duration_s integer,               -- how long the agent spent at the location
   243→  is_offline_submission boolean NOT NULL DEFAULT false,
   244→
   245→  -- Fraud / validation
   246→  fraud_check jsonb,                            -- SubmissionFraudCheck object
   247→  client_device jsonb,                          -- ClientDeviceInfo object
   248→  validation_status text NOT NULL DEFAULT 'pending'
   249→    CHECK (validation_status IN ('pending', 'validated', 'rejected', 'needs_review')),
   250→  validated_by text,
   251→  validated_at timestamptz,
   252→
   253→  -- Lineage
   254→  source text DEFAULT 'field_agent',
   255→  external_id text,
   256→  idempotency_key text UNIQUE,
   257→  baseline_snapshot_id uuid,                    -- if this observation is part of a baseline survey
   258→
   259→  created_at timestamptz NOT NULL DEFAULT now()
   260→);
   261→
   262→CREATE INDEX idx_obs_entity_date ON observations (entity_id, observed_at DESC);
   263→CREATE INDEX idx_obs_agent_date ON observations (agent_id, observed_at DESC);
   264→CREATE INDEX idx_obs_zone_vertical_date ON observations (zone_id, vertical, observed_at DESC);
   265→CREATE INDEX idx_obs_validation ON observations (validation_status, created_at DESC);
   266→CREATE INDEX idx_obs_idempotency ON observations (idempotency_key) WHERE idempotency_key IS NOT NULL;
   267→```
   268→
   269→**Migration path:** The existing `point_events` table remains operational. The `observations` table extends it for the structured data acquisition system. A view or migration can bridge the two:
   270→
   271→```sql
   272→-- Bridge view: existing point_events exposed as observations
   273→CREATE OR REPLACE VIEW observations_compat AS
   274→SELECT
   275→  id,
   276→  point_id AS entity_id,
   277→  user_id AS agent_id,
   278→  'bonamoussadi' AS zone_id,
   279→  category AS vertical,
   280→  event_type,
   281→  'field_visit' AS observation_type,
   282→  latitude AS agent_latitude,
   283→  longitude AS agent_longitude,
   284→  NULL::double precision AS agent_gps_accuracy_m,
   285→  NULL::double precision AS agent_altitude_m,
   286→  details,
   287→  CASE WHEN photo_url IS NOT NULL THEN ARRAY[photo_url] ELSE '{}' END AS photo_urls,
   288→  CASE WHEN photo_url IS NOT NULL THEN 1 ELSE 0 END AS photo_count,
   289→  created_at AS observed_at,
   290→  NULL::integer AS device_clock_offset_ms,
   291→  created_at AS received_at,
   292→  NULL::timestamptz AS synced_at,
   293→  (details->>'confidenceScore')::numeric AS gps_accuracy_m,
   294→  NULL::integer AS observation_duration_s,
   295→  false AS is_offline_submission,
   296→  details->'fraudCheck' AS fraud_check,
   297→  details->'clientDevice' AS client_device,
   298→  'pending' AS validation_status,
   299→  NULL AS validated_by,
   300→  NULL::timestamptz AS validated_at,
   301→  source,
   302→  external_id,
   303→  NULL AS idempotency_key,
   304→  NULL::uuid AS baseline_snapshot_id,
   305→  created_at
   306→FROM point_events;
   307→```

   309→#### 1.2.5 Delta
   310→
   311→Deltas are computed differences between two observations of the same entity. This extends the existing `snapshot_deltas` table.
   312→
   313→```sql
   314→-- The existing snapshot_deltas table handles weekly aggregate deltas.
   315→-- This table adds observation-pair-level deltas for finer granularity.
   316→
   317→CREATE TABLE IF NOT EXISTS entity_deltas (
   318→  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
   319→  entity_id text NOT NULL REFERENCES entities(id),
   320→  zone_id text NOT NULL REFERENCES zones(id),
   321→  vertical text NOT NULL,
   322→
   323→  -- The two observations being compared
   324→  before_observation_id uuid NOT NULL REFERENCES observations(id),
   325→  after_observation_id uuid NOT NULL REFERENCES observations(id),
   326→  before_observed_at timestamptz NOT NULL,
   327→  after_observed_at timestamptz NOT NULL,
   328→  time_gap_hours numeric NOT NULL,              -- hours between observations
   329→
   330→  -- Delta classification
   331→  delta_type text NOT NULL
   332→    CHECK (delta_type IN ('new', 'removed', 'changed', 'unchanged', 'reopened')),
   333→  delta_field text,                             -- which field changed (null for new/removed)
   334→  previous_value text,
   335→  current_value text,
   336→  delta_magnitude numeric,                      -- numeric difference if applicable
   337→  delta_direction text
   338→    CHECK (delta_direction IN ('increase', 'decrease', 'stable', 'not_applicable')),
   339→  delta_summary text,                           -- human-readable summary
   340→
   341→  -- Significance scoring
   342→  severity text NOT NULL DEFAULT 'low'
   343→    CHECK (severity IN ('critical', 'high', 'medium', 'low', 'info')),
   344→  significance_score numeric(3,2) NOT NULL DEFAULT 0.50,  -- 0.00 to 1.00
   345→  confidence_score numeric(3,2) NOT NULL DEFAULT 0.50,    -- based on observation quality
   346→
   347→  -- Aggregation support
   348→  is_meaningful boolean NOT NULL DEFAULT true,  -- false for noise/trivial changes
   349→  snapshot_date date,                           -- links to weekly snapshot if applicable
   350→
   351→  created_at timestamptz NOT NULL DEFAULT now()
   352→);
   353→
   354→CREATE INDEX idx_deltas_entity_date ON entity_deltas (entity_id, after_observed_at DESC);
   355→CREATE INDEX idx_deltas_zone_vertical ON entity_deltas (zone_id, vertical, after_observed_at DESC);
   356→CREATE INDEX idx_deltas_type_severity ON entity_deltas (delta_type, severity, after_observed_at DESC);
   357→CREATE INDEX idx_deltas_snapshot ON entity_deltas (snapshot_date DESC, vertical) WHERE snapshot_date IS NOT NULL;
   358→```

   360→#### 1.2.6 Agent
   361→
   362→Agents are field workers who collect data. This extends the existing `user_profiles` table.
   363→
   364→```sql
   365→CREATE TABLE IF NOT EXISTS agents (
   366→  id text PRIMARY KEY,                          -- same as user_profiles.id
   367→  user_profile_id text NOT NULL REFERENCES user_profiles(id),
   368→  zone_id text NOT NULL REFERENCES zones(id),
   369→  agent_code text UNIQUE NOT NULL,              -- 'AGT-014' (human-readable)
   370→
   371→  -- Capabilities
   372→  assigned_verticals text[] NOT NULL DEFAULT '{}',  -- which verticals this agent covers
   373→  assigned_sectors text[] DEFAULT '{}',             -- sub-zones within Bonamoussadi
   374→  max_daily_observations integer DEFAULT 50,
   375→
   376→  -- Performance tracking
   377→  total_observations integer NOT NULL DEFAULT 0,
   378→  validated_observations integer NOT NULL DEFAULT 0,
   379→  rejected_observations integer NOT NULL DEFAULT 0,
   380→  reliability_score numeric(3,2) NOT NULL DEFAULT 0.50,  -- 0.00 to 1.00
   381→  avg_gps_accuracy_m numeric,
   382→  avg_observation_duration_s numeric,
   383→  last_active_at timestamptz,
   384→
   385→  -- Device info
   386→  primary_device_id text,                       -- from ClientDeviceInfo.deviceId
   387→  device_platform text,
   388→  is_low_end_device boolean DEFAULT false,
   389→
   390→  -- Status
   391→  status text NOT NULL DEFAULT 'active'
   392→    CHECK (status IN ('active', 'suspended', 'inactive', 'training')),
   393→
   394→  created_at timestamptz NOT NULL DEFAULT now(),
   395→  updated_at timestamptz NOT NULL DEFAULT now()
   396→);
   397→
   398→CREATE INDEX idx_agents_zone ON agents (zone_id, status);
   399→CREATE INDEX idx_agents_reliability ON agents (reliability_score DESC);
   400→```

   419→#### 1.2.7 Media

   423→```sql
   424→CREATE TABLE IF NOT EXISTS media (
   425→  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
   426→  observation_id uuid NOT NULL REFERENCES observations(id),
   427→  entity_id text NOT NULL REFERENCES entities(id),
   428→  agent_id text NOT NULL REFERENCES agents(id),
   429→
   430→  -- File info
   431→  media_type text NOT NULL CHECK (media_type IN ('photo', 'video', 'audio')),
   432→  url text NOT NULL,                            -- Vercel Blob URL
   433→  thumbnail_url text,
   434→  file_size_bytes integer,
   435→  mime_type text,                               -- 'image/jpeg', 'image/webp'
   436→  width_px integer,
   437→  height_px integer,
   438→
   439→  -- EXIF / metadata
   440→  exif_gps_latitude double precision,
   441→  exif_gps_longitude double precision,
   442→  exif_captured_at timestamptz,
   443→  exif_device_make text,
   444→  exif_device_model text,
   445→  exif_status text CHECK (exif_status IN ('ok', 'missing', 'parse_error', 'unsupported_format', 'fallback_recovered')),
   446→
   447→  -- Classification
   448→  purpose text NOT NULL DEFAULT 'evidence'
   449→    CHECK (purpose IN ('evidence', 'storefront', 'product', 'price_tag', 'signage', 'road_condition', 'billboard_creative')),
   450→  is_primary boolean NOT NULL DEFAULT false,
   451→
   452→  -- Quality
   453→  blur_score numeric(3,2),                      -- 0.00 (sharp) to 1.00 (blurry)
   454→  brightness_score numeric(3,2),                -- 0.00 (dark) to 1.00 (overexposed)
   455→
   456→  created_at timestamptz NOT NULL DEFAULT now()
   457→);
   458→
   459→CREATE INDEX idx_media_observation ON media (observation_id);
   460→CREATE INDEX idx_media_entity ON media (entity_id, created_at DESC);
   461→```

   463→### 1.3 Vertical-Specific Extensions
   464→
   465→Each vertical stores its specific fields in the `details` JSONB column of the Entity and Observation tables. Below are the required and enrichable fields per vertical.
   466→
   467→This directly extends the existing `VERTICALS` registry in `/shared/verticals.ts`.
   468→
   469→#### 1.3.1 Pharmacy
   470→
   471→```typescript
   472→interface PharmacyDetails {
   473→  // Required on create
   474→  name: string;                    // 'Pharmacie du Bonheur'
   475→  isOpenNow: boolean;              // true
   476→
   477→  // Enrichable
   478→  openingHours: string;            // '08:00-20:00'
   479→  isOnDuty: boolean;               // true = pharmacie de garde
   480→  phone: string;                   // '+237677123456'
   481→
   482→  // Extended fields for data acquisition
   483→  formalStatus: 'formal' | 'informal' | 'unknown';  // 'formal'
   484→  licenseNumber: string;           // 'PH-DLA-2024-0142'
   485→  ownerName: string;               // (optional, privacy-sensitive)
   486→  drugCategories: string[];        // ['prescription', 'otc', 'vitamins']
   487→  hasConsultationRoom: boolean;
   488→  hasColdChain: boolean;           // cold storage for vaccines/insulin
   489→  supplierSources: string[];       // ['CENAME', 'private_wholesaler']
   490→  stockoutItems: string[];         // items currently out of stock
   491→  nearestAlternativeId: string;    // entity_id of nearest open pharmacy
   492→}
   493→```
   494→
   495→#### 1.3.2 Fuel Station
   496→
   497→```typescript
   498→interface FuelStationDetails {
   499→  // Required on create
   500→  name: string;                    // 'Total Bonamoussadi'
   501→  hasFuelAvailable: boolean;
   502→
   503→  // Enrichable
   504→  fuelTypes: string[];             // ['super', 'diesel', 'kerosene']
   505→  pricesByFuel: Record<string, number>;  // { 'super': 730, 'diesel': 690 }
   506→  quality: string;                 // 'good', 'adulterated_suspected'
   507→  paymentMethods: string[];        // ['cash', 'mobile_money']
   508→  openingHours: string;
   509→
   510→  // Extended
   511→  operator: string;                // 'TotalEnergies'
   512→  brand: string;                   // 'Total'
   513→  pumpCount: number;               // 6
   514→  hasLPG: boolean;

   658→## 2. Entity ID System
   659→
   660→### 2.1 ID Generation Strategy
   661→
   662→ADL uses a **hybrid ID system** combining type-safety, temporal ordering, and offline generation.
   663→
   664→#### Format: Prefixed UUID v7
   665→
   666→```
   667→{type_prefix}_{uuid_v7}
   668→```
   669→
   670→**Examples:**
   671→```
   672→loc_019516a0-7c3a-7def-8a0b-1c2d3e4f5a6b    -- Location
   673→ent_019516a0-7c3a-7def-8a0b-1c2d3e4f5a6b    -- Entity
   674→obs_019516a0-7c3a-7def-8a0b-1c2d3e4f5a6b    -- Observation
   675→med_019516a0-7c3a-7def-8a0b-1c2d3e4f5a6b    -- Media
   676→dlt_019516a0-7c3a-7def-8a0b-1c2d3e4f5a6b    -- Delta
   677→```
   678→
   679→**Why UUID v7:**
   680→- **Time-ordered:** The first 48 bits encode a Unix timestamp in milliseconds. This means IDs sort chronologically in B-tree indexes without extra columns.
   681→- **Offline-safe:** Any device can generate a UUID v7 without server coordination. No sequence server, no ID range pre-allocation.
   682→- **Collision-resistant:** 74 bits of random data per millisecond. At 1,000 IDs/ms the collision probability is ~10^-23.
   683→- **Existing alignment:** The codebase already uses `crypto.randomUUID()` (UUID v4) in `offlineQueue.ts` and `gen_random_uuid()` in Postgres. UUID v7 is a drop-in replacement.
   684→
   685→**Implementation:**
   686→```typescript
   687→// lib/shared/ids.ts
   688→const PREFIXES = {
   689→  location: 'loc',
   690→  entity: 'ent',
   691→  observation: 'obs',
   692→  media: 'med',
   693→  delta: 'dlt',
   694→  agent: 'agt',
   695→} as const;
   696→
   697→type EntityType = keyof typeof PREFIXES;
   698→
   699→export function generateId(type: EntityType): string {
   700→  // UUID v7: timestamp (48 bits) + version (4 bits) + random (12 bits) + variant (2 bits) + random (62 bits)
   701→  const timestamp = Date.now();
   702→  const timestampHex = timestamp.toString(16).padStart(12, '0');
   703→
   704→  const randomBytes = new Uint8Array(10);
   705→  crypto.getRandomValues(randomBytes);
   706→
   707→  // Set version 7 (0111)
   708→  randomBytes[0] = (randomBytes[0] & 0x0f) | 0x70;
   709→  // Set variant 10xx
   710→  randomBytes[2] = (randomBytes[2] & 0x3f) | 0x80;
   711→
   712→  const hex = Array.from(randomBytes, b => b.toString(16).padStart(2, '0')).join('');
   713→
   714→  const uuid = [
   715→    timestampHex.slice(0, 8),
   716→    timestampHex.slice(8, 12),
   717→    hex.slice(0, 4),
   718→    hex.slice(4, 8),
   719→    hex.slice(8, 20),
   720→  ].join('-');
   721→
   722→  return `${PREFIXES[type]}_${uuid}`;
   723→}
   724→```
   725→
   726→#### Human-Readable Agent Codes
   727→
   728→Agents also get a sequential human-readable code for field operations:
   729→
   730→```
   731→AGT-{zone_abbrev}-{sequence}
   732→AGT-BON-001    -- first agent in Bonamoussadi
   733→AGT-BON-014    -- fourteenth agent
   734→```
   735→
   736→These are assigned server-side at registration and stored alongside the UUID-based `id`.
   737→
   738→### 2.2 Entity Deduplication
   739→
   740→The "same entity seen by two different agents" problem is critical in Bonamoussadi where no formal addresses exist. The system uses a **multi-signal deduplication pipeline**.
   741→
   742→#### Deduplication Signals
   743→
   744→```
   745→+------------------------------------------------------------------+
   746→|  Agent A submits new entity                                       |
   747→|  name: "Pharmacie du Soleil"                                     |
   748→|  location: 4.0892, 9.7413                                        |
   749→|  vertical: pharmacy                                               |
   750→+---------------------------+--------------------------------------+
   751→                            |
   752→                            v
   753→+---------------------------+--------------------------------------+
   754→|  STAGE 1: Geospatial proximity filter                            |
   755→|  - Find all entities of same vertical within 50m radius          |
   756→|  - Uses geohash_precise index (geohash-9 = ~4.8m cells)         |
   757→|  - If no candidates within 50m, this is a NEW entity             |
   758→+---------------------------+--------------------------------------+
   759→                            |
   760→              candidates found (e.g., 3 pharmacies within 50m)
   761→                            |
   762→                            v
   763→+---------------------------+--------------------------------------+
   764→|  STAGE 2: Name fuzzy matching                                    |
   765→|  - Normalize: lowercase, remove accents, remove common           |
   766→|    stopwords ('pharmacie', 'de', 'du', 'la', 'le', 'chez')      |
   767→|  - Jaro-Winkler similarity threshold >= 0.80                     |
   768→|  - Exact match on first meaningful word boosts score              |
   769→|                                                                   |
   770→|  "Pharmacie du Soleil" vs "Pharmacie Soleil" -> 0.95 (MATCH)    |
   771→|  "Pharmacie du Soleil" vs "Pharmacie de la Paix" -> 0.42 (NO)   |
   772→+---------------------------+--------------------------------------+
   773→                            |
   774→              1 candidate passes (Pharmacie Soleil at 12m away)
   775→                            |
   776→                            v
   777→+---------------------------+--------------------------------------+
   778→|  STAGE 3: Photo visual similarity (future enhancement)           |
   779→|  - Compare storefront photos using perceptual hash               |
   780→|  - Hamming distance < 10 = likely same entity                    |
   781→|  - Skipped if no photos available                                |
   782→+---------------------------+--------------------------------------+
   783→                            |
   784→                            v
   785→+---------------------------+--------------------------------------+
   786→|  STAGE 4: Merge decision                                         |
   787→|  - Combined dedup score:                                         |
   788→|    score = 0.50 * geo_proximity_score                            |
   789→|          + 0.35 * name_similarity_score                          |
   790→|          + 0.15 * photo_similarity_score                         |
   791→|                                                                   |
   792→|  - score >= 0.75: AUTO-MERGE (treat as ENRICH_EVENT)             |
   793→|  - 0.50 <= score < 0.75: FLAG for admin review                  |
   794→|  - score < 0.50: CREATE as new entity                            |
   795→+---------------------------+--------------------------------------+
   796→```
   797→
   798→#### Proximity Score Calculation
   799→
   800→```typescript
   801→function geoProximityScore(distanceMeters: number): number {
   802→  // 0m = 1.0, 10m = 0.9, 25m = 0.7, 50m = 0.3, >50m = 0
   803→  if (distanceMeters > 50) return 0;
   804→  return Math.max(0, 1 - (distanceMeters / 50) ** 0.7);
   805→}
   806→```
   807→
   808→#### Merge Handling
   809→
   810→When two entities are determined to be the same:
   811→
   812→```sql
   813→-- Mark the duplicate as non-canonical, pointing to the survivor
   814→UPDATE entities
   815→SET canonical_entity_id = 'ent_SURVIVOR_ID',
   816→    is_canonical = false,
   817→    updated_at = now()
   818→WHERE id = 'ent_DUPLICATE_ID';
   819→
   820→-- Re-point all observations from the duplicate to the survivor
   821→UPDATE observations
   822→SET entity_id = 'ent_SURVIVOR_ID'
   823→WHERE entity_id = 'ent_DUPLICATE_ID';
   824→```
   825→
   826→The merged entity retains the observation history from both agents, increasing confidence.
   827→
   828→### 2.3 Entity Identity Without Formal Addresses
   829→
   830→Since Bonamoussadi has no formal address system, entity identity is established through a **composite identity fingerprint**:
   831→
   832→```typescript
   833→interface EntityFingerprint {
   834→  vertical: string;            // 'pharmacy'
   835→  geohash9: string;            // 's10tqk4w7' (~4.8m precision)
   836→  normalizedName: string;      // 'soleil' (after stopword removal)
   837→  landmarkContext: string;     // 'near_tradex_station' (optional)
   838→}
   839→
   840→// Two entities are considered the same if:
   841→// 1. Same vertical
   842→// 2. Same geohash-9 OR adjacent geohash-9 cells
   843→// 3. Name similarity >= 0.80
   844→```

   848→## 3. Baseline vs. Snapshot Logic
   849→
   850→### 3.1 Definitions
   851→
   852→```
   853→+---------------------------------------------------------------------------+
   854→|  BASELINE                                                                  |
   855→|  A complete first survey of a defined geographic area.                     |
   856→|                                                                            |
   857→|  - Covers an entire zone or sector (e.g., "Denver quarter of             |
   858→|    Bonamoussadi")                                                          |
   859→|  - Establishes the initial inventory: what entities exist, where,         |
   860→|    and in what state                                                       |
   861→|  - Requires walking every street/alley in the sector                      |
   862→|  - Time-bounded: must be completed within a defined window                |
   863→|    (e.g., 7 days for a sector)                                            |
   864→|  - Produces: initial entity records + location records + first            |
   865→|    observations for each entity found                                     |
   866→+---------------------------------------------------------------------------+
   867→
   868→+---------------------------------------------------------------------------+
   869→|  SNAPSHOT                                                                  |
   870→|  A subsequent observation of an entity or set of entities.                 |
   871→|                                                                            |
   872→|  - Revisits entities already in the baseline                              |
   873→|  - Records current state (may have changed)                               |
   874→|  - Can be full-zone (revisit everything) or targeted (revisit             |
   875→|    specific entities or verticals)                                         |
   876→|  - Weekly cadence for most verticals (daily for fuel/mobile_money)        |
   877→|  - Produces: new observations + deltas computed against previous          |
   878→|    state                                                                   |
   879→+---------------------------------------------------------------------------+
   880→```
   881→
   882→### 3.2 Baseline Survey Structure
   883→
   884→```sql
   885→CREATE TABLE IF NOT EXISTS baseline_surveys (
   886→  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
   887→  zone_id text NOT NULL REFERENCES zones(id),
   888→  sector text,                                  -- sub-zone: 'Denver', 'Tradex side'
   889→  survey_name text NOT NULL,                    -- 'Bonamoussadi Phase 1 - Denver'
   890→
   891→  -- Planning
   892→  planned_start_date date NOT NULL,
   893→  planned_end_date date NOT NULL,
   894→  assigned_agent_ids text[] NOT NULL,
   895→  target_verticals text[] NOT NULL,             -- which verticals to survey
   896→
   897→  -- Execution tracking
   898→  status text NOT NULL DEFAULT 'planned'
   899→    CHECK (status IN ('planned', 'in_progress', 'completed', 'partial', 'cancelled')),
   900→  actual_start_date date,
   901→  actual_end_date date,
   902→
   903→  -- Coverage metrics
   904→  streets_planned integer,                      -- estimated streets to walk
   905→  streets_completed integer DEFAULT 0,
   906→  entities_found integer DEFAULT 0,
   907→  observations_created integer DEFAULT 0,
   908→
   909→  -- Completion criteria
   910→  minimum_coverage_pct numeric(5,2) DEFAULT 90.00,  -- sector must be >= 90% covered
   911→  is_baseline_complete boolean NOT NULL DEFAULT false,
   912→
   913→  created_at timestamptz NOT NULL DEFAULT now(),
   914→  updated_at timestamptz NOT NULL DEFAULT now()
   915→);
   916→```
   917→
   918→### 3.3 Partial Baseline Handling
   919→
   920→A baseline may be incomplete for several reasons:
   921→- Agent could not access certain streets (flooding, security)
   922→- Agent ran out of time/battery
   923→- Some entity types were missed (e.g., found pharmacies but missed informal alcohol vendors)
   924→
   925→**Handling strategy:**
   926→
   927→```
   928→+------------------------------------------------------------------+
   929→|  Baseline survey for Denver sector                                |
   930→|  Target: all 7 verticals                                         |
   931→|  Duration: 5 days (2026-03-10 to 2026-03-14)                     |
   932→+------------------------------------------------------------------+
   933→                       |
   934→    +------------------+------------------+
   935→    |                  |                  |
   936→    v                  v                  v
   937→  Day 1-3:          Day 4:             Day 5:
   938→  Streets 1-15      Streets 16-20      Streets 21-22
   939→  (all verticals)   (all verticals)    (pharmacy + fuel only,
   940→                                        agent battery died)
   941→    |                  |                  |
   942→    v                  v                  v
   943→  COVERED            COVERED           PARTIAL
   944→                                       (mark streets 21-22 as
   945→                                        partial_coverage for
   946→                                        alcohol, billboard,
   947→                                        mobile_money, transport,
   948→                                        census verticals)
   949→```
   950→
   951→```sql
   952→-- Track coverage at street/block level
   953→CREATE TABLE IF NOT EXISTS baseline_coverage (
   954→  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
   955→  baseline_survey_id uuid NOT NULL REFERENCES baseline_surveys(id),
   956→  sector text NOT NULL,
   957→  block_or_street text NOT NULL,               -- 'Street 21' or 'Block DEN-A14'
   958→  vertical text NOT NULL,
   959→  coverage_status text NOT NULL
   960→    CHECK (coverage_status IN ('complete', 'partial', 'not_visited')),
   961→  agent_id text,
   962→  visited_at timestamptz,
   963→  notes text,
   964→  created_at timestamptz NOT NULL DEFAULT now()
   965→);
   966→```
   967→
   968→### 3.4 Versioning Strategy for Entity State
   969→
   970→Entity state is versioned through the **event-sourced observation chain**, consistent with the existing `point_events` architecture.
   971→
   972→```
   973→Entity: ent_pharmacy_001 (Pharmacie du Soleil)
   974→
   975→Time ──────────────────────────────────────────────>
   976→
   977→  obs_001           obs_002           obs_003
   978→  (CREATE)          (ENRICH)          (ENRICH)
   979→  2026-03-10        2026-03-17        2026-03-24
   980→  |                 |                 |
   981→  v                 v                 v
   982→  State v1          State v2          State v3
   983→  isOpenNow: true   isOpenNow: true   isOpenNow: false
   984→  isOnDuty: null    isOnDuty: true    isOnDuty: false
   985→  phone: null       phone: +2376..    phone: +2376..
   986→  |                 |                 |
   987→  +----- delta -----+                 |
   988→  |      d1: isOnDuty null->true      |
   989→  |                 +----- delta -----+
   990→  |                 |      d2: isOpenNow true->false
   991→  |                 |      d3: isOnDuty true->false
   992→  |                 |
   993→  +-- snapshot S1 --+-- snapshot S2 --+-- snapshot S3 --
   994→  |  (week of 3/10) | (week of 3/17) | (week of 3/24) |
   995→```
   996→
   997→**Querying historical state:**
   998→
   999→The current `loadSnapshotAsMap()` function in `snapshotEngine.ts` already supports point-in-time queries by snapshot date. The entity state at any snapshot date is the frozen `details` JSONB from the `snapshots` table for that date.
  1000→
  1001→```sql
  1002→-- What was the state of all pharmacies on 2026-03-17?
  1003→SELECT point_id, site_name, details, gaps
  1004→FROM snapshots
  1005→WHERE snapshot_date = '2026-03-17'
  1006→  AND vertical_id = 'pharmacy'
  1007→ORDER BY site_name;
  1008→```
  1009→
  1010→**Immutability rule:** Observations are never modified after creation. If an agent made an error, a corrective ENRICH_EVENT is submitted rather than editing the original observation. This preserves the complete audit trail required by Law No. 2024/017.

  1014→## 4. Delta Computation Rules
  1015→
  1016→### 4.1 Vertical-Specific Delta Definitions
  1017→
  1018→The existing `detectChangedFields()` function in `/lib/server/snapshotEngine.ts` performs field-by-field JSON comparison with the `SKIP_FIELDS` set filtering out metadata. The rules below extend this with semantic meaning per vertical.
  1019→
  1020→#### 4.1.1 Pharmacy Deltas
  1021→
  1022→| Delta Event | Fields | Severity | Minimum Interval | Example |
  1023→|------------|--------|----------|-----------------|---------|
  1024→| Pharmacy opened/closed | `isOpenNow` | **critical** | 1 hour | Was open last week, now closed |
  1025→| On-duty status changed | `isOnDuty` | **high** | 4 hours | Became pharmacie de garde |
  1026→| New pharmacy appeared | (new entity) | **high** | N/A | New informal vendor detected |
  1027→| Pharmacy disappeared | (removed entity) | **critical** | 7 days | Not observed for 2+ weeks |
  1028→| Hours changed | `openingHours` | **medium** | 24 hours | Changed from 08-20 to 08-18 |
  1029→| Stockout reported | `stockoutItems` | **high** | 4 hours | Essential medicines unavailable |
  1030→| License status changed | `formalStatus` | **medium** | 7 days | Was formal, now informal |
  1031→
  1032→#### 4.1.2 Fuel Station Deltas
  1033→
  1034→| Delta Event | Fields | Severity | Minimum Interval | Example |
  1035→|------------|--------|----------|-----------------|---------|
  1036→| Price changed | `pricesByFuel.*` | **critical** | 1 hour | Super went from 730 to 750 XAF |
  1037→| Fuel availability changed | `hasFuelAvailable` | **critical** | 1 hour | Station ran out of diesel |
  1038→| Queue length changed | `queueLength` | **medium** | 30 minutes | Queue went from short to extreme |
  1039→| New fuel type added | `fuelTypes` | **low** | 7 days | Station now offers kerosene |
  1040→| Station closed | `operational_status` | **critical** | 4 hours | Temporary closure |
  1041→| Stock level changed | `stockLevel` | **high** | 4 hours | Full to low |
  1042→

  1171→### 4.4 Distinguishing Real Change from Observation Error
  1172→
  1173→This is critical: if Agent A records a pharmacy as "open" and Agent B records it as "closed" 30 minutes later, is the pharmacy actually closed, or did Agent B make an error?
  1174→
  1175→**Decision framework:**
  1176→
  1177→```
  1178→+------------------------------------------------------------------+
  1179→|  Delta observed: field X changed from value A to value B          |
  1180→+------+-----------------------------------------------------------+
  1181→       |
  1182→       v
  1183→  Is the before-observation confidence >= 0.70?
  1184→       |
  1185→   NO  |   YES
  1186→   |   |   |
  1187→   v   |   v
  1188→  Mark |  Is the after-observation confidence >= 0.70?
  1189→  as   |       |
  1190→  LOW  |   NO  |   YES
  1191→  CONF |   |   |   |
  1192→       |   v   |   v
  1193→       |  Mark |  Are both observations from the same agent?
  1194→       |  as   |       |
  1195→       |  LOW  |   YES |   NO
  1196→       |  CONF |   |   |   |
  1197→       |       |   v   |   v
  1198→       |       |  HIGH |  Was there a 3rd observation within
  1199→       |       |  CONF |  +/- 24 hours that confirms either A or B?
  1200→       |       |       |       |
  1201→       |       |       |   YES |   NO
  1202→       |       |       |   |   |   |
  1203→       |       |       |   v   |   v
  1204→       |       |       |  Use  |  Mark delta as
  1205→       |       |       |  the  |  'needs_verification'
  1206→       |       |       |  confirmed value
  1207→```
  1208→
  1209→```typescript
  1210→function classifyDeltaConfidence(
  1211→  beforeObs: Observation,
  1212→  afterObs: Observation,
  1213→  corroboratingObs: Observation | null,
  1214→): 'confirmed' | 'likely' | 'needs_verification' | 'likely_error' {
  1215→  const beforeConf = beforeObs.confidence_score ?? 0.50;
  1216→  const afterConf = afterObs.confidence_score ?? 0.50;
  1217→
  1218→  // Both low confidence = cannot trust the delta
  1219→  if (beforeConf < 0.50 && afterConf < 0.50) return 'likely_error';
  1220→
  1221→  // Same agent, both high confidence = they corrected themselves
  1222→  if (beforeObs.agent_id === afterObs.agent_id && beforeConf >= 0.70 && afterConf >= 0.70) {
  1223→    return 'confirmed';
  1224→  }
  1225→
  1226→  // Different agents, both high confidence
  1227→  if (beforeConf >= 0.70 && afterConf >= 0.70) {
  1228→    // Check for corroborating evidence
  1229→    if (corroboratingObs) return 'confirmed';
  1230→    return 'likely';
  1231→  }
  1232→
  1233→  // One high, one low
  1234→  if (afterConf >= 0.70 && beforeConf < 0.50) return 'likely'; // trust newer observation
  1235→  if (beforeConf >= 0.70 && afterConf < 0.50) return 'needs_verification';
  1236→
  1237→  return 'needs_verification';
  1238→}
  1239→```

  1293→## 5. Data Quality Scoring
  1294→
  1295→### 5.1 Per-Observation Confidence Score
  1296→
  1297→Each observation receives a confidence score from 0.00 to 1.00 based on measurable signals.
  1298→
  1299→```typescript
  1300→interface ObservationQualitySignals {
  1301→  gpsAccuracyM: number | null;        // device-reported horizontal accuracy
  1302→  photoCount: number;                   // number of evidence photos
  1303→  photoHasExif: boolean;                // at least one photo has valid EXIF
  1304→  photoGpsMatchesSubmission: boolean;   // EXIF GPS within 1km of submission GPS
  1305→  agentReliabilityScore: number;        // agent's historical reliability (0-1)
  1306→  observationDurationS: number | null;  // time spent at location
  1307→  isOffline: boolean;                   // was this submitted offline?
  1308→  hasLandmarkDescription: boolean;      // agent provided landmark context
  1309→  deviceClockDriftMs: number | null;    // estimated clock offset from server
  1310→}
  1311→
  1312→function computeObservationConfidence(signals: ObservationQualitySignals): number {
  1313→  let score = 0.50; // base score
  1314→
  1315→  // GPS accuracy: +0.15 for <= 10m, +0.10 for <= 25m, -0.10 for > 50m, -0.20 for null
  1316→  if (signals.gpsAccuracyM !== null) {
  1317→    if (signals.gpsAccuracyM <= 10) score += 0.15;
  1318→    else if (signals.gpsAccuracyM <= 25) score += 0.10;
  1319→    else if (signals.gpsAccuracyM <= 50) score += 0.00;
  1320→    else score -= 0.10;
  1321→  } else {
  1322→    score -= 0.20;
  1323→  }
  1324→
  1325→  // Photos: +0.10 per photo (max +0.20)
  1326→  score += Math.min(signals.photoCount * 0.10, 0.20);
  1327→
  1328→  // EXIF quality: +0.05 for valid EXIF, +0.05 if GPS matches
  1329→  if (signals.photoHasExif) score += 0.05;
  1330→  if (signals.photoGpsMatchesSubmission) score += 0.05;
  1331→
  1332→  // Agent reliability: -0.15 to +0.10 based on history
  1333→  score += (signals.agentReliabilityScore - 0.50) * 0.30;
  1334→
  1335→  // Observation duration: +0.05 for >= 60s (not a drive-by)
  1336→  if (signals.observationDurationS !== null && signals.observationDurationS >= 60) {
  1337→    score += 0.05;
  1338→  }
  1339→
  1340→  // Offline penalty: -0.05 (slightly less trustworthy, no real-time validation)
  1341→  if (signals.isOffline) score -= 0.05;
  1342→
  1343→  // Landmark description bonus: +0.05
  1344→  if (signals.hasLandmarkDescription) score += 0.05;
  1345→
  1346→  // Clock drift penalty: -0.05 if drift > 5 minutes
  1347→  if (signals.deviceClockDriftMs !== null && Math.abs(signals.deviceClockDriftMs) > 300000) {
  1348→    score -= 0.05;
  1349→  }
  1350→
  1351→  // Clamp to [0.05, 1.00]
  1352→  return Math.max(0.05, Math.min(1.00, Math.round(score * 100) / 100));
  1353→}
  1354→```
  1355→
  1356→**Example scores:**
  1357→
  1358→| Scenario | GPS | Photos | EXIF Match | Agent Score | Duration | Result |
  1359→|----------|-----|--------|------------|-------------|----------|--------|
  1360→| Good field visit | 8m | 2 | Yes | 0.85 | 120s | **0.90** |
  1361→| Rushed drive-by | 45m | 0 | N/A | 0.60 | 5s | **0.43** |
  1362→| Offline submission | null | 1 | No | 0.70 | 90s | **0.46** |
  1363→| New untrusted agent | 12m | 1 | Yes | 0.30 | 60s | **0.59** |
  1364→| Verified import | N/A | 0 | N/A | 1.00 | N/A | **0.65** |

  1366→### 5.2 Per-Entity Confidence Score
  1367→
  1368→Entity confidence aggregates across all observations of that entity.
  1369→
  1370→```typescript
  1371→function computeEntityConfidence(
  1372→  observations: Array<{ confidence: number; observedAt: Date }>,
  1373→  lastObservedAt: Date,
  1374→): number {
  1375→  if (observations.length === 0) return 0.10;
  1376→
  1377→  // Recency weight: more recent observations count more
  1378→  const now = Date.now();
  1379→  const weightedScores = observations.map(obs => {
  1380→    const ageHours = (now - obs.observedAt.getTime()) / (1000 * 60 * 60);
  1381→    const recencyWeight = Math.exp(-ageHours / (24 * 30)); // half-life of ~30 days
  1382→    return { score: obs.confidence, weight: recencyWeight };
  1383→  });
  1384→
  1385→  const totalWeight = weightedScores.reduce((sum, ws) => sum + ws.weight, 0);
  1386→  const weightedAvg = weightedScores.reduce((sum, ws) => sum + ws.score * ws.weight, 0) / totalWeight;
  1387→
  1388→  // Observation count bonus: more observations = higher confidence
  1389→  // Diminishing returns: 1 obs = +0, 2 obs = +0.05, 5 obs = +0.10, 10+ obs = +0.15
  1390→  const countBonus = Math.min(0.15, Math.log2(observations.length) * 0.05);
  1391→
  1392→  // Staleness penalty: if last observation is > 30 days old, reduce confidence
  1393→  const daysSinceLastObs = (now - lastObservedAt.getTime()) / (1000 * 60 * 60 * 24);
  1394→  const stalenessPenalty = daysSinceLastObs > 30
  1395→    ? Math.min(0.30, (daysSinceLastObs - 30) / 180 * 0.30) // loses 0.30 over 6 months
  1396→    : 0;
  1397→
  1398→  // Consistency bonus: if all observations agree on key fields, +0.05
  1399→  // (implementation checks if the latest 3 observations have same core values)
  1400→
  1401→  return Math.max(0.05, Math.min(1.00,
  1402→    Math.round((weightedAvg + countBonus - stalenessPenalty) * 100) / 100
  1403→  ));
  1404→}
  1405→```
  1406→
  1407→### 5.3 Per-Delta Confidence Score
  1408→
  1409→A delta is only as trustworthy as the observations it is computed from.
  1410→
  1411→```typescript
  1412→function computeDeltaConfidence(
  1413→  beforeObservation: { confidence: number },
  1414→  afterObservation: { confidence: number },
  1415→  classification: 'confirmed' | 'likely' | 'needs_verification' | 'likely_error',
  1416→): number {
  1417→  // Base: geometric mean of the two observation scores
  1418→  const base = Math.sqrt(beforeObservation.confidence * afterObservation.confidence);
  1419→
  1420→  // Classification adjustment
  1421→  const classificationMultiplier: Record<string, number> = {
  1422→    'confirmed': 1.10,
  1423→    'likely': 1.00,
  1424→    'needs_verification': 0.75,
  1425→    'likely_error': 0.40,
  1426→  };
  1427→
  1428→  const adjusted = base * (classificationMultiplier[classification] ?? 1.00);
  1429→  return Math.max(0.05, Math.min(1.00, Math.round(adjusted * 100) / 100));
  1430→}
  1431→```

  1433→### 5.4 Agent Reliability Score
  1434→
  1435→Agent reliability is computed from their historical accuracy, updated after each admin validation.
  1436→
  1437→```typescript
  1438→function updateAgentReliability(agent: Agent, validationResult: 'validated' | 'rejected'): number {
  1439→  const total = agent.validated_observations + agent.rejected_observations + 1;
  1440→  const validated = agent.validated_observations + (validationResult === 'validated' ? 1 : 0);
  1441→  const rejected = agent.rejected_observations + (validationResult === 'rejected' ? 1 : 0);
  1442→
  1443→  // Base accuracy rate
  1444→  const accuracyRate = validated / total;
  1445→
  1446→  // Apply Bayesian smoothing (prior: 50% accuracy, weight = 5 virtual observations)
  1447→  const priorWeight = 5;
  1448→  const smoothedRate = (validated + priorWeight * 0.50) / (total + priorWeight);
  1449→
  1450→  // Bonus for consistency: if last 10 are all validated, +0.05
  1451→  // Penalty for recent rejections: if 2+ of last 5 rejected, -0.10
  1452→
  1453→  return Math.max(0.10, Math.min(0.99, Math.round(smoothedRate * 100) / 100));
  1454→}
  1455→```

  1459→## 6. Temporal Model
  1460→
  1461→### 6.1 Time Representation
  1462→
  1463→**Standard: UTC storage with Africa/Douala (WAT, UTC+1) for display.**
  1464→
  1465→```
  1466→+------------------------------------------------------------------+
  1467→|  All timestamps stored as:  timestamptz  (PostgreSQL)             |
  1468→|  All timestamps in API:     ISO 8601 with timezone offset         |
  1469→|                              2026-03-03T14:30:00+01:00            |
  1470→|  All timestamps in code:    Date objects (UTC internally)         |
  1471→|  Display timezone:          Africa/Douala (UTC+1, no DST)         |
  1472→+------------------------------------------------------------------+
  1473→```
  1474→
  1475→**Rationale:**
  1476→- Cameroon observes West Africa Time (WAT) year-round. No daylight saving time changes.
  1477→- UTC+1 is fixed, making timezone handling simple.
  1478→- `timestamptz` in PostgreSQL stores the absolute moment; display conversion is a presentation concern.
  1479→
  1480→**Every observation stores three timestamps:**
  1481→
  1482→| Field | Purpose | Source |
  1483→|-------|---------|--------|
  1484→| `observed_at` | When the agent physically observed the entity | Device clock |
  1485→| `received_at` | When the server received the data | Server clock (`DEFAULT now()`) |
  1486→| `synced_at` | When an offline observation was synced | Server clock at sync time |
  1487→
  1488→### 6.2 As-Of Queries
  1489→
  1490→"What was the state of all Bonamoussadi billboards on 2026-03-17?"
  1491→
  1492→This is already supported by the `snapshots` table. The query:
  1493→
  1494→```sql
  1495→-- Point-in-time state reconstruction
  1496→SELECT
  1497→  s.point_id AS entity_id,
  1498→  s.site_name,
  1499→  s.latitude,
  1500→  s.longitude,
  1501→  s.details,
  1502→  s.gaps,
  1503→  s.events_count
  1504→FROM snapshots s
  1505→WHERE s.snapshot_date = (
  1506→  -- Find the most recent snapshot on or before the requested date
  1507→  SELECT MAX(snapshot_date)
  1508→  FROM snapshots
  1509→  WHERE snapshot_date <= '2026-03-17'
  1510→    AND vertical_id = 'billboard'
  1511→)
  1512→AND s.vertical_id = 'billboard'
  1513→ORDER BY s.site_name;
  1514→```
  1515→
  1516→For intra-week queries (between snapshots), reconstruct from observations:
  1517→
  1518→```sql
  1519→-- Reconstruct entity state as of a specific timestamp (between snapshots)
  1520→-- Uses the latest observation for each entity on or before the target time
  1521→SELECT DISTINCT ON (o.entity_id)
  1522→  o.entity_id,
  1523→  e.name,
  1524→  o.details,
  1525→  o.observed_at
  1526→FROM observations o
  1527→JOIN entities e ON e.id = o.entity_id
  1528→WHERE e.vertical = 'billboard'
  1529→  AND e.zone_id = 'bonamoussadi'
  1530→  AND o.observed_at <= '2026-03-17T15:00:00+01:00'
  1531→  AND o.validation_status != 'rejected'
  1532→ORDER BY o.entity_id, o.observed_at DESC;
  1533→```

  1535→### 6.3 Clock Drift on Offline Mobile Devices
  1536→
  1537→Mobile devices in Bonamoussadi may have inaccurate clocks (wrong timezone, never synced with NTP, or manually set incorrectly). This is handled through **server-side drift estimation**.
  1538→
  1539→```
  1540→+------------------------------------------------------------------+
  1541→|  CLOCK DRIFT DETECTION                                            |
  1542→|                                                                   |
  1543→|  1. On app launch (online): client sends { clientTime: Date.now() }
  1544→|     Server computes: drift = serverTime - clientTime              |
  1545→|     Stored in session as device_clock_offset_ms                   |
  1546→|                                                                   |
  1547→|  2. On each submission (online or offline):                       |
  1548→|     - Client records observed_at from device clock                |
  1549→|     - Server records received_at from server clock                |
  1550→|     - If online: drift = received_at - observed_at (minus         |
  1551→|       network latency estimate of ~500ms)                         |
  1552→|     - If offline: use last known drift from session               |
  1553→|                                                                   |
  1554→|  3. Correction policy:                                            |
  1555→|     - |drift| < 5 minutes: no correction, store as-is            |
  1556→|     - 5 min <= |drift| < 24 hours: store raw, add corrected_at   |
  1557→|       field: corrected_at = observed_at + drift                   |
  1558→|     - |drift| >= 24 hours: likely wrong date, flag for review,    |
  1559→|       use received_at as fallback                                 |
  1560→+------------------------------------------------------------------+
  1561→```
  1562→
  1563→```typescript
  1564→interface ClockDriftResult {
  1565→  rawObservedAt: string;           // as reported by device
  1566→  correctedObservedAt: string;     // adjusted for drift
  1567→  driftMs: number;                 // estimated offset
  1568→  driftCategory: 'none' | 'minor' | 'major' | 'invalid';
  1569→  useFallback: boolean;            // true if drift is too large
  1570→}
  1571→
  1572→function estimateClockDrift(
  1573→  deviceObservedAt: Date,
  1574→  serverReceivedAt: Date,
  1575→  lastKnownDriftMs: number | null,
  1576→  isOffline: boolean,
  1577→): ClockDriftResult {
  1578→  const NETWORK_LATENCY_MS = 500;
  1579→  const MINOR_THRESHOLD_MS = 5 * 60 * 1000;      // 5 minutes
  1580→  const MAJOR_THRESHOLD_MS = 24 * 60 * 60 * 1000; // 24 hours
  1581→
  1582→  let driftMs: number;
  1583→  if (isOffline && lastKnownDriftMs !== null) {
  1584→    driftMs = lastKnownDriftMs;
  1585→  } else {
  1586→    driftMs = serverReceivedAt.getTime() - deviceObservedAt.getTime() - NETWORK_LATENCY_MS;
  1587→  }
  1588→
  1589→  const absDrift = Math.abs(driftMs);
  1590→  let category: ClockDriftResult['driftCategory'];
  1591→  let useFallback = false;
  1592→
  1593→  if (absDrift < MINOR_THRESHOLD_MS) {
  1594→    category = 'none';
  1595→  } else if (absDrift < MAJOR_THRESHOLD_MS) {
  1596→    category = 'minor';
  1597→  } else {
  1598→    category = 'invalid';
  1599→    useFallback = true;
  1600→  }
  1601→
  1602→  const corrected = useFallback
  1603→    ? serverReceivedAt.toISOString()
  1604→    : new Date(deviceObservedAt.getTime() + (category === 'none' ? 0 : driftMs)).toISOString();
  1605→
  1606→  return {
  1607→    rawObservedAt: deviceObservedAt.toISOString(),
  1608→    correctedObservedAt: corrected,
  1609→    driftMs,
  1610→    driftCategory: category,
  1611→    useFallback,
  1612→  };
  1613→}
  1614→```

  1616→### 6.4 Retention and Archival Policies
  1617→
  1618→```
  1619→+------------------------------------------------------------------+
  1620→|  DATA RETENTION POLICY                                            |
  1621→|  (Aligned with Law No. 2024/017 compliance deadline June 2026)   |
  1622→+------------------------------------------------------------------+
  1623→
  1624→  Tier 1: HOT DATA (PostgreSQL primary, Supabase)
  1625→  ------------------------------------------------
  1626→  - observations:        last 90 days
  1627→  - entities:            all (current state)
  1628→  - entity_deltas:       last 90 days
  1629→  - snapshots:           last 52 weeks (1 year)
  1630→  - snapshot_deltas:     last 52 weeks
  1631→  - snapshot_stats:      last 52 weeks
  1632→  - media URLs:          last 90 days (Vercel Blob)
  1633→  - agents:              all (active lifecycle)
  1634→
  1635→  Tier 2: WARM DATA (PostgreSQL archive partition / separate schema)
  1636→  ------------------------------------------------
  1637→  - observations:        90 days to 2 years (compressed, queryable)
  1638→  - entity_deltas:       90 days to 2 years
  1639→  - snapshots:           1 to 3 years (monthly aggregation)
  1640→  - media metadata:      retained indefinitely (URLs may expire)
  1641→  - media files:         moved to cold storage after 90 days
  1642→
  1643→  Tier 3: COLD DATA (object storage / export)
  1644→  ------------------------------------------------
  1645→  - observations:        > 2 years (exported to Parquet/CSV)
  1646→  - media files:         > 90 days (moved to cheaper storage)
  1647→  - full database dumps: monthly, encrypted, retained 5 years
  1648→
  1649→  Data Deletion (Law No. 2024/017 compliance)
  1650→  ------------------------------------------------
  1651→  - Agent personal data: deletable upon request (right to erasure)
  1652→  - Observation data:    anonymized (agent_id replaced with hash)
  1653→                         after agent deletion request
  1654→  - Entity data:         not personal data, retained indefinitely
  1655→  - Media containing faces: auto-blurred after 90 days (future)
  1656→```
  1657→
  1658→**Archival implementation:**
  1659→
  1660→```sql
  1661→-- Partition observations by month for efficient archival
  1662→-- (to be implemented when volume justifies it)
  1663→
  1664→-- Archive old observations (run monthly)
  1665→CREATE OR REPLACE FUNCTION archive_old_observations()
  1666→RETURNS void AS $$
  1667→BEGIN
  1668→  INSERT INTO observations_archive
  1669→  SELECT * FROM observations
  1670→  WHERE observed_at < now() - interval '90 days'
  1671→    AND id NOT IN (SELECT id FROM observations_archive);
  1672→
  1673→  DELETE FROM observations
  1674→  WHERE observed_at < now() - interval '90 days';
  1675→END;
  1676→$$ LANGUAGE plpgsql;
  1677→
  1678→-- Archive old snapshots (keep only weekly for > 1 year, monthly for > 3 years)
  1679→CREATE OR REPLACE FUNCTION compact_old_snapshots()
  1680→RETURNS void AS $$
  1681→BEGIN
  1682→  -- Keep one snapshot per month for data older than 1 year
  1683→  DELETE FROM snapshots
  1684→  WHERE snapshot_date < current_date - interval '1 year'
  1685→    AND snapshot_date NOT IN (
  1686→      SELECT DISTINCT ON (date_trunc('month', snapshot_date))
  1687→        snapshot_date
  1688→      FROM snapshots
  1689→      WHERE snapshot_date < current_date - interval '1 year'
  1690→      ORDER BY date_trunc('month', snapshot_date), snapshot_date DESC
  1691→    );
  1692→END;
  1693→$$ LANGUAGE plpgsql;
  1694→```

  1709→## Summary of Key Design Decisions
  1710→
  1711→| Decision | Choice | Rationale |
  1712→|----------|--------|-----------|
  1713→| ID system | Prefixed UUID v7 | Offline-safe, time-ordered, no coordination needed |
  1714→| Storage | PostgreSQL (Supabase) + JSONB for vertical details | Matches existing stack, flexible schema for 7 verticals |
  1715→| Event model | Event-sourced (observations are immutable) | Audit trail for Law No. 2024/017, no data loss |
  1716→| Deduplication | GPS proximity + name fuzzy matching + photo similarity | No formal addresses exist in Bonamoussadi |
  1717→| Delta computation | Field-level diff with severity scoring per vertical | Extends existing `snapshotEngine.ts` approach |
  1718→| Time handling | UTC storage, WAT display, server-side clock drift correction | Simple (no DST in Cameroon), handles offline devices |
  1719→| Confidence scoring | Multi-signal weighted score (GPS, photos, agent history, EXIF) | Transparent quality measurement |
  1720→| Snapshot cadence | Weekly (Mon 03:00 UTC) for aggregate, per-observation for granular | Matches existing cron; daily for fuel/mobile_money |
  1721→| Baseline surveys | Structured survey with coverage tracking per sector per vertical | Handles partial surveys explicitly |
  1722→| Retention | 90 days hot, 2 years warm, 5 years cold with anonymization | Law No. 2024/017 compliance |

  1726→## Migration Path from Current System
  1727→
  1728→The design is **additive** -- it does not break any existing tables or APIs. Implementation can proceed in phases:
  1729→
  1730→1. **Phase 1:** Add `zones` table, formalize Bonamoussadi as a zone. No code changes needed.
  1731→2. **Phase 2:** Add `agents` table, backfill from `user_profiles`. Add `baseline_surveys` table.
  1732→3. **Phase 3:** Add `locations` table with geohash indexes. Add deduplication pipeline.
  1733→4. **Phase 4:** Extend `point_events` -> `observations` (add columns via ALTER TABLE or create new table with bridge view).
  1734→5. **Phase 5:** Add `entity_deltas` table for observation-pair deltas. Extend `snapshotEngine.ts` with severity scoring.
  1735→6. **Phase 6:** Add `media` table. Migrate from single `photo_url` to multi-photo support.
  1736→7. **Phase 7:** Add confidence scoring to observation submission pipeline.
  1737→
  1738→Each phase can be deployed independently. The existing app continues to function at every step.
