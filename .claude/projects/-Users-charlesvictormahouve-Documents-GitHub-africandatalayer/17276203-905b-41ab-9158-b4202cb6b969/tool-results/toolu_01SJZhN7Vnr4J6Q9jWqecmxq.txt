    23‚Üí## 1. Service Boundaries
    24‚Üí
    25‚Üí### 1.1 Current Monolith Decomposition Analysis
    26‚Üí
    27‚ÜíADL today runs as a single Vercel deployment -- every API route, the React SPA, auth logic, fraud checking, photo upload, event projection, and leaderboard calculation all deploy together in one `vercel.json` routing config. This is appropriate for the current stage.
    28‚Üí
    29‚Üí**Current module map (logical boundaries that already exist in code):**
    30‚Üí
    31‚Üí```
    32‚Üí+-----------------------------------------------------------------+
    33‚Üí|  Vercel Deployment (monolith)                                    |
    34‚Üí|                                                                  |
    35‚Üí|  [AUTH MODULE]          [SUBMISSION MODULE]     [USER MODULE]    |
    36‚Üí|  api/auth/auth.ts       api/submissions/        api/user/        |
    37‚Üí|  api/auth/register.ts     index.ts (GET/POST)     index.ts       |
    38‚Üí|  api/auth/[...auth].ts    [id].ts (GET/PUT/DEL)                  |
    39‚Üí|  api/auth/callback/                                              |
    40‚Üí|  api/auth/signin/       [LEADERBOARD MODULE]    [HEALTH MODULE]  |
    41‚Üí|  lib/auth.ts            api/leaderboard/        api/health/      |
    42‚Üí|                           index.ts                index.ts       |
    43‚Üí|                                                                  |
    44‚Üí|  [STORAGE LAYER]        [PROJECTION ENGINE]     [FRAUD ENGINE]   |
    45‚Üí|  lib/server/storage/    lib/server/             lib/server/      |
    46‚Üí|    index.ts               pointProjection.ts      submissionFraud |
    47‚Üí|    postgresStore.ts                               .ts            |
    48‚Üí|    edgeConfigStore.ts   [ACCESS CONTROL]                         |
    49‚Üí|    edgeFallbackStore.ts lib/server/                              |
    50‚Üí|    types.ts               submissionAccess.ts                    |
    51‚Üí|                                                                  |
    52‚Üí|  [SHARED DOMAIN]        [CLIENT OFFLINE]                         |
    53‚Üí|  shared/types.ts        lib/client/                              |
    54‚Üí|  shared/geofence.ts       offlineQueue.ts                        |
    55‚Üí|  types.ts (UI types)      submissionSync.ts                      |
    56‚Üí+-----------------------------------------------------------------+
    57‚Üí```
    58‚Üí
    59‚Üí**Analysis of coupling and cohesion:**
    60‚Üí
    61‚Üí| Boundary | Cohesion | Coupling to others | Verdict |
    62‚Üí|----------|----------|-------------------|---------|
    63‚Üí| Auth module | High -- isolated JWT/cookie/OAuth logic | Low -- only `lib/auth.ts` used by other modules via `requireUser()` | Clean boundary, could split easily |
    64‚Üí| Submission module | Medium -- combines CRUD, photo upload, fraud, projection | High -- calls storage, fraud, projection, geofence, blob upload, IP geolocation, user profile XP update | Core domain, keep together for now |
    65‚Üí| Leaderboard module | High -- single GET aggregation | Medium -- reads all events, user profiles | Could be async/cached worker |
    66‚Üí| User module | High -- profile CRUD | Low -- only storage layer | Clean boundary |
    67‚Üí| Storage layer | High -- abstract store interface | Low -- only `pg` driver | Good abstraction; already supports driver swap |
    68‚Üí| Fraud engine | High -- EXIF parsing, haversine, photo fraud metadata | Medium -- called inline during POST submission | Candidate for async extraction |
    69‚Üí| Projection engine | High -- event folding, gap analysis | Low -- pure functions, no I/O | Excellent isolation |
    70‚Üí
    71‚Üí### 1.2 Proposed Service Boundaries
    72‚Üí
    73‚ÜíGiven that ADL is an early-stage startup with a small team, over-decomposition into microservices would be premature. The recommendation is a **modular monolith** with clear internal boundaries, preparing for extraction only when measured bottlenecks demand it.
    74‚Üí
    75‚Üí**Phase 1 (MVP, now -- 1K users): Keep as monolith**
    76‚Üí
    77‚ÜíNo changes to deployment topology. Enforce boundaries through:
    78‚Üí- TypeScript module interfaces (the `StorageStore` interface in `lib/server/storage/types.ts` is a good pattern)
    79‚Üí- No cross-module direct database access; always go through the storage abstraction
    80‚Üí- Fraud detection called inline during submission POST is acceptable at current scale
    81‚Üí
    82‚Üí**Phase 2 (1K -- 10K users): Extract async processing**
    83‚Üí
    84‚Üí```
    85‚Üí+--------------------+     +---------------------+
    86‚Üí|  ADL API Monolith  |     |  Async Workers      |
    87‚Üí|  (Vercel Functions) |---->|  (Vercel Cron/      |
    88‚Üí|                    |     |   QStash webhooks)   |
    89‚Üí|  - Auth            |     |                     |
    90‚Üí|  - Submissions     |     |  - Fraud scoring    |
    91‚Üí|  - User profiles   |     |  - Photo processing |
    92‚Üí|  - Health          |     |  - Leaderboard      |
    93‚Üí|  - Point projection|     |    recalculation    |
    94‚Üí|                    |     |  - XP recalculation  |
    95‚Üí+--------+-----------+     +----------+----------+
    96‚Üí         |                            |
    97‚Üí         +--------+-------------------+
    98‚Üí                  |
    99‚Üí         +--------v----------+
   100‚Üí         |  Supabase Postgres |
   101‚Üí         |  + Vercel Blob     |
   102‚Üí         +--------------------+
   103‚Üí```
   104‚Üí
   105‚ÜíExtraction triggers:
   106‚Üí- Fraud scoring (EXIF extraction + IP geolocation) adds > 200ms p95 to submission POST latency
   107‚Üí- Leaderboard computation (`getPointEvents()` loads ALL events) exceeds 500ms
   108‚Üí- Photo resize/optimize pipeline needed for bandwidth savings
   109‚Üí
   110‚Üí**Phase 3 (10K -- 100K users): Bounded context extraction**
   111‚Üí
   112‚ÜíOnly split if the monolith deployment exceeds Vercel function size limits or team grows to 3+ squads.
   113‚Üí
   114‚Üí| Service | Owns | Extract when |
   115‚Üí|---------|------|-------------|
   116‚Üí| Submission Service | point_events, photo upload, geofencing | Team split or function cold start > 3s |
   117‚Üí| Identity Service | user_profiles, auth, registration | Regulatory requirement (data residency per country) |
   118‚Üí| Analytics Service | leaderboard, contribution stats, projections | Computation time exceeds 1s or data consumers need independent SLA |
   119‚Üí| Fraud Service | fraud_signals, EXIF analysis, IP scoring | ML model integration or real-time scoring pipeline needed |
   120‚Üí
   121‚Üí### 1.3 Service Ownership Matrix
   122‚Üí
   123‚Üí| Module | Owner (Role) | On-call | Runbook location |
   124‚Üí|--------|-------------|---------|-----------------|
   125‚Üí| Auth (`api/auth/`, `lib/auth.ts`) | Full-stack lead | Primary | `/docs/runbooks/auth.md` |
   126‚Üí| Submissions (`api/submissions/`) | Full-stack lead | Primary | `/docs/runbooks/submissions.md` |
   127‚Üí| Storage (`lib/server/storage/`) | Full-stack lead | Primary | `/docs/runbooks/storage.md` |
   128‚Üí| Fraud (`lib/server/submissionFraud.ts`) | Full-stack lead (later: Fraud specialist) | Secondary | `/docs/runbooks/fraud.md` |
   129‚Üí| Offline queue (`lib/client/offlineQueue.ts`) | Frontend lead | Primary | `/docs/runbooks/offline.md` |
   130‚Üí| Infrastructure (Vercel, Supabase) | Cloud engineer | Primary | `/docs/runbooks/infra.md` |
   131‚Üí
   132‚Üí### 1.4 Decision Criteria: Split vs. Keep Together
   133‚Üí
   134‚Üí| Signal | Action |
   135‚Üí|--------|--------|
   136‚Üí| Two modules need different scaling profiles (e.g., reads vs. writes) | Consider split |
   137‚Üí| A module has a different deployment cadence | Consider split |
   138‚Üí| A module needs a different runtime (e.g., Python ML) | Extract |
   139‚Üí| Team size < 5 developers | Keep together |
   140‚Üí| Shared database tables between modules | Keep together |
   141‚Üí| Module adds > 200ms latency to critical path synchronously | Extract to async |
   142‚Üí| Module failure should not take down the whole system | Extract with circuit breaker |

   157‚Üí#### Versioning Strategy
   158‚Üí
   159‚Üí**Current state:** No API versioning exists. All endpoints are unversioned.
   160‚Üí
   161‚Üí**Recommended approach:** URL-prefix versioning when breaking changes are needed:
   162‚Üí
   163‚Üí```
   164‚Üí/api/v1/submissions    (future, when breaking change is introduced)
   165‚Üí/api/submissions       (current, treated as v0/unversioned)
   166‚Üí```
   167‚Üí
   168‚Üí**When to version:**
   169‚Üí- Removing a field from a response body
   170‚Üí- Changing the type of a response field
   171‚Üí- Removing an endpoint
   172‚Üí- Changing authentication scheme
   173‚Üí
   174‚Üí**When NOT to version (additive changes are safe):**
   175‚Üí- Adding a new field to a response body
   176‚Üí- Adding a new endpoint
   177‚Üí- Adding a new query parameter
   178‚Üí- Adding a new optional field to a request body

   375‚Üí
   376‚Üí| Field | Type | Required | Notes |
   377‚Üí|-------|------|----------|-------|
   378‚Üí| `eventType` | `"CREATE_EVENT"` \| `"ENRICH_EVENT"` | No (defaults to `CREATE_EVENT`) | |
   379‚Üí| `category` | `"pharmacy"` \| `"fuel_station"` \| `"mobile_money"` | Yes | Also accepts `PHARMACY`, `FUEL`, `MOBILE_MONEY` |
   380‚Üí| `location` | `{ latitude, longitude }` | No (falls back to photo GPS or IP) | |
   381‚Üí| `pointId` | string (UUID) | Required for `ENRICH_EVENT` | Target point to enrich |

   384‚Üí| `secondImageBase64` | string | No | Optional second photo |
   385‚Üí
   386‚Üí**Category-specific required fields for CREATE_EVENT:**
   387‚Üí
   388‚Üí| Category | Required fields (validated by `listCreateMissingFields`) |
   389‚Üí|----------|--------------------------------------------------------|
   390‚Üí| `pharmacy` | `name`, `isOpenNow` |
   391‚Üí| `mobile_money` | `providers` |
   392‚Üí| `fuel_station` | `name`, `hasFuelAvailable` |
   393‚Üí
   394‚Üí**Category-specific enrichable fields (validated by `ENRICHABLE_FIELDS`):**
   395‚Üí
   396‚Üí| Category | Enrichable fields |
   397‚Üí|----------|-------------------|
   398‚Üí| `pharmacy` | `openingHours`, `isOpenNow`, `isOnDuty` |
   399‚Üí| `mobile_money` | `merchantIdByProvider`, `paymentMethods`, `openingHours`, `providers` |
   400‚Üí| `fuel_station` | `fuelTypes`, `pricesByFuel`, `quality`, `paymentMethods`, `openingHours`, `hasFuelAvailable` |
   401‚Üí
   402‚Üí**Geofencing rules:**
   403‚Üí- Non-admin users: location must be within Bonamoussadi bounds (`shared/geofence.ts`)
   404‚Üí- Admin users: global scope allowed
   405‚Üí
   406‚Üí**Photo validation pipeline (synchronous in current POST handler):**
   407‚Üí1. Parse base64 data URI, validate MIME type (jpeg/png/webp/heic/heif)
   408‚Üí2. Validate size <= 8 MB (`MAX_SUBMISSION_IMAGE_BYTES`)
   409‚Üí3. Extract EXIF metadata (GPS, timestamp, device make/model)
   410‚Üí4. If photo has GPS: verify distance to submission location <= 1 km
   411‚Üí5. If photo has GPS + IP available: verify distance to IP <= 50 km
   412‚Üí6. Upload to Vercel Blob
   413‚Üí7. Build fraud check metadata and store in `details.fraudCheck`
   414‚Üí
   415‚Üí**Response 201:** Full `PointEvent` object
   416‚Üí**Response 400:** Various validation errors
   417‚Üí**Response 401:** `{ "error": "Unauthorized" }`
   418‚Üí**Response 500:** Photo storage failures
   419‚Üí
   420‚Üí**Current gap identified:** The `X-Idempotency-Key` header is sent by `lib/client/offlineQueue.ts` (line 233: `await sendFn(item.payload, { idempotencyKey: item.idempotencyKey })`) but **never checked server-side** in `api/submissions/index.ts`. This means replayed submissions after network failures create duplicate events.

   856‚Üí### 3.3 Event Sourcing Schema Evolution Strategy
   857‚Üí
   858‚ÜíThe `details` JSONB column in `point_events` is the primary extension point. This is a good design for a schema-on-read system.
   859‚Üí
   860‚Üí**Rules for evolving the event schema:**
   861‚Üí
   862‚Üí1. **Never remove fields from `details`.** Old events retain their original shape.
   863‚Üí2. **Add new fields with normalization in the projection layer.** See `normalizeDetailsForCategory()` in `pointProjection.ts` for the pattern: it handles aliases like `hasCashAvailable` -> `hasMin50000XafAvailable`, `hours` -> `openingHours`.
   864‚Üí3. **Version the event structure implicitly.** The presence/absence of fields indicates the version. If explicit versioning is needed, add a `_schemaVersion` field to `details`.
   865‚Üí4. **Handle missing columns gracefully.** The `phoneColumnState` pattern in `postgresStore.ts` (lines 8, 134-157) demonstrates runtime column detection. This is fragile but pragmatic for MVP. Replace with proper migrations in Phase 2.
   866‚Üí
   867‚Üí**Migration strategy for schema changes:**
   868‚Üí
   869‚Üí| Change type | Approach | Downtime |
   870‚Üí|-------------|----------|----------|
   871‚Üí| Add column to `user_profiles` | `ALTER TABLE ... ADD COLUMN` with `DEFAULT` | Zero |
   872‚Üí| Add column to `point_events` | `ALTER TABLE ... ADD COLUMN` with `DEFAULT NULL` | Zero |
   873‚Üí| Add new field to `details` JSONB | No schema change; update projection logic | Zero |
   874‚Üí| Rename a `details` field | Add alias in `normalizeDetailsForCategory()`; both old and new names work | Zero |
   875‚Üí| Add index | `CREATE INDEX CONCURRENTLY` | Zero |
   876‚Üí| Partition `point_events` | Create partitioned table, migrate data, swap | Maintenance window |
   877‚Üí| Change column type | Create new column, backfill, update code, drop old | Phased rollout |
   878‚Üí
   879‚Üí### 3.4 Read Model / Projection Design
   880‚Üí
   881‚Üí**Current projection architecture:**
   882‚Üí
   883‚Üí```
   884‚Üípoint_events (source of truth)
   885‚Üí    |
   886‚Üí    v
   887‚ÜíprojectPointsFromEvents()          -- lib/server/pointProjection.ts
   888‚Üí    |
   889‚Üí    |  1. Sort events by createdAt ASC
   890‚Üí    |  2. Group by pointId
   891‚Üí    |  3. For each group:
   892‚Üí    |     - First event initializes the ProjectedPoint
   893‚Üí    |     - Subsequent events merge details (last-write-wins per field)
   894‚Üí    |     - Location updates to latest event's location
   895‚Üí    |     - photoUrl updates to latest non-null
   896‚Üí    |     - gaps recalculated after each merge
   897‚Üí    |  4. Sort results by updatedAt DESC
   898‚Üí    |

   906‚Üí

   908‚ÜíCREATE TABLE projected_points (

   910‚Üí    category    TEXT NOT NULL,

   917‚Üí    gaps        TEXT[] NOT NULL DEFAULT '{}',

   924‚Üí    ST_SetSRID(ST_MakePoint(longitude, latitude), 4326)

  1045‚Üí## 5. Failure Handling Strategy
  1046‚Üí
  1047‚Üí### 5.1 Circuit Breaker Patterns
  1048‚Üí
  1049‚Üí**External services that can fail:**
  1050‚Üí
  1051‚Üí| Service | Used by | Failure mode | Circuit breaker strategy |
  1052‚Üí|---------|---------|-------------|------------------------|
  1053‚Üí| Supabase PostgreSQL | All API endpoints | Connection timeout, pool exhaustion | `StorageUnavailableError` (exists in `lib/server/db.ts`). Returns 503 with `code: "storage_unavailable"`. **Enhancement:** add circuit breaker state (closed/open/half-open) with 5 failures in 60s to open, 30s cooldown. |
  1054‚Üí| Vercel Blob | `POST /api/submissions` | Upload failure | Currently returns 500. **Enhancement:** retry once, then return 503 with client-retryable flag. |
  1055‚Üí| ipapi.co (IP geolocation) | `POST /api/submissions` | Timeout (3s), rate limit | Currently fails silently (returns `null`). This is correct -- location falls back to photo GPS or explicit `location`. |
  1056‚Üí| EXIF parsing (exifr) | `POST /api/submissions` | Corrupt image data | Currently caught with try/catch (line 513 of `api/submissions/index.ts`). Falls back to explicit location. Correct behavior. |
  1057‚Üí| Google OAuth | Auth flow | Google down | Auth.js handles this. Users can still use credentials login. |
  1058‚Üí
  1059‚Üí**Proposed circuit breaker implementation:**
  1060‚Üí
  1061‚Üí```typescript
  1062‚Üí// lib/server/circuitBreaker.ts (proposed)
  1063‚Üíinterface CircuitState {
  1064‚Üí  failures: number;
  1065‚Üí  lastFailure: number;
  1066‚Üí  state: 'closed' | 'open' | 'half-open';
  1067‚Üí}
  1068‚Üí
  1069‚Üíconst breakers = new Map<string, CircuitState>();
  1070‚Üíconst FAILURE_THRESHOLD = 5;
  1071‚Üíconst RESET_TIMEOUT_MS = 30_000;
  1072‚Üí
  1073‚Üífunction getBreaker(name: string): CircuitState {
  1074‚Üí  if (!breakers.has(name)) {
  1075‚Üí    breakers.set(name, { failures: 0, lastFailure: 0, state: 'closed' });
  1076‚Üí  }
  1077‚Üí  return breakers.get(name)!;
  1078‚Üí}
  1079‚Üí
  1080‚Üífunction shouldAllow(name: string): boolean {
  1081‚Üí  const breaker = getBreaker(name);
  1082‚Üí  if (breaker.state === 'closed') return true;
  1083‚Üí  if (breaker.state === 'open') {
  1084‚Üí    if (Date.now() - breaker.lastFailure > RESET_TIMEOUT_MS) {
  1085‚Üí      breaker.state = 'half-open';
  1086‚Üí      return true; // allow one probe
  1087‚Üí    }
  1088‚Üí    return false;
  1089‚Üí  }
  1090‚Üí  return true; // half-open: allow probe
  1091‚Üí}
  1092‚Üí```
  1093‚Üí
  1094‚Üí**Note:** In a serverless environment, circuit breaker state is per-instance and ephemeral. For cross-instance state, use Upstash Redis (Phase 2).
  1095‚Üí
  1096‚Üí### 5.2 Retry Policies
  1097‚Üí
  1098‚Üí**Current offline queue retry (from `lib/client/offlineQueue.ts`):**
  1099‚Üí
  1100‚Üí```
  1101‚ÜíflushOfflineQueue() (lines 204-287):
  1102‚Üí  - For each pending item:
  1103‚Üí    1. Skip if nextRetryAt > now (backoff not elapsed)
  1104‚Üí    2. Mark as 'syncing', increment attempts
  1105‚Üí    3. Call sendFn(payload, { idempotencyKey })
  1106‚Üí    4. On success: remove from IndexedDB
  1107‚Üí    5. On retryable error:
  1108‚Üí       - retryCount += 1
  1109‚Üí       - baseDelay = min(30000, 1000 * 2^retryCount)
  1110‚Üí       - jitter = random(0, 1000)
  1111‚Üí       - nextRetryAt = now + baseDelay + jitter
  1112‚Üí       - Mark as 'failed'
  1113‚Üí    6. On permanent error (retryable=false):
  1114‚Üí       - Record in SYNC_ERROR_STORE
  1115‚Üí       - Remove from queue
  1116‚Üí```
  1117‚Üí
  1118‚Üí**Analysis and improvements:**
  1119‚Üí
  1120‚Üí| Aspect | Current | Recommendation |
  1121‚Üí|--------|---------|---------------|
  1122‚Üí| Max retry delay | 30s cap | Increase to 5 minutes for extended offline |
  1123‚Üí| Max retries | Unlimited | Add max of 20 retries before permanent failure |
  1124‚Üí| Jitter | Random 0-1000ms | Good. Keep. |
  1125‚Üí| Backoff base | 1000ms * 2^n | Good exponential curve. |
  1126‚Üí| Permanent failure classification | Only `retryable: false` errors | Add: 400-level responses (except 408, 429) should be permanent |
  1127‚Üí| Queue size limit | Unbounded | Warn at 200, reject at 500 with user notification |
  1128‚Üí| Stale item cleanup | None | Remove items older than 7 days |
  1129‚Üí

  1132‚Üí| Service | Max retries | Backoff | Timeout |

  1181‚Üí### 5.4 Data Consistency Guarantees
  1182‚Üí
  1183‚Üí| Data type | Consistency model | Rationale |
  1184‚Üí|-----------|------------------|-----------|
  1185‚Üí| `point_events` (writes) | **Strong** -- single PostgreSQL writer with `ON CONFLICT` upsert | Append-only log must not lose events |
  1186‚Üí| `ProjectedPoint` (reads) | **Eventually consistent** -- up to 30s stale due to `s-maxage=30` cache | Acceptable for map display; contributors see their own submissions immediately (cache bypass on auth) |
  1187‚Üí| `user_profiles` (writes) | **Strong** -- single writer upsert | Profile updates are low-frequency |
  1188‚Üí| `user_profiles` (reads) | **Strong** -- no caching on authenticated reads | User sees own XP immediately |
  1189‚Üí| Leaderboard | **Eventually consistent** -- 30s cache + full recomputation | Leaderboard staleness is acceptable |
  1190‚Üí| Offline queue sync | **At-least-once delivery** (currently) | Idempotency gap means duplicates possible |
  1191‚Üí
  1192‚Üí**Critical gap: At-least-once without idempotency = duplicates**
  1193‚Üí
  1194‚ÜíThe offline queue generates an `idempotencyKey` per item (`lib/client/offlineQueue.ts` line 111) and sends it as a header, but the server never checks it. If a request succeeds but the client doesn't receive the response (network drop after server commit), the client retries and creates a duplicate event.
  1195‚Üí
  1196‚Üí**Fix (implement in Phase 1):**
  1197‚Üí
  1198‚Üí```sql
  1199‚ÜíCREATE TABLE idempotency_keys (
  1200‚Üí    key         TEXT PRIMARY KEY,
  1201‚Üí    user_id     TEXT NOT NULL,
  1202‚Üí    event_id    UUID NOT NULL REFERENCES point_events(id),
  1203‚Üí    response    JSONB NOT NULL,
  1204‚Üí    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  1205‚Üí    expires_at  TIMESTAMPTZ NOT NULL DEFAULT NOW() + INTERVAL '24 hours'
  1206‚Üí);
  1207‚Üí
  1208‚ÜíCREATE INDEX idx_idem_expires ON idempotency_keys (expires_at);
  1209‚Üí```
  1210‚Üí
  1211‚ÜíServer-side check in `POST /api/submissions`:
  1212‚Üí1. Read `X-Idempotency-Key` header
  1213‚Üí2. Look up in `idempotency_keys` table
  1214‚Üí3. If found and not expired: return cached response (201 with original event)
  1215‚Üí4. If not found: proceed with normal flow, store response after commit
  1216‚Üí5. Cron job: `DELETE FROM idempotency_keys WHERE expires_at < NOW()` daily
  1217‚Üí
  1218‚Üí### 5.5 Dead Letter Queue for Failed Async Operations

  1280‚Üí### 6.2 Cache Invalidation for Event-Sourced Data
  1281‚Üí
  1282‚ÜíThe event-sourced architecture makes cache invalidation straightforward because the source of truth (point_events) is append-only.
  1283‚Üí
  1284‚Üí**Current approach (acceptable for MVP):**
  1285‚Üí- 30-second `s-maxage` means new submissions appear within 30s for anonymous users
  1286‚Üí- Authenticated users bypass cache (`cache-control: no-store` is the default in `lib/server/http.ts`)
  1287‚Üí
  1288‚Üí**Phase 2 approach:**
  1289‚Üí1. **Write-through to projection table:** When `insertPointEvent()` succeeds, also upsert `projected_points`. API reads from `projected_points` directly instead of replaying all events.
  1290‚Üí2. **Revalidation on write:** After a successful POST to `/api/submissions`, the API could return a `X-Revalidate: /api/submissions` header that the edge layer uses to purge the cached response.
  1291‚Üí3. **Supabase Realtime for admin:** Subscribe to `point_events` INSERT via Supabase Realtime WebSocket. Admin dashboard updates in real-time without polling.
  1292‚Üí
  1293‚Üí**Stale data tolerance by consumer:**
  1294‚Üí
  1295‚Üí| Consumer | Staleness tolerance | Strategy |
  1296‚Üí|----------|-------------------|----------|
  1297‚Üí| Anonymous map viewer | 30s | Edge cache with `stale-while-revalidate` |
  1298‚Üí| Authenticated contributor (own data) | 0s | Cache bypass for authenticated requests |
  1299‚Üí| Admin dashboard | 5s | Supabase Realtime push (Phase 2) |
  1300‚Üí| Leaderboard | 60s | Increase `s-maxage` to 60 for leaderboard |
  1301‚Üí| Webhook consumers | < 10s | Event-driven push on write |
  1302‚Üí| Data export | Minutes | Batch job, not cached |
  1303‚Üí
  1304‚Üí### 6.3 Cache Warming for Projected Points
  1305‚Üí
  1306‚Üí**Problem:** First request after cache expiry (or cold start) must load all events and project them. This is the slowest path.
  1307‚Üí
  1308‚Üí**Phase 2 solution: Materialized projection table with warm reads**
  1309‚Üí
  1310‚Üí1. **On deploy:** Vercel build step or post-deploy hook triggers `/api/cron/rebuild-projections` to fully rebuild the `projected_points` table.
  1311‚Üí2. **On write:** Each `insertPointEvent()` incrementally updates the relevant `projected_points` row.
  1312‚Üí3. **GET /api/submissions:** Reads directly from `projected_points` table -- single indexed query, no full event replay.
  1313‚Üí4. **Fallback:** If `projected_points` is empty/missing, fall back to full event replay (current behavior).
  1314‚Üí
  1315‚Üí**Estimated latency improvement:**
  1316‚Üí- Current (full replay of 1000 events): ~200ms
  1317‚Üí- With materialized table (indexed query): ~20ms
  1318‚Üí- At 100K events: current would be ~5s; materialized stays at ~20ms
  1319‚Üí

  1323‚Üí
  1324‚Üí### 7.1 Offline Queue Sync Protocol
  1325‚Üí
  1326‚Üí**Current design (from `lib/client/offlineQueue.ts`):**
  1327‚Üí
  1328‚Üí```
  1329‚ÜíSTATE MACHINE PER QUEUE ITEM
  1330‚Üí==============================
  1331‚Üí
  1332‚Üí  +----------+     enqueue()     +---------+
  1333‚Üí  |  (none)  | ----------------> | PENDING |
  1334‚Üí  +----------+                   +----+----+
  1335‚Üí                                      |
  1336‚Üí                            flushOfflineQueue()
  1337‚Üí                                      |
  1338‚Üí                                 +----v-----+
  1339‚Üí                                 | SYNCING  |
  1340‚Üí                                 +----+-----+
  1341‚Üí                                      |
  1342‚Üí                          +-----------+-----------+
  1343‚Üí                          |                       |
  1344‚Üí                     sendFn() OK            sendFn() ERROR
  1345‚Üí                          |                       |
  1346‚Üí                    +-----v-----+          +------v------+
  1347‚Üí                    |  (removed  |          | retryable?  |
  1348‚Üí                    |  from IDB) |          +------+------+
  1349‚Üí                    +-----------+                  |
  1350‚Üí                                           +------+------+
  1351‚Üí                                           |             |
  1352‚Üí                                         Yes            No
  1353‚Üí                                           |             |
  1354‚Üí                                    +------v---+   +----v--------+
  1355‚Üí                                    |  FAILED  |   | (permanent  |
  1356‚Üí                                    | (w/ next |   |  failure    |
  1357‚Üí                                    |  RetryAt)|   |  recorded,  |
  1358‚Üí                                    +----------+   |  removed)   |
  1359‚Üí                                                   +-------------+
  1360‚Üí```
  1361‚Üí
  1362‚Üí**Improvements to current design:**
  1363‚Üí
  1364‚Üí1. **Add explicit MAX_RETRIES (20)**
  1365‚Üí   After 20 retries, move to permanent failure even if error is "retryable". The current design retries forever.
  1366‚Üí
  1367‚Üí2. **Add queue staleness detection**
  1368‚Üí   Items older than 7 days should be moved to permanent failure with reason "stale_queue_item".
  1369‚Üí
  1370‚Üí3. **Add queue size limit**
  1371‚Üí   Max 500 items. Beyond that, show user a warning and reject new enqueues until items are synced.
  1372‚Üí
  1373‚Üí4. **Add connectivity detection before flush**
  1374‚Üí   Before calling `flushOfflineQueue()`, check `navigator.onLine` AND attempt a lightweight fetch to `/api/health`. Only flush if both pass.
  1375‚Üí
  1376‚Üí5. **Add progress callback**
  1377‚Üí   `flushOfflineQueue()` should accept an `onProgress(synced, failed, total)` callback for UI updates.
  1378‚Üí
  1379‚Üí6. **Batch sync (Phase 2)**
  1380‚Üí   Instead of sending items one-by-one, batch up to 10 items in a single `POST /api/submissions/batch` request to reduce round trips on slow connections.
  1381‚Üí
  1382‚Üí### 7.2 Conflict Resolution Strategy
  1383‚Üí
  1384‚Üí**Conflict types in ADL:**
  1385‚Üí
  1386‚Üí| Conflict | Likelihood | Resolution |
  1387‚Üí|----------|-----------|------------|
  1388‚Üí| Two users CREATE the same POI | Medium | **Accept both** as separate points. Admin merges via point_id reassignment. |
  1389‚Üí| Two users ENRICH the same field on the same point | Low | **Last-write-wins** at the field level. The `mergeDetails()` function in `pointProjection.ts` (line 136) already implements this. |
  1390‚Üí| Offline queue replays create duplicate events | High (current gap) | **Idempotency key** (Section 5.4). Server returns cached response for duplicate key. |
  1391‚Üí| User submits enrichment for a gap that was already filled by someone else | Medium | Current: returns 400 "ENRICH_EVENT must include at least one currently missing field". **Improvement:** Accept the enrichment but store as a competing value. Admin or confidence scoring resolves. |
  1392‚Üí
  1393‚Üí**Conflict resolution for offline queue:**
  1394‚Üí
  1395‚Üí```
  1396‚ÜíOFFLINE CONFLICT SCENARIO
  1397‚Üí==========================
  1398‚Üí
  1399‚ÜíTime T0: User A offline, enqueues ENRICH for Point P (field: openingHours)
  1400‚ÜíTime T1: User B online, successfully enriches Point P (field: openingHours)
  1401‚ÜíTime T2: User A comes online, queue flushes
  1402‚Üí
  1403‚ÜíCurrent behavior: User A gets 400 error because openingHours is no longer a gap.
  1404‚ÜíThis goes to permanent failure in sync error store.
  1405‚Üí
  1406‚ÜíProposed behavior:
  1407‚Üí1. Server detects the field was already filled
  1408‚Üí2. Instead of 400, store as a "competing enrichment" event
  1409‚Üí3. Return 201 with a flag: { "competingFields": ["openingHours"] }
  1410‚Üí4. Admin dashboard shows competing values for review
  1411‚Üí5. Client removes item from queue (success)
  1412‚Üí```
