   271→**Migration runner script (`scripts/migrate.mjs`):**
   272→
   273→```javascript
   274→#!/usr/bin/env node
   275→// scripts/migrate.mjs
   276→// Usage: node scripts/migrate.mjs [--dry-run]
   277→
   278→import { readFileSync, readdirSync } from 'fs';
   279→import { join } from 'path';
   280→import pg from 'pg';
   281→
   282→const pool = new pg.Pool({
   283→  connectionString: process.env.ADL_POSTGRES_URL || process.env.POSTGRES_URL,
   284→  ssl: { rejectUnauthorized: false },
   285→});
   286→
   287→const DRY_RUN = process.argv.includes('--dry-run');
   288→
   289→async function main() {
   290→  // Create migrations tracking table if it doesn't exist
   291→  await pool.query(`
   292→    CREATE TABLE IF NOT EXISTS _migrations (
   293→      id SERIAL PRIMARY KEY,
   294→      filename TEXT NOT NULL UNIQUE,
   295→      applied_at TIMESTAMPTZ DEFAULT NOW()
   296→    );
   297→  `);
   298→
   299→  // Get already-applied migrations
   300→  const { rows: applied } = await pool.query(
   301→    'SELECT filename FROM _migrations ORDER BY filename'
   302→  );
   303→  const appliedSet = new Set(applied.map(r => r.filename));
   304→
   305→  // Read migration files
   306→  const migrationsDir = join(import.meta.dirname, '..', 'migrations');
   307→  const files = readdirSync(migrationsDir)
   308→    .filter(f => f.endsWith('.sql'))
   309→    .sort();
   310→
   311→  let count = 0;
   312→  for (const file of files) {
   313→    if (appliedSet.has(file)) {
   314→      console.log(`SKIP  ${file} (already applied)`);
   315→      continue;
   316→    }
   317→
   318→    const sql = readFileSync(join(migrationsDir, file), 'utf-8');
   319→    console.log(`${DRY_RUN ? 'WOULD APPLY' : 'APPLY'}  ${file}`);
   320→
   321→    if (!DRY_RUN) {
   322→      const client = await pool.connect();
   323→      try {
   324→        await client.query('BEGIN');
   325→        await client.query(sql);
   326→        await client.query(
   327→          'INSERT INTO _migrations (filename) VALUES ($1)',
   328→          [file]
   329→        );
   330→        await client.query('COMMIT');
   331→        count++;
   332→      } catch (err) {
   333→        await client.query('ROLLBACK');
   334→        console.error(`FAILED  ${file}: ${err.message}`);
   335→        process.exit(1);
   336→      } finally {
   337→        client.release();
   338→      }
   339→    }
   340→  }
   341→
   342→  console.log(`\nDone. ${count} migration(s) applied.`);
   343→  await pool.end();
   344→}
   345→
   346→main().catch(err => {
   347→  console.error(err);
   348→  process.exit(1);
   349→});
   350→```

   403→### 2.3 Database Migration Strategy
   404→
   405→```
   406→MIGRATION WORKFLOW
   407→═══════════════════
   408→
   409→  Developer writes migration
   410→       │
   411→       ▼
   412→  Test on Neon dev branch
   413→       │
   414→       ▼
   415→  PR to staging branch
   416→       │
   417→       ▼
   418→  GitHub Actions runs:
   419→    1. Create Neon branch from staging
   420→    2. Run migration on branch
   421→    3. Run tests against branch
   422→    4. If pass: merge PR, run migration on staging
   423→    5. Delete temporary Neon branch
   424→       │
   425→       ▼
   426→  PR from staging to main
   427→       │
   428→       ▼
   429→  Production deploy:
   430→    1. GitHub Action runs migration on Neon main branch
   431→    2. Vercel deploys new code
   432→    3. Code handles both old and new schema (backward compatible)
   433→```
   434→
   435→**Rules for safe migrations:**
   436→
   437→1. **Always additive.** Never DROP columns or tables in the same release as code changes. Use a two-phase approach: (a) deploy code that stops using the column, (b) drop the column in the next release.
   438→2. **Always with defaults.** New columns must have DEFAULT values so existing rows are valid.
   439→3. **Always idempotent.** Use `IF NOT EXISTS` and `IF EXISTS` so migrations can be re-run safely.
   440→4. **Always tested.** Run `node scripts/migrate.mjs --dry-run` before applying.
   441→5. **Never break reads.** The Vercel deployment is zero-downtime (atomic swap), but the database migration runs before the new code deploys. The old code must be able to read from the new schema.
   442→
   443→### 2.4 Deployment Strategy: Blue/Green via Vercel
   444→
   445→Vercel provides atomic deployments by default. Every deploy creates a new immutable deployment URL. The production alias (`africandatalayer.vercel.app`) is atomically swapped to point to the new deployment only after the build succeeds. This is effectively blue/green deployment.
   446→
   447→```
   448→DEPLOYMENT FLOW
   449→════════════════
   450→
   451→  git push origin main
   452→       │
   453→       ▼
   454→  Vercel Build Phase
   455→    1. Install dependencies (npm ci)
   456→    2. Run build (vite build)
   457→    3. Bundle API functions
   458→    4. If build fails: STOP (old deployment stays live)
   459→       │
   460→       ▼
   461→  Vercel Deploy Phase
   462→    1. Upload static assets to CDN
   463→    2. Deploy serverless functions
   464→    3. Atomic alias swap: production URL → new deployment
   465→    4. Old deployment stays available at its unique URL
   466→       │
   467→       ▼
   468→  Post-Deploy Verification
   469→    1. GitHub Action hits /api/health on new deployment
   470→    2. If health check fails: alert on Slack/email
   471→    3. Manual rollback if needed
   472→```
   473→
   474→**Rollback procedure:**
   475→
   476→```bash
   477→# Option 1: Vercel Dashboard
   478→# Go to Deployments > find the last good deployment > "..." > "Promote to Production"
   479→
   480→# Option 2: Vercel CLI
   481→vercel ls                          # List recent deployments
   482→vercel promote <deployment-url>    # Promote a specific deployment to production
   483→
   484→# Option 3: Git revert (creates new deployment)
   485→git revert HEAD
   486→git push origin main
   487→```
   488→
   489→**Rollback decision criteria:**
   490→- `/api/health` returns non-200 for > 2 minutes after deploy
   491→- Sentry error rate increases > 5x compared to pre-deploy baseline
   492→- Field agents report inability to submit data
   493→- Dashboard shows zero new submissions for > 30 minutes during active collection hours (08:00-18:00 WAT)

   665→    console.log(JSON.stringify({
   666→      event: 'api_response',
   667→      path: req.url,
   668→      status: 200,
   669→      durationMs: Date.now() - startTime,

   936→### 3.6 Implementing the Metrics Collection
   937→
   938→Since ADL runs on Vercel serverless functions (no persistent process), metrics collection is done through:
   939→
   940→1. **Sentry transactions** for performance metrics (automatic with SDK)
   941→2. **Structured console.log** for custom metrics (parsed in Vercel Logs)
   942→3. **Database queries** for data quality metrics (run by cron jobs)
   943→4. **Custom telemetry endpoint** for client-side metrics
   944→
   945→```typescript
   946→// api/telemetry/index.ts (NEW)
   947→// Receives client-side sync metrics from field agents
   948→
   949→export async function POST(req: Request): Promise<Response> {
   950→  const body = await req.json();
   951→
   952→  // Validate telemetry payload
   953→  const { agentId, metrics } = body;
   954→  if (!agentId || !metrics) {
   955→    return new Response('Invalid payload', { status: 400 });
   956→  }
   957→
   958→  // Store in database for dashboard queries
   959→  // Using point_events-style append-only pattern
   960→  await query(
   961→    `INSERT INTO agent_telemetry
   962→     (agent_id, metric_type, value, metadata, created_at)
   963→     VALUES ($1, $2, $3, $4, NOW())`,
   964→    [agentId, metrics.type, metrics.value, JSON.stringify(metrics.metadata)]
   965→  );
   966→
   967→  return new Response('OK', { status: 200 });
   968→}
   969→```
   970→
   971→```typescript
   972→// lib/client/telemetry.ts (NEW)
   973→// Sends sync health metrics to server
   974→
   975→export async function reportSyncMetrics(metrics: {
   976→  queueDepth: number;
   977→  photosPending: number;
   978→  connectionClass: string;
   979→  syncSuccessCount: number;
   980→  syncFailureCount: number;
   981→  oldestQueueItemAge: number; // minutes
   982→}) {
   983→  // Best-effort: do not block the sync loop
   984→  try {
   985→    await fetch('/api/telemetry', {
   986→      method: 'POST',
   987→      headers: { 'Content-Type': 'application/json' },
   988→      body: JSON.stringify({
   989→        agentId: getCurrentUserId(),
   990→        metrics: {
   991→          type: 'sync_health',
   992→          value: metrics.queueDepth,
   993→          metadata: metrics,
   994→        },
   995→      }),
   996→    });
   997→  } catch {
   998→    // Silently ignore telemetry failures
   999→  }
  1000→}
  1001→
  1002→// Call every 5 minutes from the sync loop
  1003→```

  2004→### 6.4 Code Review and Merge Process
  2005→
  2006→```
  2007→CODE REVIEW PROCESS
  2008→═════════════════════
  2009→
  2010→BRANCHING MODEL:
  2011→  main        ← production (auto-deploys to Vercel)
  2012→  staging     ← staging (preview deployment, migration testing)
  2013→  feature/*   ← feature branches (preview deployments)
  2014→  fix/*       ← bug fix branches
  2015→  chore/*     ← maintenance branches
  2016→
  2017→PR WORKFLOW:
  2018→
  2019→  1. Developer creates feature branch from staging
  2020→     git checkout staging && git pull
  2021→     git checkout -b feature/add-bandwidth-sync
  2022→
  2023→  2. Developer pushes and creates PR to staging
  2024→     git push -u origin feature/add-bandwidth-sync
  2025→     gh pr create --base staging --title "Add bandwidth-aware sync engine"
  2026→
  2027→  3. CI runs automatically:
  2028→     - Lint check (ESLint)
  2029→     - Format check (Prettier)
  2030→     - Type check (tsc --noEmit)
  2031→     - Unit tests
  2032→     - Build verification
  2033→     - Vercel preview deployment
  2034→
  2035→  4. Code review requirements:
  2036→     - At least 1 approval required
  2037→     - All CI checks must pass
  2038→     - No merge conflicts
  2039→     - Preview deployment tested for visual regression
  2040→
  2041→  5. Merge to staging:
  2042→     - Squash and merge (clean history)
  2043→     - Delete feature branch
  2044→     - If migration files present: migration runs on staging DB
  2045→
  2046→  6. Staging verification:
  2047→     - Test on staging preview URL
  2048→     - Verify new functionality works
  2049→     - Check Sentry for new errors
  2050→
  2051→  7. PR from staging to main:
  2052→     - Create PR: staging → main
  2053→     - Abbreviated review (already reviewed in step 4)
  2054→     - Merge (regular merge commit to preserve staging history)
  2055→     - Vercel auto-deploys to production
  2056→
  2057→  8. Post-deploy verification:
  2058→     - Health check (automated in CI)
  2059→     - Monitor Sentry error rate
  2060→     - Spot-check affected features
  2061→
  2062→CODE REVIEW CHECKLIST:
  2063→  □ Does it have tests?
  2064→  □ Does it handle errors gracefully?
  2065→  □ Does it work offline? (if client-side)
  2066→  □ Are new environment variables documented in .env.example?
  2067→  □ Are database changes backward-compatible?
  2068→  □ Does it respect the data protection law? (no PII in logs)
  2069→  □ Is the TypeScript strict? (no `any` without justification)
  2070→  □ Are API responses consistent with existing patterns?
  2071→  □ Does it have structured logging?
  2072→```
