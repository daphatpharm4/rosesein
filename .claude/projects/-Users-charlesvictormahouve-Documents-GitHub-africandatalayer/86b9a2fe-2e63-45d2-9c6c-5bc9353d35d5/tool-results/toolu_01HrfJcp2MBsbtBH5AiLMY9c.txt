   343→### 1.5 Relationship Model Between Entities
   344→
   345→Entities are not linked by foreign keys. Instead, relationships are expressed through:
   346→
   347→```
   348→RELATIONSHIP MODEL
   349→==================
   350→
   351→1. SPATIAL PROXIMITY (implicit)
   352→   - Entities within 25m of each other are "co-located"
   353→   - Used for: fuel station + mobile money agent, building + ground-floor businesses
   354→   - Computed at query time via haversine distance
   355→
   356→2. SAME-POINT ENRICHMENT (explicit, already implemented)
   357→   - Multiple events with the same point_id form one projected entity
   358→   - ENRICH_EVENT adds data to a CREATE_EVENT
   359→
   360→3. CROSS-VERTICAL REFERENCE (new, via details JSONB)
   361→   - A census_proxy building can reference its ground-floor businesses:
   362→     details.relatedPointIds = ["point-id-of-pharmacy", "point-id-of-bar"]
   363→   - A fuel_station can reference a co-located mobile_money agent:
   364→     details.coLocatedPointIds = ["point-id-of-momo-kiosk"]

   366→
   367→4. ROAD SEGMENT ADJACENCY (new, for transport_road)
   368→   - Road segments reference connected segments:
   369→     details.connectedSegmentIds = ["point-id-segment-north", "point-id-segment-south"]

   505→## 2. Entity ID Strategy
   506→
   507→### 2.1 ID Generation Scheme
   508→
   509→The current system uses two ID patterns:
   510→- **Event IDs:** `crypto.randomUUID()` -- random UUIDs (correct for immutable event log)
   511→- **Point IDs:** Either `crypto.randomUUID()` for field-created entities OR `seed-point-osm-overpass-{osmId}` for imported entities
   512→
   513→The proposed strategy formalizes this into a **composite, deterministic-when-possible** approach:
   514→
   515→```
   516→POINT ID GENERATION RULES
   517→==========================
   518→
   519→1. IMPORTED ENTITIES (from external sources)
   520→   Format: "ext-{source}-{externalId}"
   521→   Examples:
   522→     ext-osm-4382447689
   523→     ext-google-ChIJ...
   524→     ext-csv-row-042
   525→
   526→   Rationale: Deterministic. Same external entity always produces
   527→   same point_id, enabling dedup across import runs.
   528→
   529→2. FIELD-CREATED ENTITIES (new entities discovered by agents)
   530→   Format: "{category}-{geohash6}-{shortUuid}"
   531→   Examples:
   532→     pharmacy-s16gdp-a1b2c3d4
   533→     mobile_money-s16gdq-e5f6g7h8
   534→
   535→   Where:
   536→     - category = vertical ID
   537→     - geohash6 = 6-character geohash of GPS coordinates (~1.2km precision)
   538→     - shortUuid = first 8 characters of a UUIDv4
   539→
   540→   Rationale: Partially deterministic (category + location encoded),
   541→   but with random suffix to avoid collision. The geohash enables
   542→   spatial clustering of IDs for dedup candidates.
   543→
   544→3. LEGACY COMPATIBILITY
   545→   Existing point_ids (seed-point-*, random UUIDs) remain valid.
   546→   The system must accept any non-empty string as a point_id.
   547→```
   548→
   549→Implementation (extending existing `identifier.ts`):
   550→
   551→```typescript
   552→import { encode as geohashEncode } from "ngeohash";
   553→
   554→export function generatePointId(
   555→  category: string,
   556→  latitude: number,
   557→  longitude: number,
   558→): string {
   559→  const geohash = geohashEncode(latitude, longitude, 6);
   560→  const shortUuid = crypto.randomUUID().replace(/-/g, "").slice(0, 8);
   561→  return `${category}-${geohash}-${shortUuid}`;
   562→}
   563→
   564→export function generateImportPointId(source: string, externalId: string): string {
   565→  return `ext-${source}-${externalId}`;
   566→}
   567→
   568→// Extract geohash from a field-created point ID (for spatial dedup)
   569→export function extractGeohash(pointId: string): string | null {
   570→  const parts = pointId.split("-");
   571→  if (parts.length < 3) return null;
   572→  const geohash = parts[parts.length - 2];
   573→  if (!geohash || geohash.length !== 6) return null;
   574→  return geohash;
   575→}
   576→```
   577→
   578→### 2.2 Handling Informal Businesses With No Official ID
   579→
   580→Informal businesses (unlicensed bars, street vendors, mobile money kiosks without signage) are the hardest to identify. The strategy:
   581→
   582→```
   583→INFORMAL ENTITY IDENTIFICATION
   584→================================
   585→
   586→Problem: An informal alcohol vendor at a street corner has no name,
   587→no license, no signage. How do we create a stable identity?
   588→
   589→Solution: LOCATION + CATEGORY + DESCRIPTORS = identity
   590→
   591→1. The point_id is generated from GPS + category + random suffix
   592→   (as above). The GPS coordinates ARE the primary identifier.
   593→
   594→2. The "name" field for informal entities uses a descriptive pattern:
   595→   "[Category] near [landmark]"
   596→   Examples:
   597→     "Informal bar near Carrefour Bonamoussadi"
   598→     "Mobile money agent opposite Pharmacie Makepe"
   599→     "Fuel vendor (jerry cans) at Bonamoussadi junction"
   600→
   601→3. The isFormal field (present on all business verticals) is set to false.
   602→
   603→4. Photo evidence is the primary verification. The photo of the
   604→   location serves as a fingerprint for the entity.
   605→
   606→5. For subsequent visits, agents find existing entities via:
   607→   a. Map-based proximity search (see entities within 25m)
   608→   b. Visual matching against stored photos
   609→   c. Descriptive name matching
   610→```
   611→
   612→### 2.3 Deduplication Strategy
   613→
   614→Deduplication operates at three levels:
   615→
   616→```
   617→DEDUP LEVEL 1: IMPORT-TIME (before insertion)
   618→=============================================
   619→- Check external_id against existing events:
   620→  SELECT 1 FROM point_events WHERE external_id = $1 LIMIT 1
   621→- If match found, skip import (already implemented in buildCombinedEvents)
   622→- This prevents OSM re-imports from creating duplicate entities
   623→
   624→DEDUP LEVEL 2: SUBMISSION-TIME (before creating new point)
   625→==========================================================
   626→When a field agent submits a CREATE_EVENT:
   627→
   628→1. Compute candidate radius: 25m for most verticals, 50m for transport_road
   629→2. Find existing projected points within radius AND same category:
   630→
   631→   SELECT point_id, site_name, latitude, longitude
   632→   FROM snapshots
   633→   WHERE snapshot_date = (SELECT MAX(snapshot_date) FROM snapshots)
   634→     AND category = $1
   635→     AND earth_distance(
   636→       ll_to_earth(latitude, longitude),
   637→       ll_to_earth($2, $3)
   638→     ) < $4
   639→
   640→3. If candidates found:
   641→   a. Score each candidate using name similarity (Levenshtein) + distance
   642→   b. If best score > 0.7 similarity AND < 25m distance:
   643→      - Prompt agent: "This looks like [existing entity name]. Enrich it?"
   644→      - If agent confirms: convert to ENRICH_EVENT for that point_id
   645→      - If agent says "No, this is different": create new entity
   646→
   647→DEDUP LEVEL 3: ADMIN REVIEW (post-submission)
   648→==============================================
   649→Admin queue includes a "potential duplicates" flag:
   650→- System identifies pairs of points with same category, < 50m apart
   651→- Admin can merge: all events from point B re-assigned to point A
   652→- Merge is a new event_type: 'MERGE_EVENT' (point B -> point A)
   653→```
   654→
   655→```
   656→DEDUP FLOW DIAGRAM
   657→
   658→  Agent submits CREATE_EVENT
   659→          │
   660→          ▼
   661→  ┌───────────────────┐
   662→  │ Check external_id │──── match found ────► SKIP (already exists)
   663→  └───────┬───────────┘
   664→          │ no match
   665→          ▼
   666→  ┌───────────────────┐
   667→  │ Spatial search     │
   668→  │ same category      │
   669→  │ within 25m radius  │
   670→  └───────┬───────────┘
   671→          │
   672→    ┌─────┴─────┐
   673→    │           │
   674→  no match    candidates found
   675→    │           │
   676→    ▼           ▼
   677→  CREATE    ┌─────────────────┐
   678→  new       │ Show to agent:  │
   679→  entity    │ "Is this the    │
   680→            │  same as X?"    │
   681→            └───┬─────────┬───┘
   682→                │         │
   683→              "Yes"     "No"
   684→                │         │
   685→                ▼         ▼
   686→            ENRICH     CREATE
   687→            existing   new entity
   688→            entity
   689→```
   690→
   691→### 2.4 Entity Lifecycle
   692→
   693→```
   694→ENTITY STATE MACHINE
   695→=====================
   696→
   697→  ┌──────────┐   first CREATE_EVENT    ┌──────────┐
   698→  │          │─────────────────────────►│          │
   699→  │ (none)   │                          │  ACTIVE  │
   700→  │          │                          │          │
   701→  └──────────┘                          └────┬─────┘
   702→                                             │
   703→                              ┌──────────────┼──────────────┐
   704→                              │              │              │
   705→                        ENRICH_EVENT    no events for    CLOSE_EVENT
   706→                        (updates data)  N weeks          (explicit)
   707→                              │              │              │
   708→                              ▼              ▼              ▼
   709→                         ┌──────────┐  ┌──────────┐  ┌──────────┐
   710→                         │  ACTIVE  │  │ DORMANT  │  │  CLOSED  │
   711→                         │(enriched)│  │          │  │          │
   712→                         └──────────┘  └────┬─────┘  └──────────┘
   713→                                            │
   714→                                    agent re-verifies
   715→                                    (new ENRICH_EVENT)
   716→                                            │
   717→                                            ▼
   718→                                       ┌──────────┐
   719→                                       │  ACTIVE  │
   720→                                       │(revived) │
   721→                                       └──────────┘
   722→
   723→State Determination Rules (computed, not stored):
   724→- ACTIVE:  Has events within the last 8 weeks
   725→- DORMANT: Last event is 8-16 weeks old, no CLOSE_EVENT
   726→- CLOSED:  Has a CLOSE_EVENT, or last event > 16 weeks old
   727→           and not re-verified
   728→
   729→The system does NOT store lifecycle state as a column.
   730→It is computed from the event timeline during projection.
   731→```
   732→
   733→Implementation logic for the projection engine:
   734→
   735→```typescript
   736→type EntityLifecycle = "active" | "dormant" | "closed";
   737→
   738→const DORMANT_THRESHOLD_WEEKS = 8;
   739→const CLOSED_THRESHOLD_WEEKS = 16;
   740→
   741→function computeLifecycle(point: ProjectedPoint, referenceDate: Date): EntityLifecycle {
   742→  const lastEvent = new Date(point.updatedAt);
   743→  const weeksSinceLastEvent = (referenceDate.getTime() - lastEvent.getTime())
   744→    / (7 * 24 * 60 * 60 * 1000);
   745→
   746→  // Check for explicit close event (stored in details)
   747→  if (point.details.isClosed === true) return "closed";
   748→
   749→  if (weeksSinceLastEvent > CLOSED_THRESHOLD_WEEKS) return "closed";
   750→  if (weeksSinceLastEvent > DORMANT_THRESHOLD_WEEKS) return "dormant";
   751→  return "active";
   752→}
   753→```

   757→## 3. Baseline vs Snapshot Logic
   758→
   759→### 3.1 Terminology
   760→
   761→```
   762→DEFINITIONS
   763→============
   764→
   765→BASELINE:   The first complete survey of a zone for a given vertical.
   766→            All known entities are captured. This establishes the
   767→            "ground truth" against which future changes are measured.
   768→
   769→SNAPSHOT:   A periodic capture of current state. May be full (all
   770→            entities revisited) or partial (subset visited).
   771→
   772→DELTA:      The computed difference between two snapshots. Already
   773→            implemented in snapshotEngine.ts.
   774→
   775→ZONE:       A geofenced area. Currently Bonamoussadi is one zone.
   776→            Sub-zones (A, B, Makepe) will be defined for route planning.
   777→```
   778→
   779→### 3.2 Baseline Capture Process
   780→
   781→The baseline is the first `runWeeklySnapshot()` that meets the completeness threshold:
   782→
   783→```
   784→BASELINE QUALIFICATION CRITERIA
   785→================================
   786→
   787→A snapshot qualifies as "baseline" for a vertical if:
   788→
   789→1. Coverage: >= 80% of estimated entities have been captured
   790→   (estimate comes from Teammate 1's field capture scope)
   791→
   792→2. Completeness: >= 60% of captured entities have zero gaps
   793→   (all enrichable fields filled)
   794→
   795→3. Photo evidence: >= 90% of captured entities have a primary photo
   796→
   797→4. Zone coverage: Every sub-zone has at least 1 captured entity
   798→   (prevents clustering all captures in one corner)
   799→
   800→Estimated entities per vertical (from Teammate 1 analysis):
   801→  transport_road:   50-80  road segments
   802→  mobile_money:    100-200 agent points
   803→  pharmacy:         10-15  formal + 20-40 informal = 30-55
   804→  fuel_station:      5-10  stations
   805→  alcohol_outlet:   80-150 points of sale
   806→  billboard:        30-60  billboard locations
   807→  census_proxy:    560+    buildings/businesses
   808→```
   809→
   810→SQL to track baseline designation:
   811→
   812→```sql
   813→-- Add baseline tracking to snapshot_stats
   814→ALTER TABLE snapshot_stats
   815→  ADD COLUMN IF NOT EXISTS is_baseline boolean NOT NULL DEFAULT false;
   816→
   817→-- A trigger or application logic sets is_baseline = true when
   818→-- the criteria above are met for the first time per vertical.
   819→```
   820→
   821→### 3.3 Snapshot Schedule
   822→
   823→```
   824→SNAPSHOT CADENCE BY VERTICAL
   825→==============================
   826→
   827→Vertical          | Snapshot Frequency | Rationale
   828→------------------+--------------------+----------------------------------
   829→transport_road    | Daily (rainy season) | Road conditions change within hours during
   830→                  | Weekly (dry season)  | June-Oct rainy season. High commercial value.
   831→mobile_money      | Weekly              | High agent turnover, weekly churn visible
   832→pharmacy          | Bi-weekly           | Formal pharmacies are stable; informal turnover
   833→fuel_station      | Weekly              | Stock-out events are time-sensitive
   834→alcohol_outlet    | Bi-weekly           | Moderate turnover, some night-only ops
   835→billboard         | Monthly             | Slow-changing inventory
   836→census_proxy      | Monthly             | Buildings change slowly
   837→
   838→Implementation: The existing runWeeklySnapshot() runs on a single schedule.
   839→Rather than changing this, the system will:
   840→1. Run the snapshot engine weekly for ALL verticals
   841→2. For daily transport_road snapshots, add a separate cron endpoint
   842→3. Monthly verticals still get weekly snapshots (harmless -- deltas
   843→   will be "unchanged" most weeks)
   844→```
   845→
   846→### 3.4 Handling Partial Snapshots
   847→
   848→When not all entities are visited in a snapshot period, the system must avoid false "removed" deltas.
   849→
   850→```
   851→PARTIAL SNAPSHOT RULES
   852→=======================
   853→
   854→Problem: Agent visits 60 of 100 mobile money kiosks this week.
   855→The other 40 are NOT removed -- they just weren't visited.
   856→
   857→Solution: CARRY-FORWARD with decay
   858→
   859→1. The snapshot engine already handles this correctly: it projects
   860→   ALL events for all time, not just events from the current period.
   861→   If a kiosk had a CREATE_EVENT 3 weeks ago and no new events,
   862→   it still appears in the current projection.
   863→
   864→2. A point is only marked "removed" in deltas if:
   865→   a. An explicit CLOSE_EVENT is submitted, OR
   866→   b. The entity has been in DORMANT state (> 8 weeks with no events)
   867→      AND an agent physically visited the location and confirmed closure
   868→
   869→3. The "completion_rate" in snapshot_stats tracks what percentage
   870→   of known entities received at least one event this snapshot period:
   871→
   872→   completion_rate = (points with events this period) / total_points * 100
   873→
   874→4. A snapshot with < 50% completion_rate is flagged as "partial"
   875→   and its deltas carry a lower confidence weight (see Section 6).
   876→
   877→5. For partial snapshots, the delta computation adds a flag:
   878→
   879→   delta.isFromPartialSnapshot = completion_rate < 50
   880→```
   881→
   882→### 3.5 Temporal Consistency Rules
   883→
   884→```
   885→TEMPORAL CONSISTENCY
   886→=====================
   887→
   888→Rule 1: EVENT ORDERING
   889→  Events are always ordered by created_at timestamp.
   890→  If two events have the same timestamp, the one with the later
   891→  insertion (higher UUID in v7 format, or later DB insert order) wins.
   892→
   893→Rule 2: NO FUTURE EVENTS
   894→  The system rejects events with created_at > now() + 1 hour.
   895→  (1-hour tolerance for timezone/clock drift on mobile devices.)
   896→
   897→Rule 3: NO RETROACTIVE EVENTS
   898→  Events older than 72 hours are flagged for admin review.
   899→  This catches delayed offline-queue syncs.
   900→  Events older than 7 days are rejected outright.
   901→
   902→Rule 4: SNAPSHOT IMMUTABILITY
   903→  Once a snapshot is written for a given (snapshot_date, point_id),
   904→  it is never updated (ON CONFLICT DO NOTHING, already implemented).
   905→  Corrections create new events that appear in the NEXT snapshot.
   906→
   907→Rule 5: DELTA BASELINE CONSISTENCY
   908→  Deltas always compare against the immediately previous snapshot.
   909→  The system does not skip snapshots even if they were partial.
   910→  This ensures a continuous chain:
   911→    snapshot_1 -> delta_1_2 -> snapshot_2 -> delta_2_3 -> snapshot_3 -> ...
   912→
   913→Rule 6: TIMEZONE HANDLING
   914→  All timestamps are stored in UTC (timestamptz).
   915→  Cameroon is UTC+1 (WAT). The snapshot_date uses the UTC date.
   916→  A snapshot for "2026-03-03" covers events through 2026-03-03T23:59:59Z.
   917→```

   923→### 4.1 Current Implementation
   924→
   925→The existing `computeDeltas()` in `snapshotEngine.ts` already handles:

  1014→### 4.3 Delta Categories
  1015→
  1016→Extending the existing `DeltaType` with sub-categories:
  1017→
  1018→```typescript
  1019→// Existing (keep)
  1020→type DeltaType = "new" | "removed" | "changed" | "unchanged";
  1021→
  1022→// New: delta significance classification
  1023→type DeltaSignificance = "high" | "medium" | "low";
  1024→
  1025→// New: enriched delta with significance
  1026→interface EnrichedDelta extends SnapshotDelta {
  1027→  significance: DeltaSignificance;
  1028→  isPublishable: boolean;       // significance === "high" || "medium"
  1029→  isFromPartialSnapshot: boolean;
  1030→}
  1031→
  1032→// Configuration per vertical
  1033→const DELTA_SIGNIFICANCE: Record<string, Record<string, DeltaSignificance>> = {
  1034→  pharmacy: {
  1035→    isOpenNow: "high",
  1036→    isOnDuty: "high",
  1037→    openingHours: "medium",
  1038→    isLicensed: "medium",
  1039→    phone: "low",
  1040→    name: "low",
  1041→  },
  1042→  fuel_station: {
  1043→    hasFuelAvailable: "high",
  1044→    pricesByFuel: "high",
  1045→    fuelPrice: "high",
  1046→    fuelTypes: "medium",
  1047→    queueLength: "medium",
  1048→    paymentMethods: "low",
  1049→  },
  1050→  // ... (full map per vertical as defined above)
  1051→};
  1052→
  1053→function classifyDelta(delta: SnapshotDelta): DeltaSignificance {
  1054→  // New/removed entities are always HIGH
  1055→  if (delta.deltaType === "new" || delta.deltaType === "removed") return "high";
  1056→  if (delta.deltaType === "unchanged") return "low";
  1057→
  1058→  // For changed fields, look up the significance map
  1059→  const verticalMap = DELTA_SIGNIFICANCE[delta.verticalId];
  1060→  if (!verticalMap || !delta.deltaField) return "medium";
  1061→  return verticalMap[delta.deltaField] ?? "medium";
  1062→}
  1063→```

  1065→### 4.4 Confidence Thresholds for Publishing
  1066→
  1067→```
  1068→DELTA PUBLICATION RULES
  1069→========================
  1070→
  1071→A delta is published (sent to API consumers, shown in dashboard) if:
  1072→
  1073→1. The entity's confidence score >= 40 (see Section 6)
  1074→   AND
  1075→2. The delta significance is HIGH or MEDIUM
  1076→   AND
  1077→3. At least ONE of:
  1078→   a. The entity has a photo from the current snapshot period
  1079→   b. The entity has been verified by 2+ independent agents
  1080→   c. The delta is from an entity with eventsCount >= 3
  1081→
  1082→Exception: For transport_road blockage events (isBlocked = true),
  1083→publish immediately regardless of confidence score if the agent
  1084→has a photo. Road blockages are time-critical.
  1085→
  1086→Unpublished deltas are still recorded in snapshot_deltas but flagged:
  1087→  delta.isPublished = false
  1088→```

  1090→### 4.5 Delta Aggregation for Reporting
  1091→
  1092→```
  1093→AGGREGATION TIERS
  1094→==================
  1095→
  1096→DAILY ROLLUP (transport_road only during rainy season):
  1097→  - Count of blocked/unblocked road segments
  1098→  - New blockage events
  1099→  - Blockages resolved
  1100→
  1101→WEEKLY ROLLUP (all verticals):
  1102→  - This is the primary reporting cadence
  1103→  - Already implemented via snapshot_stats table
  1104→  - Metrics: total_points, new_count, removed_count, changed_count
  1105→  - Statistical: week_over_week_growth, moving_avg_4w, z-scores, anomalies
  1106→
  1107→MONTHLY ROLLUP (for commercial reports):
  1108→  Computed by aggregating 4 weekly snapshot_stats:
  1109→
  1110→  monthly_stats = {
  1111→    vertical_id,
  1112→    month,
  1113→    avg_total_points:     AVG(total_points) over 4 weeks,
  1114→    total_new:            SUM(new_count),
  1115→    total_removed:        SUM(removed_count),
  1116→    total_changed:        SUM(changed_count),
  1117→    net_growth:           SUM(new_count) - SUM(removed_count),
  1118→    churn_rate:           SUM(removed_count) / AVG(total_points) * 100,
  1119→    avg_completion_rate:  AVG(completion_rate),
  1120→    price_trend:          [weekly avg_price values for trend line],
  1121→  }
  1122→```
  1123→
  1124→SQL for monthly rollup (to be run as a scheduled job):
  1125→
  1126→```sql
  1127→CREATE TABLE IF NOT EXISTS monthly_stats (
  1128→  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  1129→  month date NOT NULL,              -- First day of month
  1130→  vertical_id text NOT NULL,
  1131→  avg_total_points numeric,
  1132→  total_new integer DEFAULT 0,
  1133→  total_removed integer DEFAULT 0,
  1134→  total_changed integer DEFAULT 0,
  1135→  net_growth integer DEFAULT 0,
  1136→  churn_rate numeric,
  1137→  avg_completion_rate numeric,
  1138→  UNIQUE(month, vertical_id)
  1139→);
  1140→
  1141→INSERT INTO monthly_stats (month, vertical_id, avg_total_points, total_new,
  1142→  total_removed, total_changed, net_growth, churn_rate, avg_completion_rate)
  1143→SELECT
  1144→  date_trunc('month', snapshot_date)::date AS month,
  1145→  vertical_id,
  1146→  ROUND(AVG(total_points), 1),
  1147→  SUM(new_count),
  1148→  SUM(removed_count),
  1149→  SUM(changed_count),
  1150→  SUM(new_count) - SUM(removed_count),
  1151→  CASE WHEN AVG(total_points) > 0
  1152→    THEN ROUND(SUM(removed_count)::numeric / AVG(total_points) * 100, 2)
  1153→    ELSE 0
  1154→  END,
  1155→  ROUND(AVG(completion_rate), 1)
  1156→FROM snapshot_stats
  1157→WHERE snapshot_date >= date_trunc('month', CURRENT_DATE - interval '1 month')
  1158→  AND snapshot_date < date_trunc('month', CURRENT_DATE)
  1159→GROUP BY date_trunc('month', snapshot_date), vertical_id
  1160→ON CONFLICT (month, vertical_id) DO UPDATE SET
  1161→  avg_total_points = EXCLUDED.avg_total_points,
  1162→  total_new = EXCLUDED.total_new,
  1163→  total_removed = EXCLUDED.total_removed,
  1164→  total_changed = EXCLUDED.total_changed,
  1165→  net_growth = EXCLUDED.net_growth,
  1166→  churn_rate = EXCLUDED.churn_rate,
  1167→  avg_completion_rate = EXCLUDED.avg_completion_rate;
  1168→```

  1660→## 6. Data Quality Scoring
  1661→
  1662→### 6.1 Confidence Score System
  1663→
  1664→The confidence score is a 0-100 integer that expresses how reliable a data point is. It is computed from multiple factors and stored in `details.confidenceScore`.
  1665→
  1666→### 6.2 Scoring Factors
  1667→
  1668→```
  1669→CONFIDENCE SCORE FACTORS
  1670→=========================
  1671→
  1672→Factor                    Weight    Range    Description
  1673→───────────────────────────────────────────────────────────────
  1674→recency                   25%       0-25     How recent is the last observation?
  1675→sourceCount               20%       0-20     How many independent events contribute?
  1676→photoEvidence             20%       0-20     Quality of photographic evidence
  1677→gpsAccuracy               15%       0-15     GPS precision and cross-validation
  1678→reviewerApproval          10%       0-10     Has an admin approved this data?
  1679→fieldCompleteness         10%       0-10     What % of enrichable fields are filled?
  1680→───────────────────────────────────────────────────────────────
  1681→TOTAL                    100%       0-100
  1682→```
  1683→
  1684→### 6.3 Score Computation Formula
  1685→
  1686→```typescript
  1687→interface ConfidenceFactors {
  1688→  recency: number;           // 0-25
  1689→  sourceCount: number;       // 0-20
  1690→  photoEvidence: number;     // 0-20
  1691→  gpsAccuracy: number;       // 0-15
  1692→  reviewerApproval: number;  // 0-10
  1693→  fieldCompleteness: number; // 0-10
  1694→}
  1695→
  1696→function computeConfidenceScore(point: ProjectedPoint, referenceDate: Date): number {
  1697→
  1698→  // ── RECENCY (0-25) ──────────────────────────────────
  1699→  const daysSinceUpdate = (referenceDate.getTime() - new Date(point.updatedAt).getTime())
  1700→    / (24 * 60 * 60 * 1000);
  1701→
  1702→  let recency: number;
  1703→  if (daysSinceUpdate <= 7)       recency = 25;  // This week
  1704→  else if (daysSinceUpdate <= 14) recency = 22;  // Last week
  1705→  else if (daysSinceUpdate <= 30) recency = 18;  // This month
  1706→  else if (daysSinceUpdate <= 60) recency = 12;  // Last month
  1707→  else if (daysSinceUpdate <= 90) recency = 6;   // 2-3 months ago
  1708→  else                            recency = 0;   // Stale
  1709→
  1710→  // ── SOURCE COUNT (0-20) ─────────────────────────────
  1711→  // Multiple independent observations increase confidence
  1712→  const events = point.eventsCount;
  1713→  let sourceCount: number;
  1714→  if (events >= 5)       sourceCount = 20;
  1715→  else if (events >= 3)  sourceCount = 15;
  1716→  else if (events >= 2)  sourceCount = 10;
  1717→  else                   sourceCount = 5;   // Single observation
  1718→
  1719→  // Bonus: if events come from different users (multi-source)
  1720→  // This requires checking event user_ids -- simplified here
  1721→  // as a bonus applied during projection
  1722→
  1723→  // ── PHOTO EVIDENCE (0-20) ──────────────────────────
  1724→  let photoEvidence = 0;
  1725→  if (point.photoUrl) photoEvidence += 10;           // Has primary photo
  1726→  if (point.details.hasSecondaryPhoto) photoEvidence += 5;  // Has secondary photo
  1727→  if (point.details.fraudCheck) {
  1728→    const fc = point.details.fraudCheck as SubmissionFraudCheck;
  1729→    if (fc.primaryPhoto?.exifStatus === "ok") photoEvidence += 5; // EXIF validated
  1730→  }
  1731→
  1732→  // ── GPS ACCURACY (0-15) ────────────────────────────
  1733→  let gpsAccuracy = 5;  // Base: GPS present (required for submission)
  1734→  if (point.details.fraudCheck) {
  1735→    const fc = point.details.fraudCheck as SubmissionFraudCheck;
  1736→    // Photo GPS matches submission GPS
  1737→    if (fc.primaryPhoto?.submissionGpsMatch === true) gpsAccuracy += 5;
  1738→    // IP location matches submission GPS
  1739→    if (fc.primaryPhoto?.ipGpsMatch === true) gpsAccuracy += 5;
  1740→  }
  1741→
  1742→  // ── REVIEWER APPROVAL (0-10) ───────────────────────
  1743→  let reviewerApproval = 0;
  1744→  if (point.details.reviewerApproved === true) reviewerApproval = 10;
  1745→  // Partial credit for not-yet-reviewed
  1746→  else if (point.eventsCount >= 2) reviewerApproval = 3;
  1747→
  1748→  // ── FIELD COMPLETENESS (0-10) ──────────────────────
  1749→  const totalEnrichable = point.gaps.length +
  1750→    Object.keys(point.details).filter(k =>
  1751→      !["fraudCheck","clientDevice","source","externalId","isImported",
  1752→        "hasPhoto","hasSecondaryPhoto","secondPhotoUrl","confidenceScore",
  1753→        "lastSeenAt","name","siteName"].includes(k)
  1754→    ).length;
  1755→  const filledFields = totalEnrichable > 0
  1756→    ? (totalEnrichable - point.gaps.length) / totalEnrichable
  1757→    : 0;
  1758→  const fieldCompleteness = Math.round(filledFields * 10);
  1759→
  1760→  // ── TOTAL ──────────────────────────────────────────
  1761→  const total = recency + sourceCount + photoEvidence + gpsAccuracy
  1762→    + reviewerApproval + fieldCompleteness;
  1763→
  1764→  return Math.min(100, Math.max(0, total));
  1765→}
  1766→```
  1767→
  1768→### 6.4 Score Thresholds
  1769→
  1770→```
  1771→CONFIDENCE SCORE THRESHOLDS
  1772→=============================
  1773→
  1774→Score Range    Label              Action
  1775→──────────────────────────────────────────────────────
  1776→80-100         HIGH CONFIDENCE    Auto-publish, no review needed
  1777→60-79          GOOD               Publish, review within 72h
  1778→40-59          MODERATE           Publish with caveat, review within 48h
  1779→20-39          LOW                Do not publish, queue for review
  1780→0-19           VERY LOW           Flag for investigation, may be fraudulent
  1781→
  1782→EXAMPLES:
  1783→
  1784→Score 85 -- "Pharmacie de Bonamoussadi"
  1785→  recency=25 (visited yesterday) + sourceCount=15 (3 events from 2 agents)
  1786→  + photoEvidence=15 (primary + EXIF ok) + gpsAccuracy=15 (all GPS match)
  1787→  + reviewerApproval=10 (admin approved) + fieldCompleteness=5
  1788→  = 85 HIGH CONFIDENCE
  1789→
  1790→Score 55 -- "Informal bar near junction" (single import)
  1791→  recency=25 (this week) + sourceCount=5 (1 event)
  1792→  + photoEvidence=10 (primary photo only, no EXIF) + gpsAccuracy=5 (GPS only)
  1793→  + reviewerApproval=0 (not reviewed) + fieldCompleteness=10 (all fields filled)
  1794→  = 55 MODERATE
  1795→
  1796→Score 25 -- "Pharmacy 300148855" (OSM import, no field verification)
  1797→  recency=0 (imported 3+ months ago, never re-verified)
  1798→  + sourceCount=5 (1 import event) + photoEvidence=0 (no photo)
  1799→  + gpsAccuracy=5 (GPS from OSM) + reviewerApproval=0
  1800→  + fieldCompleteness=3 (name only, many gaps)
  1801→  = 13 VERY LOW -> not published until field-verified
  1802→```
  1803→
  1804→### 6.5 Score Lifecycle
  1805→
  1806→```
  1807→SCORE EVOLUTION OVER TIME
  1808→==========================
  1809→
  1810→Day 0:  Entity created by field agent with photo
  1811→        Score = ~55 (recency=25, source=5, photo=10, gps=10, review=0, fields=5)
  1812→
  1813→Day 2:  Admin approves submission
  1814→        Score = ~65 (review bonus +10)
  1815→
  1816→Day 7:  Second agent verifies entity with ENRICH_EVENT
  1817→        Score = ~80 (source +5, fields +5)
  1818→
  1819→Day 30: No new events, recency decays
  1820→        Score = ~62 (recency drops from 25 to 18)
  1821→
  1822→Day 60: Still no new events
  1823→        Score = ~50 (recency drops to 12)
  1824→
  1825→Day 90: Entity enters DORMANT state
  1826→        Score = ~38 (recency = 0, entity deprioritized)
  1827→
  1828→The score decays naturally with time, ensuring that only
  1829→recently-verified data maintains high confidence.
  1830→This drives the collection workflow: agents are directed
  1831→to re-verify entities whose scores are dropping.
  1832→```
  1833→
  1834→### 6.6 Integration with Existing Codebase
  1835→
  1836→The confidence score computation integrates at two points:
  1837→
  1838→1. **At submission time** (in `POST /api/submissions`): Compute initial score and store in `details.confidenceScore`
  1839→
  1840→2. **At snapshot time** (in `runWeeklySnapshot`): Recompute all scores with current reference date, update in snapshot rows
  1841→
  1842→```typescript
  1843→// In snapshotEngine.ts, extend buildSnapshotRows:
  1844→function buildSnapshotRows(snapshotDate: string, points: ProjectedPoint[]): SnapshotRow[] {
  1845→  const referenceDate = new Date(snapshotDate);
  1846→  return points.map((p) => {
  1847→    // Recompute confidence score at snapshot time
  1848→    const freshScore = computeConfidenceScore(p, referenceDate);
  1849→    const details = { ...p.details, confidenceScore: freshScore };
  1850→
  1851→    return {
  1852→      snapshotDate,
  1853→      verticalId: p.category,
  1854→      pointId: p.pointId,
  1855→      category: p.category,
  1856→      siteName: (details.siteName as string) ?? (details.name as string) ?? null,
  1857→      latitude: p.location.latitude,
  1858→      longitude: p.location.longitude,
  1859→      details: details as Record<string, unknown>,
  1860→      gaps: p.gaps,
  1861→      eventsCount: p.eventsCount,
  1862→      photoUrl: p.photoUrl ?? null,
  1863→      source: p.source ?? null,
  1864→      externalId: p.externalId ?? null,
  1865→    };
  1866→  });
  1867→}
  1868→```
