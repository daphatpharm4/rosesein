    20→## Existing Security Baseline
    21→
    22→Before prescribing controls, here is what ADL already has in place and where the gaps are:
    23→
    24→| Layer | Current State | Gap |
    25→|---|---|---|
    26→| **Authentication** | `@auth/core` with Google OAuth + email/password credentials; session-based JWTs; CSRF protection via `csrfToken` | No MFA; no role-based access beyond `isAdmin` boolean; no session timeout enforcement; no account lockout after failed attempts |
    27→| **Authorization** | Single `isAdmin` flag on user session; API routes check `requireUser()` | No granular RBAC (field agent vs. team lead vs. admin vs. data consumer); no per-vertical access control |
    28→| **Data Encryption** | HTTPS in transit (Vercel enforces TLS); Neon PostgreSQL encrypts at rest | No field-level encryption for PII; no client-side encryption before sync; photos stored in Vercel Blob without additional encryption layer |
    29→| **Fraud Detection** | EXIF GPS extraction, haversine distance checks, IP geolocation cross-referencing (`submissionFraud.ts`) | GPS spoofing detection is limited to EXIF/haversine; no device fingerprinting; no behavioral anomaly detection |
    30→| **API Security** | Vercel serverless functions with JWT validation | No explicit rate limiting beyond Vercel defaults; no input validation library; CORS not explicitly configured (relies on Vercel defaults) |
    31→| **Photo Integrity** | Photos uploaded to Vercel Blob with URLs stored in `point_events` | No hash-on-capture; no tamper detection; no chain-of-custody for evidence photos |
    32→| **Offline Security** | IndexedDB queue (`adl_offline_queue`) stores submissions locally | IndexedDB is not encrypted; device theft exposes queued data; no remote wipe capability |
    33→| **Audit Trail** | Event-sourced `point_events` table is append-only | No dedicated security audit log; no login/logout tracking; no admin action logging |
    34→| **Data Protection Compliance** | None formalized | No privacy policy; no consent workflow; no DPO appointed; no data subject rights mechanism; June 2026 deadline approaching |

    38→## 1. Threat Model
    39→
    40→### 1.1 Methodology
    41→
    42→This threat model uses a risk-based approach adapted for ADL's operational context: field agents collecting ground-truth data with consumer Android phones in Bonamoussadi, Douala. Threats are ranked by **Likelihood x Impact** on a 5-point scale each (1=Very Low, 5=Very High), producing a risk score of 1--25.

    47→

    49→|---|---|
    50→| **Description** | Field agent's phone is stolen or lost during data collection in Bonamoussadi. Douala has significant petty crime, and agents visibly using phones in commercial areas are targets. |
    51→| **Likelihood** | 4 (High) -- Phone theft is common in Douala's commercial areas; agents use phones openly for photo capture and GPS recording |

   161→## 2. Security Controls
   162→
   163→### 2.1 Mobile Device Security
   164→
   165→ADL uses a PWA on consumer Android phones (not company-issued devices), which limits MDM options. Controls must work within this constraint.
   166→
   167→#### 2.1.1 Minimum Device Requirements (Enforced at Login)
   168→
   169→```typescript
   170→// lib/client/devicePolicy.ts
   171→export interface DeviceComplianceResult {
   172→  compliant: boolean;
   173→  violations: string[];
   174→}
   175→
   176→export function checkDeviceCompliance(): DeviceComplianceResult {
   177→  const violations: string[] = [];
   178→
   179→  // Check HTTPS context (implies no MitM proxy downgrade)
   180→  if (location.protocol !== 'https:') {
   181→    violations.push('HTTPS required');
   182→  }
   183→
   184→  // Check for mock location capability (Android)
   185→  // Note: PWAs cannot directly detect mock locations, but we can
   186→  // check the Geolocation API permission state
   187→  if (!navigator.geolocation) {
   188→    violations.push('Geolocation not available');
   189→  }
   190→
   191→  // Check for minimum screen resolution (prevents headless automation)
   192→  if (screen.width < 320 || screen.height < 480) {
   193→    violations.push('Screen resolution too low');
   194→  }
   195→
   196→  // Check for IndexedDB availability (required for offline queue)
   197→  if (!window.indexedDB) {
   198→    violations.push('IndexedDB not available');
   199→  }
   200→
   201→  return {
   202→    compliant: violations.length === 0,
   203→    violations,
   204→  };
   205→}
   206→```
   207→
   208→#### 2.1.2 Session and Local Data Protection
   209→
   210→| Control | Implementation | Mitigates |
   211→|---|---|---|
   212→| **Session timeout** | Set JWT `maxAge` to 8 hours (one field shift); force re-authentication after inactivity | T1, T2 |
   213→| **Automatic queue flush** | On app foreground and every 5 minutes when online, flush IndexedDB queue to server | T1 (reduce exposure window) |
   214→| **IndexedDB data minimization** | Store only submission payloads in queue; do not cache historical data locally | T1 |
   215→| **Photo purge after sync** | Delete photos from Vercel Blob URL cache after confirmed server receipt | T1 |
   216→| **Remote session revocation** | Admin endpoint to invalidate all sessions for a specific user ID by rotating the per-user `sub` claim | T1, T2 |

   218→#### 2.1.3 Remote Wipe (Application-Level)
   219→
   220→Since ADL cannot enforce MDM on personal devices, implement application-level wipe:
   221→
   222→```typescript
   223→// lib/client/remoteWipe.ts
   224→export async function executeAppWipe(): Promise<void> {
   225→  // 1. Clear IndexedDB offline queue
   226→  const databases = await indexedDB.databases();
   227→  for (const db of databases) {
   228→    if (db.name) indexedDB.deleteDatabase(db.name);
   229→  }
   230→
   231→  // 2. Clear all localStorage and sessionStorage
   232→  localStorage.clear();
   233→  sessionStorage.clear();
   234→
   235→  // 3. Unregister service worker
   236→  const registrations = await navigator.serviceWorker.getRegistrations();
   237→  for (const reg of registrations) {
   238→    await reg.unregister();
   239→  }
   240→
   241→  // 4. Clear caches
   242→  const cacheNames = await caches.keys();
   243→  for (const name of cacheNames) {
   244→    await caches.delete(name);
   245→  }
   246→
   247→  // 5. Redirect to login
   248→  window.location.href = '/login?wiped=true';
   249→}
   250→```
   251→
   252→Trigger: The app checks a server-side `wipe_requested` flag on every sync attempt. If set, execute wipe before any other operation.
   253→
   254→```typescript
   255→// Added to sync engine check
   256→const statusResponse = await fetch('/api/user/status');
   257→const status = await statusResponse.json();
   258→if (status.wipeRequested) {
   259→  await executeAppWipe();
   260→  return; // Stop sync
   261→}
   262→```

   264→### 2.2 Authentication and Authorization
   265→
   266→#### 2.2.1 Multi-Factor Authentication
   267→
   268→Add OTP-based MFA for all agent accounts. SMS OTP is the most practical for Cameroon (agents have phone numbers; authenticator apps require explanation and training).
   269→
   270→| Step | Implementation |
   271→|---|---|
   272→| **OTP generation** | Server generates 6-digit TOTP, stores hash in `user_profiles.mfa_secret` |
   273→| **OTP delivery** | Send via SMS using a Cameroon-compatible provider (Africa's Talking, Twilio with Cameroon numbers, or MTN/Orange SMS API) |
   274→| **OTP verification** | Verify within 5-minute window; max 3 attempts before lockout |
   275→| **When required** | On every login (not every request -- JWT session handles ongoing auth) |
   276→| **Fallback** | Admin can generate a bypass code for agents who lose phone access; bypass codes are single-use and logged |
   277→
   278→**Important consideration for Cameroon:** SMS delivery in Cameroon can be delayed, especially during network congestion. Set OTP validity to 10 minutes (not the typical 5) and allow re-send after 60 seconds.

   332→### 2.3 Data Encryption
   333→
   334→#### 2.3.1 Encryption at Rest
   335→
   336→| Layer | Current | Target |
   337→|---|---|---|
   338→| **Neon PostgreSQL** | AES-256 at rest (Neon default) | Sufficient for pilot; add column-level encryption for PII fields in Phase 2 |
   339→| **Vercel Blob (photos)** | Server-side encryption (Vercel default) | Sufficient; photos are of businesses/infrastructure, not individuals |
   340→| **IndexedDB (client)** | No encryption | Encrypt queue payloads with a session-derived key (see below) |
   341→
   342→#### 2.3.2 Client-Side Queue Encryption
   343→
   344→Encrypt IndexedDB contents so stolen devices yield no readable data:
   345→
   346→```typescript
   347→// lib/client/queueEncryption.ts
   348→const ALGORITHM = 'AES-GCM';
   349→
   350→export async function deriveKey(sessionToken: string): Promise<CryptoKey> {
   351→  const encoder = new TextEncoder();
   352→  const keyMaterial = await crypto.subtle.importKey(
   353→    'raw',
   354→    encoder.encode(sessionToken),
   355→    'PBKDF2',
   356→    false,
   357→    ['deriveKey']
   358→  );
   359→
   360→  return crypto.subtle.deriveKey(
   361→    {
   362→      name: 'PBKDF2',
   363→      salt: encoder.encode('adl-queue-v1'),
   364→      iterations: 100000,
   365→      hash: 'SHA-256',
   366→    },
   367→    keyMaterial,
   368→    { name: ALGORITHM, length: 256 },
   369→    false,
   370→    ['encrypt', 'decrypt']
   371→  );
   372→}
   373→
   374→export async function encryptPayload(
   375→  data: string,
   376→  key: CryptoKey
   377→): Promise<{ iv: string; ciphertext: string }> {
   378→  const iv = crypto.getRandomValues(new Uint8Array(12));
   379→  const encoded = new TextEncoder().encode(data);
   380→  const ciphertext = await crypto.subtle.encrypt(
   381→    { name: ALGORITHM, iv },
   382→    key,
   383→    encoded
   384→  );
   385→  return {
   386→    iv: btoa(String.fromCharCode(...iv)),
   387→    ciphertext: btoa(String.fromCharCode(...new Uint8Array(ciphertext))),
   388→  };
   389→}
   390→
   391→export async function decryptPayload(
   392→  encrypted: { iv: string; ciphertext: string },
   393→  key: CryptoKey
   394→): Promise<string> {
   395→  const iv = Uint8Array.from(atob(encrypted.iv), c => c.charCodeAt(0));
   396→  const ciphertext = Uint8Array.from(atob(encrypted.ciphertext), c => c.charCodeAt(0));
   397→  const decrypted = await crypto.subtle.decrypt(
   398→    { name: ALGORITHM, iv },
   399→    key,
   400→    ciphertext
   401→  );
   402→  return new TextDecoder().decode(decrypted);
   403→}
   404→```
   405→
   406→#### 2.3.3 Field-Level Encryption for PII
   407→
   408→For fields that qualify as personal data under Cameroon Law No. 2024/017 (business owner names, phone numbers, agent identifiers):
   409→
   410→```sql
   411→-- Create an encryption key managed via Neon/Supabase secrets
   412→-- Field-level encryption using pgcrypto extension
   413→
   414→CREATE EXTENSION IF NOT EXISTS pgcrypto;
   415→

   417→-- Application layer handles encrypt/decrypt using a server-side key
   418→-- stored in Vercel environment variables (FIELD_ENCRYPTION_KEY)
   419→```
   420→
   421→**Implementation approach:** Application-level encryption is preferred over database-level because:
   422→1. Neon PostgreSQL's pgcrypto works but creates key management complexity
   423→2. Server-side Node.js encryption using `crypto.createCipheriv` with AES-256-GCM keeps the key in Vercel env vars
   424→3. Encrypted fields are stored as base64 strings in the JSONB `details` column
   425→4. Decryption happens at query time in the API layer, only for authorized roles

   459→#### 2.4.2 Input Validation
   460→
   461→Add Zod schemas for all API inputs (builds on Cloud Architect's recommendation):
   462→
   463→```typescript
   464→// lib/shared/validation.ts
   465→import { z } from 'zod';
   466→
   467→export const submissionSchema = z.object({
   468→  category: z.enum([
   469→    'pharmacy', 'fuel_station', 'mobile_money',
   470→    'alcohol_outlet', 'billboard', 'transport_road', 'census_proxy'
   471→  ]),
   472→  latitude: z.number().min(4.0755).max(4.0999),   // Bonamoussadi bounds
   473→  longitude: z.number().min(9.7185).max(9.7602),   // Bonamoussadi bounds
   474→  details: z.record(z.unknown()).refine(
   475→    (d) => JSON.stringify(d).length < 50000,
   476→    'Details payload too large'
   477→  ),
   478→  photoUrl: z.string().url().optional(),
   479→  idempotencyKey: z.string().uuid(),
   480→});
   481→
   482→// Sanitize all string inputs to prevent XSS/injection
   483→export function sanitizeString(input: string): string {
   484→  return input
   485→    .replace(/[<>]/g, '')           // Strip HTML angle brackets

   514→### 2.5 Photo/Evidence Integrity
   515→
   516→#### 2.5.1 Hash-on-Capture
   517→
   518→Generate a SHA-256 hash of each photo immediately upon capture, before any upload:
   519→
   520→```typescript
   521→// lib/client/photoIntegrity.ts
   522→export async function hashPhoto(file: File): Promise<string> {
   523→  const buffer = await file.arrayBuffer();
   524→  const hashBuffer = await crypto.subtle.digest('SHA-256', buffer);
   525→  const hashArray = Array.from(new Uint8Array(hashBuffer));
   526→  return hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
   527→}
   528→
   529→export interface PhotoEvidence {
   530→  file: File;
   531→  sha256: string;
   532→  capturedAt: string;       // ISO 8601
   533→  deviceInfo: string;       // navigator.userAgent (truncated)
   534→  gpsAtCapture: {
   535→    latitude: number;
   536→    longitude: number;
   537→    accuracy: number;
   538→  } | null;
   539→}
   540→```
   541→
   542→#### 2.5.2 Server-Side Verification
   543→
   544→```typescript
   545→// In api/submissions.ts POST handler
   546→async function verifyPhotoIntegrity(
   547→  photoUrl: string,
   548→  clientHash: string
   549→): Promise<boolean> {
   550→  const response = await fetch(photoUrl);
   551→  const buffer = await response.arrayBuffer();
   552→  const serverHash = createHash('sha256').update(Buffer.from(buffer)).digest('hex');
   553→  return serverHash === clientHash;
   554→}
   555→```

   664→## 3. Privacy and Consent Framework
   665→
   666→### 3.1 Data Inventory and Legal Basis
   667→
   668→Under Cameroon Law No. 2024/017, ADL must identify all personal data collected and establish a legal basis for processing.
   669→
   670→#### 3.1.1 Personal Data Collected by ADL
   671→
   672→| Data Element | Vertical(s) | Personal Data? | Sensitivity | Legal Basis |
   673→|---|---|---|---|---|
   674→| **GPS coordinates of businesses** | All | Yes (if business is identifiable with an individual owner) | Low | Legitimate interest (Article 6) |
   675→| **Photos of business premises** | All | Yes (if individuals are incidentally captured; if business facade identifies an individual) | Medium | Consent (for identifiable individuals); Legitimate interest (for premises only) |
   676→| **Business name** | All | Yes (if sole proprietorship or clearly identified with an individual) | Low | Legitimate interest |
   677→| **Business owner name** | Pharmacy, alcohol, fuel | Yes | Medium | Consent |
   678→| **Business phone number** | Mobile money, pharmacy | Yes | Medium | Consent |
   679→| **Agent GPS tracks** | Internal | Yes (employee monitoring) | Medium | Employment contract (legitimate interest) |
   680→| **Agent identity (name, email, phone)** | Internal | Yes | Medium | Employment contract |
   681→| **Agent device information** | Internal | Yes | Low | Employment contract |
   682→| **Road condition photos** | Transport roads | Potentially (incidental capture of people/vehicles) | Low | Legitimate interest (public interest) |
   683→| **Building photos** | Census proxy | Potentially (incidental capture of residents) | Low | Legitimate interest |
   684→
   685→#### 3.1.2 Legal Basis Analysis Under Law No. 2024/017
   686→
   687→Cameroon's law recognizes the following legal bases (modeled after GDPR):
   688→
   689→1. **Consent (Article 7):** Required when processing has no other legal basis. Must be free, specific, informed, and unambiguous.
   690→2. **Legitimate interest (Article 6):** ADL's primary basis for most business data -- mapping commercial infrastructure serves a legitimate purpose (market intelligence, urban planning, public safety).
   691→3. **Public interest:** Potential basis for road condition and census-proxy data.
   692→4. **Contractual necessity:** Basis for processing agent data under employment/contractor agreements.
   693→
   694→**Key legal determination needed:** Whether a photo of a business facade constitutes personal data of the business owner. Under Cameroonian civil law tradition (French-influenced), the answer likely depends on whether the individual is identifiable. For sole proprietorships where the owner's name is on the facade, it qualifies as personal data. For branded chain outlets (Total fuel station, MTN agent point), it likely does not.

   829→### 3.5 Cross-Border Data Transfer Controls
   830→
   831→ADL's infrastructure stores data outside Cameroon:
   832→- **Vercel serverless functions:** Deployed to Vercel's edge network; nearest PoP is likely in Europe (no African PoP yet)

   967→3. Admin triggers remote wipe for the agent's account:
   968→   - Set wipe_requested = true on user_profiles
   969→   - Revoke all active sessions (rotate JWT secret for this user)
   970→   - Mark account as suspended (is_active = false)
   971→4. If device had unsynced submissions:
   972→   - Note the approximate count and recency from agent's memory
   973→   - Mark potentially compromised data window in incident log
   974→
   975→WITHIN 24 HOURS:
   976→5. Agent files police report at nearest Douala commissariat
   977→6. ADL receives copy of police report (PV) for records
   978→7. Assess data exposure:
   979→   - Was IndexedDB queue encrypted? (If encryption implemented: LOW risk)
   980→   - How many unsynced submissions were on device?
   981→   - Did any submissions contain PII (business owner names, photos)?
   982→8. If PII was on device and queue was NOT encrypted:
   983→   - This qualifies as a personal data breach under Law No. 2024/017
   984→   - Proceed to regulatory notification procedure (Section 4.2.2)
   985→
   986→WITHIN 48 HOURS:
   987→9. Issue replacement access to agent (new account or reactivated with new credentials + MFA)
   988→10. Agent re-collects any lost unsynced data
   989→11. Document lessons learned
   990→
   991→SEVERITY: SEV-2 (unless PII exposure confirmed, then SEV-1)
   992→```

  1280→## 5. Field Agent Security Training
  1281→
  1282→### 5.1 Training Program Overview
  1283→
  1284→| Module | Duration | Delivery | Frequency |
  1285→|---|---|---|---|
  1286→| **M1: Device Security** | 45 minutes | In-person, hands-on | At onboarding + annual refresh |
  1287→| **M2: Data Handling** | 30 minutes | In-person with scenarios | At onboarding + annual refresh |
  1288→| **M3: Consent Collection** | 60 minutes | In-person with role-play | At onboarding + quarterly refresh |
  1289→| **M4: Social Engineering** | 30 minutes | In-person with examples | At onboarding + semi-annual refresh |
  1290→| **M5: Incident Reporting** | 20 minutes | In-person + quick-reference card | At onboarding + annual refresh |
  1291→| **M6: Physical Safety** | 30 minutes | In-person with local context | At onboarding + annual refresh |
  1292→
  1293→**Total onboarding training time:** ~3.5 hours (can be split across 2 days)
  1294→**Language:** French (primary), with key terms in English for the app interface
  1295→**Assessment:** Short quiz (10 questions) at end of each module; must score 80% to be activated as an agent
  1296→
  1297→### 5.2 Module Details
  1298→
  1299→#### M1: Device Security
  1300→
  1301→**Learning objectives:** Agents can secure their device, enable screen lock, and know what to do if their phone is stolen.
  1302→
  1303→Content:
  1304→1. **Screen lock:** Set a 6-digit PIN or fingerprint; do NOT use pattern lock (easy to observe)
  1305→2. **App lock:** If available, set app-specific lock on the ADL PWA
  1306→3. **Avoid public Wi-Fi:** Use mobile data (MTN/Orange) for syncing; public Wi-Fi in Douala is not secure
  1307→4. **Keep phone updated:** Accept Android security updates when prompted
  1308→5. **Do not root/jailbreak:** Rooted phones compromise security; ADL may refuse submissions from rooted devices in the future
  1309→6. **Charging security:** Do not use public USB charging stations (juice-jacking risk); use personal power banks or wall chargers
  1310→7. **What to do if phone is stolen or lost:**
  1311→   - Immediately call team lead: [Phone number]
  1312→   - Do NOT try to recover the phone yourself
  1313→   - File a police report at the nearest commissariat within 24 hours
  1314→   - ADL will remotely disable your account and wipe app data
  1315→
  1316→**Hands-on exercise:** Each agent configures screen lock and demonstrates sync while connected to mobile data.
  1317→
  1318→#### M2: Data Handling
  1319→
  1320→**Learning objectives:** Agents understand what data they can and cannot collect, and how to handle it.
  1321→
  1322→**DO:**
  1323→- Only collect data for the vertical(s) you are assigned
  1324→- Sync your data as frequently as possible (do not accumulate a large offline queue)
  1325→- Use the ADL app for all data entry (do not use notes apps, WhatsApp, or paper)
  1326→- Report any data errors you notice (wrong location, wrong category)
  1327→
  1328→**DO NOT:**
  1329→- Take photos of people's faces (focus on business facades and infrastructure)
  1330→- Collect data about people's personal lives, health conditions, or financial details
  1331→- Share your login credentials with anyone, including other agents
  1332→- Share ADL data with anyone outside ADL (including screenshots of the app)
  1333→- Store ADL data outside the app (no WhatsApp photos, no email exports)
  1334→- Continue using ADL if your account is suspended (contact team lead)
  1335→
  1336→**Scenario exercise:** Present 5 field situations and ask agents to identify what they should and should not do.
  1337→
  1338→#### M3: Consent Collection
  1339→
  1340→**Learning objectives:** Agents can explain ADL's purpose, collect consent, and handle refusals professionally.
  1341→
  1342→**Role-play scenarios (in French):**
  1343→
  1344→1. **Willing business owner:** Agent approaches, explains, owner agrees, agent records consent and collects data.
  1345→
  1346→2. **Reluctant business owner:** Owner is suspicious. Agent remains calm, explains data subject rights, offers to leave the info card, and respects the refusal. Agent records `consent_refused` and collects only non-PII data (GPS + category).
  1347→
  1348→3. **Informal operator:** Agent approaches an informal mobile money agent who is not registered. Agent explains that ADL maps services to improve access. Does NOT mention regulatory compliance or enforcement. Collects only publicly observable data if consent is not given.
  1349→
  1350→4. **Person objects to being photographed:** Agent apologizes, explains they are only photographing the business facade, and offers to retake the photo when the person has moved. If person insists, agent skips the photo and submits text-only.
  1351→
  1352→5. **Language barrier:** Business owner speaks only a local language (e.g., Pidgin English, Bassa, Duala). Agent uses basic phrases or asks a nearby person to help translate the core message. If cannot communicate, skip the consent-required data and record `consent_refused`.

  1385→#### M4: Social Engineering Awareness
  1386→
  1387→**Learning objectives:** Agents recognize manipulation attempts and do not disclose ADL information.
  1388→
  1389→**Scenarios to recognize:**
  1390→
  1391→1. **"I'm from the government, show me your data":** No government official can demand to see ADL data on your phone. Ask for their identification, explain you will report the request to ADL management, and leave. Call your team lead.
  1392→
  1393→2. **"I work for ADL too, let me use your phone":** Never let anyone use your ADL account. Credential sharing is grounds for termination.
  1394→
  1395→3. **"My boss wants to know what you are recording":** Explain ADL's purpose using the standard script. Do not show the app interface or data to anyone.
  1396→
  1397→4. **"Send me the photos on WhatsApp":** ADL data is never shared via WhatsApp, email, or any channel outside the app. Decline politely.
  1398→
  1399→5. **"Give me your login, I forgot mine":** Each agent has their own credentials. They must contact the team lead for password reset. Never share credentials.

  1529→## Appendix A: Key Provisions of Cameroon Law No. 2024/017
  1530→
  1531→For quick reference by the ADL team:
  1532→
  1533→| Article | Provision | ADL Relevance |
  1534→|---|---|---|
  1535→| **Art. 2** | Applies to any processing of personal data by a person in Cameroon or using means located in Cameroon | ADL is subject -- agents in Cameroon, data about Cameroon |
  1536→| **Art. 5** | Defines personal data as any information relating to an identified or identifiable natural person | Business owner names, photos with faces, phone numbers qualify |
  1537→| **Art. 6** | Processing must have a legal basis (consent, contract, legal obligation, public interest, legitimate interest) | ADL relies on legitimate interest (mapping) and consent (PII) |
  1538→| **Art. 7** | Consent must be free, specific, informed, and unambiguous; can be withdrawn at any time | Consent workflow required for PII collection |
  1539→| **Art. 15-16** | Data controller must register with the DPA before processing | ADL must register before pilot launch |
  1540→| **Art. 20** | DPO appointment required in certain circumstances | ADL likely requires a DPO (systematic monitoring) |
  1541→| **Art. 28-32** | Cross-border transfers require adequate protection in recipient country or appropriate safeguards | SCCs and DPAs required for Vercel, Neon, Google |
  1542→| **Art. 34-37** | Data subject rights: access, rectification, erasure, objection | Must implement technical and organizational measures |
  1543→| **Art. 45** | Breach notification to DPA within 72 hours | Incident response plan must be operational |
  1544→| **Art. 52-55** | Penalties: fines up to 100 million XAF (~$150,000 USD); imprisonment for certain violations | Material financial and personal risk |
  1545→| **Art. 60** | 6-month transitional period for compliance (expires June 23, 2026) | Hard deadline for all controls |
