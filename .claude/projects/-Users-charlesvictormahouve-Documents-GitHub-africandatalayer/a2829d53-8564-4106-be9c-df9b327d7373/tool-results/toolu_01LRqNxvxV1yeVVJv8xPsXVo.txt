INCIDENT RESPONSE PLAN\n\n### 4.1 INCIDENT CLASSIFICATION (Operations/Incident Response)\n\n| Severity | Definition | Examples | Response Time | Escalation |\n|---|---|---|---|---|\n| **SEV-1: Critical** | Active breach; agent physical danger; complete outage | Database compromise, agent assault, ransomware, unauthorized bulk export | Immediate (30 min) | CEO + DPO + legal |\n| **SEV-2: High** | Confirmed incident without active exploitation | Stolen device with unsynced data, compromised credentials, Vercel/Neon outage >1 hour | Within 2 hours | CTO + DPO |\n| **SEV-3: Medium** | Suspected incident; limited impact | Unusual API patterns, failed MFA attempts, single agent impersonation | Within 24 hours | CTO |\n| **SEV-4: Low** | Minor event; no data exposure | Password resets, expired certs, phishing attempts | Within 72 hours | Engineering lead |\n\n### 4.2 RESPONSE PROCEDURES (Runbooks)\n\n#### 4.2.1 Device Theft/Loss Runbook (Operations)\n\n**Trigger:** Agent reports device stolen/lost\n\n**Immediate (30 minutes):**\n1. Agent calls team lead or ADL emergency number\n2. Team lead logs incident in `security_audit_log`\n3. Admin triggers remote wipe:\n   - Set `wipe_requested = true` on `user_profiles`\n   - Revoke all active sessions (rotate JWT secret for user)\n   - Mark account suspended (`is_active = false`)\n4. Note unsynced submission count from agent memory\n\n**Within 24 hours:**\n5. Agent files police report at nearest commissariat\n6. ADL receives copy of police report (PV)\n7. Assess data exposure:\n   - Was IndexedDB queue encrypted? (If yes: LOW risk)\n   - How many unsynced submissions?\n   - Did submissions contain PII?\n8. If PII on device AND queue NOT encrypted → **Regulatory breach notification required** (Section 4.2.2)\n\n**Within 48 hours:**\n9. Issue replacement access (new account or reactivated with new credentials + MFA)\n10. Agent re-collects lost unsynced data\n11. Document lessons learned\n\n**Severity:** SEV-2 (SEV-1 if PII exposure confirmed)\n\n#### 4.2.2 Data Breach Regulatory Notification Runbook (DPO/Legal)\n\n**Trigger:** Confirmed exposure of personal data to unauthorized parties\n\n**Breach classification:**\n- Unencrypted device theft with PII: **BREACH**\n- Database compromise: **BREACH**\n- API exploitation with data exfiltration: **BREACH**\n- Vercel/Neon provider breach affecting ADL: **BREACH**\n- Stolen device with encrypted queue, no PII: **NOT A BREACH** (document and close)\n\n**Within 72 hours (legal deadline under Law No. 2024/017):**\n1. DPO prepares notification to Data Protection Authority:\n   - Nature of breach\n   - Categories & approximate number of data subjects\n   - Categories & approximate number of records\n   - Likely consequences\n   - Measures taken/proposed\n   - DPO contact\n2. DPO submits to:\n   - Cameroon Data Protection Authority (when operational)\n   - Until operational: Ministry of Posts and Telecommunications (transitional authority)\n   - Via email + registered letter\n3. If HIGH risk to data subjects:\n   - Notify affected data subjects directly\n   - Use French notification template (Section 4.3.2)\n   - Notify via: phone call (primary), email (secondary), physical visit (if feasible)\n\n**Within 7 days:**\n4. Complete internal investigation\n5. Implement remediation\n6. Update DPA with investigation report\n\n**Within 30 days:**\n7. Complete post-incident review (Section 4.5)\n8. Update security controls\n9. File final report with DPA\n\n#### 4.2.3 Agent Compromise (Credential Theft/Impersonation) Runbook (Operations/Security)\n\n**Trigger:** Suspected unauthorized agent credential use\n\n**Detection signals:**\n- Submissions from unusual IP addresses\n- Submissions outside assigned zone\n- Submissions at unusual hours (3 AM)\n- Multiple simultaneous sessions\n- GPS patterns inconsistent with walking speed (>50 km/h)\n- EXIF/GPS mismatch on photos\n- Sudden change in submission volume/pattern\n\n**Immediate:**\n1. Suspend agent account (`is_active = false`)\n2. Revoke all active sessions\n3. Trigger remote wipe\n4. Contact real agent via known phone to verify\n\n**If impersonation confirmed:**\n5. Flag all submissions from suspected compromise window\n6. Admin reviews flagged submissions for data integrity\n7. Delete or quarantine fraudulent submissions\n8. Agent creates new account with new credentials + MFA\n9. Investigate compromise method (shared creds → training; phishing → training; malware → device inspection)\n10. If fraudulent data served to customers, notify them\n\n**Severity:** SEV-2 (SEV-1 if large fraudulent data volume distributed)\n\n#### 4.2.4 System Outage Runbook (Operations/Engineering)\n\n**Trigger:** Platform unavailable >15 minutes\n\n**Triage:**\n1. Check Vercel status page (vercel.com/status)\n2. Check Neon status page (neon.tech/status)\n3. Check `/api/health` endpoint from external monitor\n4. Check GitHub Actions for failed deployments\n\n**If Vercel outage:**\n- Field agents continue collecting offline (IndexedDB queue)\n- Communicate to team leads: \"Continue collecting; data will sync when service restores\"\n- Monitor Vercel status\n- No action unless >24 hours\n\n**If Neon/database outage:**\n- API returns errors; Edge Config fallback serves read-only cached data\n- Field agents can submit (queue locally) but won't persist\n- If >4 hours: activate backup database restore\n\n**If ADL application bug:**\n- Roll back to previous Vercel deployment\n- Investigate root cause\n- Fix, test, redeploy\n\n**Communication:**\n- Team leads notified within 30 minutes (WhatsApp)\n- Field agents notified within 1 hour (via team leads)\n- If >4 hours: post status to admin dashboard\n\n**Severity:** SEV-2 (if >1 hour); SEV-3 (if <1 hour)\n\n### 4.3 COMMUNICATION TEMPLATES (DPO/Operations)\n\n#### 4.3.1 Internal Incident Alert (French)\n\n```\nALERTE SECURITE ADL -- [SEV-X]\nDate/Heure: [YYYY-MM-DD HH:MM UTC]\nType d'incident: [Device theft / Data breach / Agent compromise / Outage]\nDescription: [Brief description in French]\nImpact: [What data/users/operations affected]\nActions immediates prises: [List of actions taken]\nProchaines etapes: [Next steps and responsible parties]\nPoint de contact: [Name + phone]\n\nNE PAS PARTAGER EN DEHORS DE L'EQUIPE ADL.\n```\n\n#### 4.3.2 Data Subject Breach Notification (French)\n\n```\nNOTIFICATION DE VIOLATION DE DONNEES PERSONNELLES\n\nMadame, Monsieur,\n\nNous vous informons qu'un incident de securite affectant vos donnees\npersonnelles a ete detecte le [DATE].\n\nNature de l'incident: [Description]\n\nDonnees concernees: [List of data types, e.g., nom commercial, localisation GPS, photo]\n\nConsequences probables: [Impact assessment]\n\nMesures prises: [Remediation actions taken]\n\nMesures recommandees: [Steps recipient should take]\n\nVos droits: Conformement a la Loi No. 2024/017, vous avez le droit d'acceder,\nfaire rectifier ou supprimer vos donnees.
