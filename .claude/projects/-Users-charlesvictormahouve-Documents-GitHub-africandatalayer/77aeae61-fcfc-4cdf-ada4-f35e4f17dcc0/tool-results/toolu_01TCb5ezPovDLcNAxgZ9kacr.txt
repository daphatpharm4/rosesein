    21→## Existing System Inventory
    22→
    23→Before proposing the target architecture, here is what is already deployed and operational:
    24→
    25→| Layer | Current Technology | Status |
    26→|---|---|---|
    27→| **Frontend** | React 19 + Vite 6 SPA | Deployed on Vercel |
    28→| **Backend API** | Vercel Serverless Functions (Node.js) | `/api/*` routes |
    29→| **Primary Database** | PostgreSQL (Neon/Supabase-hosted) | `point_events`, `snapshots`, `snapshot_deltas`, `snapshot_stats`, `user_profiles` |
    30→| **Legacy Fallback** | Vercel Edge Config | Read-only fallback when Postgres unavailable |
    31→| **File Storage** | Vercel Blob (`@vercel/blob`) | Photo evidence uploads |
    32→| **Offline Queue** | IndexedDB (`adl_offline_queue`) | Client-side queuing with exponential backoff |
    33→| **Sync** | Custom `submissionSync.ts` + `offlineQueue.ts` | Sequential flush with idempotency keys |
    34→| **Auth** | `@auth/core` + Google OAuth + email/password | Session-based |
    35→| **AI** | Google Gemini (`@google/genai`) | Server-side search |
    36→| **Analytics** | `@vercel/analytics` + `@vercel/speed-insights` | Page-level |
    37→| **Maps** | Leaflet + React-Leaflet | POI visualization |
    38→| **Charts** | Recharts | Dashboard analytics |
    39→| **Cron** | Vercel Cron (`vercel.json`) | Weekly snapshot engine, Mondays 03:00 UTC |
    40→| **Fraud Detection** | EXIF GPS extraction + haversine + IP geolocation | Server-side on submission |
    41→
    42→**Key architectural decision already made:** The system is event-sourced. All data enters as immutable `point_events` (CREATE_EVENT / ENRICH_EVENT), projected into `ProjectedPoint` objects, frozen into weekly `snapshots`, with `snapshot_deltas` computed between consecutive snapshots. This is the correct foundation. The architecture below extends it without replacing it.

   281→## 2. Offline-First Mobile Capture
   282→
   283→### 2.1 Current State Assessment
   284→
   285→The existing offline implementation is functional but minimal:
   286→
   287→**What works:**
   288→- IndexedDB-based queue (`adl_offline_queue` store) with `pending`/`syncing`/`failed`/`synced` states
   289→- Idempotency keys for deduplication on retry
   290→- Exponential backoff with jitter (base 1s, max 30s)
   291→- Error classification (retryable vs permanent)
   292→- Sync error recording for diagnostics
   293→- Client device profiling (`deviceProfile.ts`)
   294→
   295→**What is missing for pilot readiness:**
   296→- No local data cache (agents cannot view existing POIs offline)
   297→- No photo staging (photos are not stored locally if upload fails)
   298→- No bandwidth-aware sync (same behavior on 2G vs 4G)
   299→- No upload prioritization (text and photos treated equally)
   300→- No sync progress visibility beyond queue count
   301→- No assignment data cached locally
   302→- No battery/data usage optimization
   303→

   386→### 2.3 Photo Compression and Staging
   387→
   388→```typescript
   389→// Target: compress photos to < 500KB while maintaining evidence quality
   390→
   391→interface PhotoCompressionConfig {
   392→  maxWidth: 1280;           // px - sufficient for signage readability
   393→  maxHeight: 1280;          // px
   394→  quality: 0.7;             // JPEG quality (0-1)
   395→  targetSizeBytes: 500_000; // 500KB target
   396→  maxSizeBytes: 2_000_000;  // 2MB hard limit (reject if can't compress below)
   397→  format: 'image/jpeg';     // Always JPEG for size efficiency
   398→}
   399→
   400→// Compression pipeline:
   401→// 1. Read photo from camera/file input
   402→// 2. Draw to canvas at maxWidth/maxHeight (maintain aspect ratio)
   403→// 3. Export as JPEG at quality 0.7
   404→// 4. If > targetSize, reduce quality in steps (0.6, 0.5, 0.4)
   405→// 5. If > maxSize after quality 0.4, reduce dimensions by 50%
   406→// 6. Store compressed Blob in photo_staging IndexedDB store
   407→// 7. Original is discarded after staging
   408→
   409→// EXIF preservation:
   410→// Extract GPS coordinates BEFORE compression (canvas strips EXIF)
   411→// Store extracted GPS in the QueueItem payload alongside the photo
   412→// This is critical for fraud detection on the server
   413→```
   414→
   415→**Photo staging flow:**
   416→
   417→```
   418→  Camera capture
   419→       │
   420→       ▼
   421→  ┌─────────────────────┐
   422→  │ Extract EXIF GPS     │  Using exifr (already in dependencies)
   423→  │ from original        │  before any processing
   424→  └──────────┬──────────┘
   425→             │
   426→             ▼
   427→  ┌─────────────────────┐
   428→  │ Compress to canvas   │  1280px max, JPEG 0.7
   429→  │ → JPEG blob          │  Target: < 500KB
   430→  └──────────┬──────────┘
   431→             │
   432→             ▼
   433→  ┌─────────────────────┐
   434→  │ Store in IndexedDB   │  photo_staging store
   435→  │ photo_staging        │  Status: 'staged'
   436→  └──────────┬──────────┘
   437→             │
   438→             ▼
   439→  ┌─────────────────────┐
   440→  │ Create QueueItem     │  Links to staged photo
   441→  │ in submission_queue  │  hasPhoto: true
   442→  └─────────────────────┘
   443→```

   445→### 2.4 Queue Management and Prioritization
   446→
   447→```
   448→SYNC PRIORITY LEVELS
   449→═════════════════════
   450→
   451→Priority 1 (CRITICAL):  Road blockage events (isBlocked = true)
   452→                         These have real-time commercial value.
   453→                         Sync immediately if any connectivity exists.
   454→
   455→Priority 2 (HIGH):      Text-only submissions (CREATE_EVENT / ENRICH_EVENT
   456→                         without photo). Small payload (~1-3 KB).
   457→                         Sync over any connection including 2G.
   458→
   459→Priority 3 (NORMAL):    Submissions with photos, events older than 1 hour.
   460→                         Sync when 3G+ detected.
   461→
   462→Priority 4 (LOW):       POI cache refresh (download updated POI list).
   463→                         Sync when on 4G or WiFi.
   464→
   465→Priority 5 (DEFERRED):  Assignment downloads, map tile caching.
   466→                         Sync on WiFi or when charging.
   467→```

   469→### 2.5 Conflict Resolution Strategy
   470→
   471→```
   472→CONFLICT RESOLUTION RULES
   473→══════════════════════════
   474→
   475→The event-sourced model largely eliminates conflicts because events are
   476→immutable and append-only. Two agents can submit events for the same
   477→point_id without conflict -- both are stored and merged during projection.
   478→
   479→However, two conflict scenarios exist:
   480→
   481→SCENARIO 1: Duplicate entity creation
   482→  Agent A creates "Pharmacy near Carrefour" at point X
   483→  Agent B creates "Pharmacie du Carrefour" at point X + 15m
   484→  Both are offline, both sync later.
   485→
   486→  Resolution: Server-side dedup (Teammate 2, Section 2.3)
   487→  - Spatial proximity check (25m radius, same category)
   488→  - Name similarity scoring
   489→  - If score > 0.7: flag for admin merge review
   490→  - If score < 0.7: accept as separate entities
   491→  - Admin merge creates MERGE_EVENT
   492→
   493→SCENARIO 2: Contradictory enrichments
   494→  Agent A submits: pharmacy isOpenNow = true (at 09:00)
   495→  Agent B submits: pharmacy isOpenNow = false (at 14:00)
   496→  Both for the same point_id.
   497→
   498→  Resolution: Last-write-wins (already implemented)
   499→  - projectPointsFromEvents() orders events by created_at
   500→  - Latest event's value takes precedence for scalar fields
   501→  - Both events are preserved in the log
   502→  - Delta engine captures the change
   503→
   504→SCENARIO 3: Offline edits to stale data
   505→  Agent views cached POI data from 3 days ago (last sync).
   506→  Agent submits ENRICH_EVENT based on outdated state.
   507→
   508→  Resolution: Accept unconditionally
   509→  - The event records what the agent observed in the field
   510→  - The timestamp records when the observation was made
   511→  - If the data is older than 72 hours, flag for admin review
   512→    (Teammate 2, Section 3.5, Rule 3)
   513→  - The admin can accept or discard the late event
   514→```

   517→
   518→```
   519→BATTERY OPTIMIZATION
   520→═════════════════════
   521→
   522→1. GPS USAGE
   523→   - Request high-accuracy GPS only when agent taps "Capture"

   774→## 4. Sync Protocol
   775→
   776→### 4.1 Enhanced Sync Engine Design
   777→
   778→The existing `flushOfflineQueue()` in `offlineQueue.ts` is a good foundation. The enhanced sync engine adds bandwidth awareness, upload prioritization, and resumable photo uploads.
   779→
   780→```
   781→SYNC ENGINE STATE MACHINE
   782→═══════════════════════════
   783→
   784→  ┌──────────┐   connectivity    ┌──────────────┐
   785→  │          │   detected        │              │
   786→  │  IDLE    ├──────────────────►│  PROBING     │
   787→  │          │                   │  (measure    │
   788→  │          │◄──────────────────┤   bandwidth) │
   789→  │          │   no connectivity │              │
   790→  └────┬─────┘                   └──────┬───────┘
   791→       │                                │
   792→       │  timer tick                    │  probe complete
   793→       │  (check queue)                 │  (RTT + speed measured)
   794→       │                                │
   795→       ▼                                ▼
   796→  ┌──────────┐                   ┌──────────────┐
   797→  │ CHECK    │                   │  CLASSIFY    │
   798→  │ QUEUE    │                   │  CONNECTION  │
   799→  │          │                   │              │
   800→  │ pending  │                   │  2G: text    │
   801→  │ items?   │                   │  3G: text +  │
   802→  │          │                   │      photos  │
   803→  └────┬─────┘                   │  4G: all     │
   804→       │                         │  WiFi: all   │
   805→       │ yes, items pending      │  + cache     │
   806→       │                         └──────┬───────┘
   807→       ▼                                │
   808→  ┌──────────────────────────────────────┴──────────┐
   809→  │                                                  │
   810→  │              SYNC LOOP                           │
   811→  │                                                  │
   812→  │  1. Sort queue by priority (1=highest, 5=lowest) │
   813→  │  2. Filter by connection class:                  │
   814→  │     2G: only priority 1-2 (text, critical)       │
   815→  │     3G: priority 1-4 (text + photos)             │
   816→  │     4G/WiFi: all priorities                      │
   817→  │  3. For each item:                               │
   818→  │     a. If text-only: POST /api/submissions       │
   819→  │     b. If has photo:                             │
   820→  │        i.  POST /api/submissions (text first)    │
   821→  │        ii. Upload photo to Vercel Blob           │
   822→  │     c. On success: remove from queue             │
   823→  │     d. On failure: exponential backoff           │
   824→  │  4. Update sync status UI                        │
   825→  │                                                  │
   826→  └──────────────────────────────────────────────────┘
   827→```

   829→### 4.2 Upload Priority Rules
   830→
   831→```
   832→UPLOAD PRIORITY MATRIX
   833→═══════════════════════
   834→
   835→PRIORITY   CONTENT            SIZE     MIN CONNECTION   MAX RETRY
   836→────────────────────────────────────────────────────────────────────
   837→1          Road blockage      ~2 KB    2G (any)         Unlimited
   838→           (isBlocked=true)
   839→
   840→2          Text submissions    ~2 KB    2G (any)         10 retries
   841→           (no photo)
   842→
   843→3          Photo uploads       ~400KB   3G (>0.5 Mbps)   5 retries
   844→           (normal priority)
   845→
   846→4          POI cache download  ~250KB   3G (>1 Mbps)     3 retries
   847→           (GET, background)
   848→
   849→5          Assignments +       ~50KB    4G/WiFi          2 retries
   850→           map tiles
   851→           (background)
   852→
   853→BANDWIDTH MEASUREMENT:
   854→  The sync engine measures bandwidth by timing a small probe request:
   855→
   856→  async function measureBandwidth(): Promise<ConnectionClass> {
   857→    // Use Navigator.connection API if available (Chrome Android)
   858→    const conn = (navigator as any).connection;
   859→    if (conn?.effectiveType) {
   860→      return conn.effectiveType; // '2g', '3g', '4g'
   861→    }
   862→
   863→    // Fallback: time a HEAD request to /api/health
   864→    const start = performance.now();
   865→    try {
   866→      await fetch('/api/health', { method: 'HEAD' });
   867→      const rttMs = performance.now() - start;
   868→      if (rttMs > 2000) return '2g';
   869→      if (rttMs > 500) return '3g';
   870→      return '4g';
   871→    } catch {
   872→      return 'offline';
   873→    }
   874→  }
   875→```

   877→### 4.3 Resumable Photo Uploads
   878→
   879→```
   880→RESUMABLE UPLOAD PROTOCOL
   881→══════════════════════════
   882→
   883→For photos > 200KB, use chunked upload to handle dropped connections:
   884→
   885→1. Client splits photo Blob into 100KB chunks
   886→2. Each chunk uploaded with:
   887→   POST /api/submissions/upload-chunk
   888→   Headers:
   889→     X-Upload-Id: {queueItemId}
   890→     X-Chunk-Index: {0, 1, 2, ...}
   891→     X-Total-Chunks: {N}
   892→     Content-Type: application/octet-stream
   893→   Body: raw chunk bytes
   894→
   895→3. Server stores chunks in temporary storage (Vercel /tmp)
   896→4. When all chunks received:
   897→   - Reassemble into complete file
   898→   - Upload to Vercel Blob
   899→   - Return blob URL
   900→   - Update point_event.photo_url
   901→
   902→5. If upload interrupted:
   903→   - Client queries: GET /api/submissions/upload-status?id={queueItemId}
   904→   - Server returns: { receivedChunks: [0, 1], totalExpected: 4 }
   905→   - Client resumes from chunk 2
   906→
   907→6. Timeout: Incomplete uploads cleaned up after 1 hour
   908→
   909→SIMPLIFIED ALTERNATIVE (recommended for pilot):
   910→  For the 6-week pilot with photos < 500KB after compression,
   911→  standard single-request upload is sufficient. The retry mechanism
   912→  in flushOfflineQueue already handles failures. Implement chunked
   913→  uploads only if field testing reveals persistent upload failures.
   914→```

   916→### 4.4 Server-Side Deduplication on Receive
   917→
   918→```
   919→SERVER DEDUP FLOW
   920→══════════════════
   921→
   922→Already partially implemented via idempotency keys. Enhancement:
   923→
   924→1. IDEMPOTENCY KEY CHECK (existing)
   925→   - Client sends X-Idempotency-Key header
   926→   - Server checks: does this key already exist?
   927→   - If yes: return 200 with existing event (not 409 Conflict)
   928→   - If no: proceed with insert
   929→
   930→2. CONTENT HASH CHECK (new)
   931→   - Compute SHA-256 of (category + point_id + event_type +
   932→     latitude.toFixed(5) + longitude.toFixed(5) +
   933→     JSON.stringify(sortedDetails))
   934→   - Store hash in point_events.content_hash column
   935→   - Check for duplicate hash within last 24 hours
   936→   - If found: return 200 (treat as successful, skip insert)
   937→
   938→3. SPATIAL TEMPORAL CHECK (new, for CREATE events)
   939→   - Check: does a CREATE_EVENT exist for same category
   940→     within 25m and within last 4 hours by the same user?
   941→   - If yes: flag as potential duplicate, store but mark
   942→     for admin review
   943→
   944→SQL for content hash:
   945→  ALTER TABLE point_events
   946→    ADD COLUMN IF NOT EXISTS content_hash text;
   947→  CREATE INDEX IF NOT EXISTS idx_point_events_content_hash
   948→    ON point_events (content_hash)
   949→    WHERE content_hash IS NOT NULL;
   950→```

  1552→### 6.4 Disaster Recovery Plan
  1553→
  1554→```
  1555→DISASTER RECOVERY
  1556→══════════════════
  1557→
  1558→SCENARIO 1: Neon PostgreSQL data loss
  1559→  Prevention:
  1560→    - Neon provides point-in-time recovery (PITR) on paid plans
  1561→    - On Free Tier: no automatic PITR
  1562→
  1563→  Recovery plan (pilot):
  1564→    1. Events are the source of truth. All other tables are derived.
  1565→    2. Weekly: run pg_dump via Neon console, store in Vercel Blob
  1566→    3. If data loss: restore from latest dump
  1567→    4. Replay any events from agent offline queues that haven't
  1568→       been acknowledged
  1569→    5. Re-run snapshot engine to rebuild snapshots/deltas/stats
  1570→
  1571→  Recovery time: 1-4 hours
  1572→  Data loss window: up to 7 days (weekly backup) minus any
  1573→    events still in agent offline queues
  1574→
  1575→SCENARIO 2: Vercel Blob storage loss (photos)
  1576→  Prevention: Vercel Blob is backed by Cloudflare R2 with
  1577→    11 nines durability.
  1578→
  1579→  Recovery plan:
  1580→    - Photos are evidence, not source data. Loss is painful
  1581→      but not catastrophic -- the event data survives.
  1582→    - Re-collection of photos can be done on next field visit.
  1583→    - For pilot: accept the risk. At scale: replicate to a
  1584→      second blob store.
  1585→
  1586→SCENARIO 3: Complete Vercel account compromise
  1587→  Prevention:
  1588→    - 2FA on Vercel account
  1589→    - Separate admin accounts (no shared credentials)
  1590→    - Environment variables for all secrets
  1591→
  1592→  Recovery plan:
  1593→    1. Restore from git repository (all code is in GitHub)
  1594→    2. Re-deploy to new Vercel project
  1595→    3. Restore database from Neon backup
  1596→    4. Update DNS
  1597→    5. Recovery time: 2-4 hours
  1598→
  1599→BACKUP SCHEDULE (PILOT):
  1600→  - Database (pg_dump):  Weekly, stored in Vercel Blob
  1601→  - Code:                Continuous (GitHub, already in place)
  1602→  - Config (.env):       Documented in .env.example (already exists)
  1603→  - Photos:              No backup (accept risk for pilot)
  1604→```

  1825→### 7.4 Why Not a Native Mobile App?
  1826→
  1827→```
  1828→PWA vs NATIVE APP DECISION
  1829→════════════════════════════
  1830→
  1831→Decision: Stay with PWA (Progressive Web App) for pilot.
  1832→
  1833→FACTOR              PWA (current)                  NATIVE (future)
  1834→──────────────────────────────────────────────────────────────────
  1835→Development cost    $0 (already built)             $10,000-30,000
  1836→Time to market      0 weeks (done)                 8-12 weeks
  1837→Update deployment   Instant (Vercel deploy)        Play Store review (days)
  1838→Offline support     IndexedDB (sufficient)         SQLite (more powerful)
  1839→Camera access       Good (MediaDevices API)        Excellent (full control)
  1840→GPS accuracy        Good (Geolocation API)         Excellent (raw GNSS)
  1841→Battery control     Limited                        Full (JobScheduler)
  1842→Background sync     Service Worker (limited)       WorkManager (robust)
  1843→File size limit     ~50 MB (IndexedDB)             Unlimited (filesystem)
  1844→Install friction    "Add to Home Screen" prompt    Play Store install
  1845→Agent familiarity   Opens in Chrome (familiar)     New app (training)
  1846→──────────────────────────────────────────────────────────────────
  1847→
  1848→Verdict: PWA is sufficient for 500 POIs and 20 agents. The main
  1849→limitations (background sync, battery control) are manageable
  1850→with the adaptive sync engine described in Section 4. Switch to
  1851→native (React Native + Expo) only if pilot reveals:
  1852→  - Background sync failures > 10% of submissions
  1853→  - Battery drain complaints from agents
  1854→  - Photo quality issues requiring raw camera control
  1855→  - Need for > 50 MB local storage per agent
