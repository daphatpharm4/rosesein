   111→**Key additions explained:**

   115→- Immutable caching for Vite hashed assets (`/assets/index-QrY-Hdyj.js`).
   116→- SPA rewrite rule for client-side routing.
   117→- Cron jobs for deep health check, materialized view refresh, and nightly backup export.

   197→**`supabase/migrations/20260227_rls_policies.sql`**
   198→```sql
   199→-- Enable Row Level Security on all tables
   200→ALTER TABLE public.user_profiles ENABLE ROW LEVEL SECURITY;
   201→ALTER TABLE public.point_events ENABLE ROW LEVEL SECURITY;
   202→ALTER TABLE public.audit_log ENABLE ROW LEVEL SECURITY;
   203→
   204→-- Application role (used by the API server via POSTGRES_URL)
   205→-- Vercel functions connect as 'authenticated_api' role
   206→DO $$
   207→BEGIN
   208→  IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'adl_api') THEN
   209→    CREATE ROLE adl_api;
   210→  END IF;
   211→END $$;
   212→
   213→GRANT SELECT, INSERT, UPDATE ON public.user_profiles TO adl_api;
   214→GRANT SELECT, INSERT, UPDATE, DELETE ON public.point_events TO adl_api;
   215→GRANT SELECT, INSERT ON public.audit_log TO adl_api;
   216→GRANT SELECT ON public.point_projections TO adl_api;
   217→
   218→-- Read-only role for preview deployments
   219→DO $$
   220→BEGIN
   221→  IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'adl_readonly') THEN
   222→    CREATE ROLE adl_readonly;
   223→  END IF;
   224→END $$;
   225→
   226→GRANT SELECT ON public.user_profiles TO adl_readonly;
   227→GRANT SELECT ON public.point_events TO adl_readonly;
   228→GRANT SELECT ON public.audit_log TO adl_readonly;
   229→GRANT SELECT ON public.point_projections TO adl_readonly;
   230→
   231→-- RLS policies for user_profiles
   232→CREATE POLICY user_profiles_api_select ON public.user_profiles
   233→  FOR SELECT TO adl_api USING (true);
   234→CREATE POLICY user_profiles_api_insert ON public.user_profiles
   235→  FOR INSERT TO adl_api WITH CHECK (true);
   236→CREATE POLICY user_profiles_api_update ON public.user_profiles
   237→  FOR UPDATE TO adl_api USING (true);
   238→
   239→-- RLS policies for point_events
   240→CREATE POLICY point_events_api_all ON public.point_events
   241→  FOR ALL TO adl_api USING (true);
   242→
   243→-- RLS policies for audit_log (insert-only for API, select for admins via direct query)
   244→CREATE POLICY audit_log_api_insert ON public.audit_log
   245→  FOR INSERT TO adl_api WITH CHECK (true);
   246→CREATE POLICY audit_log_api_select ON public.audit_log
   247→  FOR SELECT TO adl_api USING (true);
   248→```

   266→### 1.3 Environment Variable Management Strategy
   267→
   268→**Variable classification:**
   269→
   270→| Variable | Environments | Sensitivity | Source |
   271→|----------|-------------|-------------|--------|
   272→| `ADL_POSTGRES_URL` | All (different values) | SECRET | Vercel Env Vars (encrypted) |
   273→| `BLOB_READ_WRITE_TOKEN` | All (different values) | SECRET | Vercel Env Vars (encrypted) |
   274→| `AUTH_SECRET` | All (different values) | SECRET | Vercel Env Vars (encrypted) |
   275→| `GOOGLE_CLIENT_ID` | Production + Preview | CONFIDENTIAL | Vercel Env Vars |
   276→| `GOOGLE_CLIENT_SECRET` | Production + Preview | SECRET | Vercel Env Vars (encrypted) |
   277→| `GEMINI_API_KEY` | Production only | SECRET | Vercel Env Vars (server only) |
   278→| `ADMIN_EMAIL` | Production only | INTERNAL | Vercel Env Vars |
   279→| `ADMIN_PASSWORD` | REMOVE | CRITICAL | Migrate to DB password_hash |
   280→| `EDGE_CONFIG` | All | CONFIDENTIAL | Vercel Env Vars |
   281→| `DATA_STORE_DRIVER` | All | PUBLIC | Vercel Env Vars (always `postgres`) |
   282→| `POSTGRES_POOL_MAX` | Per environment | PUBLIC | Vercel Env Vars |
   283→| `SENTRY_DSN` | All | PUBLIC | Vercel Env Vars |
   284→| `VERCEL_API_TOKEN` | CI only | CRITICAL | GitHub Actions Secrets only |
   285→
   286→**Naming convention for environment-specific variables:**
   287→
   288→```
   289→Production:  ADL_POSTGRES_URL = postgres://adl_api:***@db.xyz.supabase.co:5432/postgres
   290→Staging:     ADL_POSTGRES_URL = postgres://adl_api:***@db.abc.supabase.co:5432/postgres
   291→Preview:     ADL_POSTGRES_URL = postgres://adl_readonly:***@db.abc.supabase.co:5432/postgres
   292→```
   293→
   294→**Critical fix -- remove client-side API key exposure:**
   295→
   296→The current `vite.config.ts` exposes `GEMINI_API_KEY` to the browser bundle:
   297→
   298→```typescript
   299→// CURRENT -- INSECURE (vite.config.ts lines 19-22)
   300→define: {
   301→  'process.env.API_KEY': JSON.stringify(env.GEMINI_API_KEY),
   302→  'process.env.GEMINI_API_KEY': JSON.stringify(env.GEMINI_API_KEY)
   303→},
   304→```
   305→
   306→**Fix:** Remove these `define` entries. Create a server-side proxy at `api/gemini/index.ts` that uses `process.env.GEMINI_API_KEY` on the server and expose a safe endpoint to the client.

   308→### 1.4 IaC Tool Selection
   309→
   310→**Phase 1 (Now -- MVP):** No Terraform/Pulumi needed. Manage infra with:
   311→- `vercel.json` for Vercel configuration
   312→- `supabase/migrations/*.sql` for database schema
   313→- GitHub Actions for CI/CD
   314→- Vercel CLI for deployment
   315→
   316→**Phase 2 (1K-10K users):** Introduce Pulumi with TypeScript for:
   317→- Multi-environment Supabase project provisioning
   318→- Vercel project and domain configuration
   319→- Upstash Redis provisioning (rate limiting)
   320→- Edge Config management
   321→
   322→**Phase 3 (10K+ users):** Full Pulumi stack:
   323→- Cloudflare R2 for photo archival
   324→- Supabase read replicas
   325→- QStash topic management
   326→
   327→**Rationale for Pulumi over Terraform:**
   328→- TypeScript native (same language as the codebase)
   329→- Vercel and Supabase both have Pulumi providers
   330→- Richer programming constructs than HCL
   331→- Single `package.json` can include infra code

   389→### 2.1 GitHub Actions Workflow Design
   390→
   391→**`.github/workflows/ci.yml`** -- Main CI/CD pipeline:
   392→
   393→```yaml
   394→name: CI/CD Pipeline
   395→
   396→on:
   397→  push:
   398→    branches: [main, develop]
   399→  pull_request:
   400→    branches: [main, develop]
   401→
   402→concurrency:
   403→  group: ${{ github.workflow }}-${{ github.ref }}
   404→  cancel-in-progress: true
   405→
   406→env:
   407→  NODE_VERSION: "20"
   408→
   409→jobs:
   410→  # ──────────────────────────────────────────
   411→  # Stage 1: Lint & Type Check
   412→  # ──────────────────────────────────────────
   413→  lint:
   414→    name: Lint & Type Check
   415→    runs-on: ubuntu-latest
   416→    steps:
   417→      - uses: actions/checkout@v4
   418→
   419→      - uses: actions/setup-node@v4
   420→        with:
   421→          node-version: ${{ env.NODE_VERSION }}
   422→          cache: "npm"
   423→
   424→      - run: npm ci
   425→
   426→      - name: TypeScript type check
   427→        run: npx tsc --noEmit --strict
   428→
   429→      - name: Check for console.log in production code
   430→        run: |
   431→          if grep -rn "console\.log\b" api/ lib/server/ --include="*.ts" | grep -v "test\." | grep -v "\.test\."; then
   432→            echo "::warning::Found console.log statements in production code. Use structured logging instead."
   433→          fi
   434→
   435→  # ──────────────────────────────────────────
   436→  # Stage 2: Unit Tests
   437→  # ──────────────────────────────────────────
   438→  test:
   439→    name: Unit Tests
   440→    runs-on: ubuntu-latest
   441→    needs: lint
   442→    steps:
   443→      - uses: actions/checkout@v4
   444→
   445→      - uses: actions/setup-node@v4
   446→        with:
   447→          node-version: ${{ env.NODE_VERSION }}
   448→          cache: "npm"
   449→
   450→      - run: npm ci
   451→
   452→      - name: Run tests
   453→        run: node --test tests/*.test.ts
   454→        env:
   455→          NODE_OPTIONS: "--experimental-strip-types"
   456→
   457→      - name: Upload test results
   458→        if: always()
   459→        uses: actions/upload-artifact@v4
   460→        with:
   461→          name: test-results
   462→          path: test-results/
   463→
   464→  # ──────────────────────────────────────────
   465→  # Stage 3: Build & Bundle Analysis
   466→  # ──────────────────────────────────────────
   467→  build:
   468→    name: Build & Bundle Analysis
   469→    runs-on: ubuntu-latest
   470→    needs: lint
   471→    steps:
   472→      - uses: actions/checkout@v4
   473→
   474→      - uses: actions/setup-node@v4
   475→        with:
   476→          node-version: ${{ env.NODE_VERSION }}
   477→          cache: "npm"
   478→
   479→      - run: npm ci
   480→
   481→      - name: Build
   482→        run: npm run build
   483→        env:
   484→          NODE_ENV: production
   485→
   486→      - name: Check bundle size
   487→        run: |
   488→          MAX_MAIN_KB=350
   489→          MAX_TOTAL_KB=800
   490→          MAIN_SIZE=$(du -sk dist/assets/index-*.js | awk '{print $1}')
   491→          TOTAL_SIZE=$(du -sk dist/assets/ | awk '{print $1}')
   492→          echo "Main bundle: ${MAIN_SIZE}KB (limit: ${MAX_MAIN_KB}KB)"
   493→          echo "Total assets: ${TOTAL_SIZE}KB (limit: ${MAX_TOTAL_KB}KB)"
   494→          if [ "$MAIN_SIZE" -gt "$MAX_MAIN_KB" ]; then
   495→            echo "::error::Main bundle exceeds ${MAX_MAIN_KB}KB limit (${MAIN_SIZE}KB)"
   496→            exit 1
   497→          fi
   498→          if [ "$TOTAL_SIZE" -gt "$MAX_TOTAL_KB" ]; then
   499→            echo "::warning::Total assets exceed ${MAX_TOTAL_KB}KB soft limit (${TOTAL_SIZE}KB)"
   500→          fi
   501→
   502→      - name: Upload build artifacts
   503→        uses: actions/upload-artifact@v4
   504→        with:
   505→          name: build-dist
   506→          path: dist/
   507→          retention-days: 7
   508→
   509→  # ──────────────────────────────────────────
   510→  # Stage 4: Security Scanning
   511→  # ──────────────────────────────────────────
   512→  security:
   513→    name: Security Scan
   514→    runs-on: ubuntu-latest
   515→    needs: lint
   516→    steps:
   517→      - uses: actions/checkout@v4
   518→
   519→      - uses: actions/setup-node@v4
   520→        with:
   521→          node-version: ${{ env.NODE_VERSION }}
   522→          cache: "npm"
   523→
   524→      - run: npm ci
   525→
   526→      - name: npm audit (production deps)
   527→        run: npm audit --production --audit-level=high
   528→        continue-on-error: true
   529→
   530→      - name: Check for known vulnerable packages
   531→        run: |
   532→          npx audit-ci --high --report-type full
   533→
   534→      - name: Check for secrets in code
   535→        uses: trufflesecurity/trufflehog@main
   536→        with:
   537→          extra_args: --only-verified
   538→
   539→  # ──────────────────────────────────────────
   540→  # Stage 5: Database Migration Dry Run (PRs only)
   541→  # ──────────────────────────────────────────
   542→  migration-check:
   543→    name: Migration Dry Run
   544→    runs-on: ubuntu-latest
   545→    if: github.event_name == 'pull_request'
   546→    steps:
   547→      - uses: actions/checkout@v4
   548→
   549→      - name: Check for new migration files
   550→        id: migration-check
   551→        run: |
   552→          CHANGED=$(git diff --name-only origin/${{ github.base_ref }}...HEAD -- 'supabase/migrations/')
   553→          if [ -n "$CHANGED" ]; then
   554→            echo "has_migrations=true" >> "$GITHUB_OUTPUT"
   555→            echo "Changed migration files:"
   556→            echo "$CHANGED"
   557→          else
   558→            echo "has_migrations=false" >> "$GITHUB_OUTPUT"
   559→            echo "No migration changes detected."
   560→          fi
   561→
   562→      - name: Validate SQL syntax
   563→        if: steps.migration-check.outputs.has_migrations == 'true'
   564→        run: |
   565→          for f in supabase/migrations/*.sql; do
   566→            echo "Checking $f..."
   567→            # Basic SQL syntax validation
   568→            if ! grep -qE '(CREATE|ALTER|DROP|INSERT|UPDATE|DELETE|GRANT|REVOKE|DO)' "$f"; then
   569→              echo "::warning::$f does not contain recognizable SQL statements"
   570→            fi
   571→          done
   572→
   573→      - name: Label PR with migration tag
   574→        if: steps.migration-check.outputs.has_migrations == 'true'
   575→        uses: actions/github-script@v7
   576→        with:
   577→          script: |
   578→            github.rest.issues.addLabels({
   579→              owner: context.repo.owner,
   580→              repo: context.repo.repo,
   581→              issue_number: context.issue.number,
   582→              labels: ['database-migration']

   725→### 2.4 Database Migration Pipeline
   726→
   727→**Safe migration process (zero-downtime):**
   728→
   729→```
   730→Step 1: PR contains new file in supabase/migrations/
   731→Step 2: CI auto-labels PR with 'database-migration'
   732→Step 3: migration-check job validates SQL syntax
   733→Step 4: Developer triggers manual migration.yml workflow on staging (dry run first)
   734→Step 5: Verify staging works with new schema
   735→Step 6: PR merges to main
   736→Step 7: Developer triggers migration.yml on production (dry run, then apply)
   737→Step 8: main deploys new code that uses the new schema
   738→```
   739→
   740→**Critical rules:**
   741→- Migrations MUST be backwards-compatible (old code must work with new schema)
   742→- Never DROP columns or tables in the same deploy as code changes
   743→- Use a two-phase approach: Phase 1 adds columns/tables, Phase 2 (next release) removes old ones
   744→- Always add `IF NOT EXISTS` / `IF EXISTS` guards
   745→- Test migrations against a copy of production data before applying
   746→
   747→### 2.5 Rollback Strategy
   748→
   749→**Application rollback (instant):**
   750→```bash
   751→# List recent deployments
   752→vercel ls --prod
   753→
   754→# Instantly rollback to a specific deployment
   755→vercel rollback <deployment-url> --prod
   756→
   757→# Or promote a previous deployment
   758→vercel promote <deployment-url>
   759→```
   760→
   761→**Database rollback:** Migrations are forward-only. To "undo" a migration:
   762→1. Write a new migration that reverts the changes
   763→2. Apply via the migration pipeline
   764→3. This is why backwards-compatible migrations are essential
   765→
   766→**Emergency rollback procedure:**
   767→1. Vercel instant rollback to last known-good deployment (< 30 seconds)
   768→2. If DB schema change caused the issue: apply reverse migration
   769→3. If data corruption: restore from Supabase PITR (see section 4)

   849→### 3.1 Logging Strategy
   850→
   851→**Current state:** Minimal logging. `console.info("[SUBMISSION_DEVICE]", ...)` only in non-production (line 673 of `api/submissions/index.ts`). `console.error("Postgres pool error", error)` in `lib/server/db.ts`.
   852→
   853→**Structured logging module -- `lib/server/logger.ts`:**
   854→
   855→```typescript
   856→type LogLevel = "debug" | "info" | "warn" | "error";
   857→
   858→interface LogEntry {
   859→  level: LogLevel;
   860→  message: string;
   861→  timestamp: string;
   862→  requestId?: string;
   863→  userId?: string;
   864→  action?: string;
   865→  duration_ms?: number;
   866→  metadata?: Record<string, unknown>;
   867→}
   868→
   869→const LOG_LEVEL_PRIORITY: Record<LogLevel, number> = {
   870→  debug: 0,
   871→  info: 1,
   872→  warn: 2,
   873→  error: 3,
   874→};
   875→
   876→const MIN_LEVEL: LogLevel =
   877→  (process.env.LOG_LEVEL as LogLevel) || (process.env.NODE_ENV === "production" ? "info" : "debug");
   878→
   879→function shouldLog(level: LogLevel): boolean {
   880→  return LOG_LEVEL_PRIORITY[level] >= LOG_LEVEL_PRIORITY[MIN_LEVEL];
   881→}
   882→
   883→function emit(entry: LogEntry): void {
   884→  if (!shouldLog(entry.level)) return;
   885→  const output = JSON.stringify(entry);
   886→  if (entry.level === "error") {
   887→    console.error(output);
   888→  } else if (entry.level === "warn") {
   889→    console.warn(output);
   890→  } else {
   891→    console.log(output);
   892→  }
   893→}
   894→
   895→export const logger = {
   896→  debug: (message: string, metadata?: Record<string, unknown>) =>
   897→    emit({ level: "debug", message, timestamp: new Date().toISOString(), ...metadata }),
   898→  info: (message: string, metadata?: Record<string, unknown>) =>
   899→    emit({ level: "info", message, timestamp: new Date().toISOString(), ...metadata }),
   900→  warn: (message: string, metadata?: Record<string, unknown>) =>
   901→    emit({ level: "warn", message, timestamp: new Date().toISOString(), ...metadata }),
   902→  error: (message: string, metadata?: Record<string, unknown>) =>
   903→    emit({ level: "error", message, timestamp: new Date().toISOString(), ...metadata }),
   904→};
   905→```
   906→
   907→**What to log (and what NOT to log):**
   908→
   909→| Event | Level | Log? | Fields |
   910→|-------|-------|------|--------|
   911→| Submission created | `info` | YES | `eventId`, `userId`, `category`, `eventType`, `duration_ms` |
   912→| Submission validation failed | `warn` | YES | `reason`, `userId`, `category` |
   913→| Fraud check flagged | `warn` | YES | `eventId`, `userId`, `riskScore`, `signals` |
   914→| Photo upload failed | `error` | YES | `eventId`, `errorCode`, `errorMessage` |
   915→| DB query timeout | `error` | YES | `query` (first 100 chars), `timeout_ms` |
   916→| DB pool error | `error` | YES | `errorCode`, `errorMessage` |
   917→| Auth login success | `info` | YES | `userId`, `provider` |
   918→| Auth login failed | `warn` | YES | `identifier` (hashed), `reason` |
   919→| Health check | `debug` | NO in prod | `db_status`, `latency_ms` |
   920→| Request body contents | - | NEVER | Contains PII, photos |
   921→| Password hashes | - | NEVER | Security |
   922→| Full SQL queries with params | - | NEVER | May contain PII |

   969→### 3.4 Distributed Tracing for Serverless Functions
   970→
   971→Vercel Functions have limited tracing support. Implement lightweight request tracing:
   972→
   973→**Request ID propagation -- `lib/server/requestContext.ts`:**
   974→
   975→```typescript
   976→export function getRequestId(request: Request): string {
   977→  return (
   978→    request.headers.get("x-vercel-id") ??
   979→    request.headers.get("x-request-id") ??
   980→    crypto.randomUUID()
   981→  );
   982→}
   983→
   984→export function withRequestTiming<T>(
   985→  label: string,
   986→  fn: () => Promise<T>,
   987→  requestId: string,
   988→): Promise<T> {
   989→  const start = performance.now();
   990→  return fn().then(
   991→    (result) => {
   992→      const duration = Math.round(performance.now() - start);
   993→      console.log(JSON.stringify({
   994→        level: "info",
   995→        requestId,
   996→        trace: label,
   997→        duration_ms: duration,
   998→        timestamp: new Date().toISOString(),
   999→      }));
  1000→      return result;
  1001→    },
  1002→    (error) => {
  1003→      const duration = Math.round(performance.now() - start);
  1004→      console.error(JSON.stringify({
  1005→        level: "error",
  1006→        requestId,
  1007→        trace: label,
  1008→        duration_ms: duration,
  1009→        error: error instanceof Error ? error.message : String(error),
  1010→        timestamp: new Date().toISOString(),
  1011→      }));
  1012→      throw error;
  1013→    },
  1014→  );
  1015→}
  1016→```

  1041→
  1042→The current `/api/health/index.ts` only checks `SELECT 1`. Enhance it:
  1043→
  1044→**Enhanced `api/health/index.ts`:**
  1045→
  1046→```typescript
  1047→import { query } from "../../lib/server/db.js";
  1048→
  1049→interface HealthCheckResult {
  1050→  status: "ok" | "degraded" | "error";
  1051→  db: "ok" | "error";
  1052→  db_latency_ms: number | null;
  1053→  blob: "ok" | "unchecked";
  1054→  version: string;
  1055→  region: string | null;
  1056→  uptime_s: number;
  1057→  ts: string;
  1058→}
  1059→
  1060→const startTime = Date.now();
  1061→
  1062→export async function GET(request: Request): Promise<Response> {
  1063→  const isDeepCheck = new URL(request.url).searchParams.get("deep") === "true";
  1064→  const result: HealthCheckResult = {
  1065→    status: "ok",
  1066→    db: "error",
  1067→    db_latency_ms: null,
  1068→    blob: "unchecked",
  1069→    version: process.env.VERCEL_GIT_COMMIT_SHA?.slice(0, 7) ?? "dev",
  1070→    region: process.env.VERCEL_REGION ?? null,
  1071→    uptime_s: Math.round((Date.now() - startTime) / 1000),
  1072→    ts: new Date().toISOString(),
  1073→  };
  1074→
  1075→  // Database check
  1076→  const dbStart = performance.now();
  1077→  try {
  1078→    await query("SELECT 1");
  1079→    result.db = "ok";
  1080→    result.db_latency_ms = Math.round(performance.now() - dbStart);
  1081→  } catch {
  1082→    result.status = "error";
  1083→    result.db_latency_ms = Math.round(performance.now() - dbStart);
  1084→  }
  1085→
  1086→  // Deep check: verify table accessibility
  1087→  if (isDeepCheck && result.db === "ok") {
  1088→    try {
  1089→      await query("SELECT count(*) FROM point_events WHERE created_at > now() - interval '1 day'");
  1090→    } catch {
  1091→      result.status = "degraded";
  1092→    }
  1093→  }
  1094→
  1095→  // DB latency warning
  1096→  if (result.db_latency_ms !== null && result.db_latency_ms > 5000) {
  1097→    result.status = result.status === "ok" ? "degraded" : result.status;
  1098→  }
  1099→
  1100→  const httpStatus = result.status === "error" ? 503 : 200;
  1101→  return new Response(JSON.stringify(result), {
  1102→    status: httpStatus,
  1103→    headers: {
  1104→      "Content-Type": "application/json",
  1105→      "Cache-Control": "no-cache, no-store",
  1106→    },
  1107→  });
  1108→}
  1109→```

  1195→

  1272→### 4.2 Blob Storage Backup
  1273→
  1274→**Current:** Photos stored in Vercel Blob under `submissions/{eventId}-{timestamp}.{ext}`.
  1275→
  1276→**Backup strategy:**
  1277→1. Vercel Blob has no built-in backup. Photos are referenced by URL in `point_events.photo_url`.
  1278→2. Nightly script lists all blob URLs from `point_events.photo_url` and verifies accessibility.
  1279→3. Phase 2: Copy blobs older than 30 days to Cloudflare R2 as cold archive.
  1280→
  1281→**Blob verification script:**
  1282→
  1283→```bash
  1284→#!/bin/bash
  1285→# Verify all photo URLs are accessible
  1286→psql "$ADL_POSTGRES_URL" -t -c "SELECT DISTINCT photo_url FROM point_events WHERE photo_url IS NOT NULL AND photo_url != ''" | while read -r url; do
  1287→  url=$(echo "$url" | xargs)  # trim whitespace
  1288→  if [ -z "$url" ]; then continue; fi
  1289→  STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$url" --max-time 10)
  1290→  if [ "$STATUS" != "200" ]; then
  1291→    echo "[blob-check] MISSING: $url (HTTP $STATUS)"
  1292→  fi
  1293→done

  1324→### 4.5 Disaster Recovery Runbook
  1325→
  1326→**DR Scenario 1: Vercel Outage (app unreachable)**
  1327→
  1328→```
  1329→Time 0:00 - Detect (health check fails, Vercel status page confirms)
  1330→  Action: Nothing to do. Wait for Vercel to recover.
  1331→  Mitigation: Vercel SLA is 99.99% for Pro tier.
  1332→  If > 1 hour:
  1333→    1. Redirect DNS to a static "maintenance" page on Cloudflare Pages
  1334→    2. Communicate via status page / Twitter
  1335→
  1336→Time 0:15 - Notify users
  1337→  Action: Post on ADL status page: "We are experiencing intermittent issues.
  1338→          Your offline submissions are safely queued and will sync when
  1339→          service resumes."
  1340→```
  1341→
  1342→**DR Scenario 2: Database Corruption / Accidental Data Loss**
  1343→
  1344→```
  1345→Time 0:00 - Detect (API returns 500, health check shows db: error)
  1346→  Action: Verify via Supabase dashboard
  1347→
  1348→Time 0:05 - Assess scope
  1349→  Query: SELECT count(*) FROM point_events;
  1350→  Query: SELECT max(created_at) FROM point_events;
  1351→  Compare with last backup checkpoint count.
  1352→
  1353→Time 0:15 - If data loss confirmed:
  1354→  Option A (< 1 hour of data lost, Supabase Pro):
  1355→    1. Use Supabase PITR to restore to timestamp just before corruption
  1356→    2. supabase db restore --to <timestamp>
  1357→
  1358→  Option B (> 1 hour of data lost, Free tier):
  1359→    1. Restore from nightly CSV backup:
  1360→       psql $POSTGRES_URL -c "BEGIN;"
  1361→       psql $POSTGRES_URL -c "\COPY point_events FROM 'point_events.csv' WITH CSV HEADER"
  1362→       psql $POSTGRES_URL -c "COMMIT;"
  1363→    2. Data between backup and corruption is lost.
  1364→    3. Users' offline queues may contain some of the lost data --
  1365→       it will re-sync when they come online.
  1366→
  1367→Time 0:30 - Verify restoration
  1368→  Run: SELECT count(*) FROM point_events;
  1369→  Run: /api/health?deep=true
  1370→  Run: Manual test: create a submission via the app
  1371→
  1372→Time 1:00 - Post-incident
  1373→  Write incident report. Update RPO assessment.
  1374→```
  1375→
  1376→**DR Scenario 3: Supabase Region Outage (full database unavailable)**
  1377→
  1378→```
  1379→Time 0:00 - Detect (all DB queries fail, Supabase status confirms)
  1380→
  1381→Time 0:05 - Enable offline-only mode:
  1382→  1. Set Edge Config flag: { "maintenance_mode": true }
  1383→  2. App detects maintenance_mode and switches to offline-only UI
  1384→  3. All submissions queue to IndexedDB
  1385→
  1386→Time 0:10 - Communicate:
  1387→  "We are temporarily in offline mode. Your submissions are saved locally
  1388→   and will sync automatically when database service resumes."
  1389→
  1390→Time ?:?? - Supabase recovers:
  1391→  1. Verify connectivity: curl /api/health
  1392→  2. Clear maintenance_mode flag in Edge Config
  1393→  3. Users' apps will auto-flush offline queues
  1394→
  1395→Post-incident:
  1396→  Evaluate multi-region database setup if outage > 4 hours.
  1397→```

  1664→```

  1666→======================
  1667→
  1668→DETECTION
  1669→=========
  1670→Signals:
  1671→- Vercel dashboard shows function invocation spike
  1672→- /api/health response time > 5 seconds
  1673→- Database connection count approaching max (5 in current config)
  1674→
  1675→IMMEDIATE ACTIONS (0-5 minutes)
  1676→===============================
  1677→1. Increase database connection pool
  1678→   Update Vercel env var:

  1680→

  1686→
  1687→3. Enable aggressive caching
  1688→   If read traffic is the bottleneck:
  1689→   - Increase s-maxage on public endpoints
  1690→   - POST /api/submissions cache is never used (correct)
  1691→
  1692→MEDIUM-TERM ACTIONS (5-30 minutes)
  1693→===================================
  1694→4. If database is the bottleneck:

  1765→### 6.3 API Rate Limiting Implementation
  1766→
  1767→**No rate limiting currently exists.** This is the highest-priority security gap per the system design doc (section 2.2).
  1768→
  1769→**Implementation with Vercel Edge Middleware + Upstash Redis:**
  1770→
  1771→```bash
  1772→npm install @upstash/ratelimit @upstash/redis
  1773→```
  1774→
  1775→**`middleware.ts` (Vercel Edge Middleware at project root):**
  1776→
  1777→```typescript
  1778→import { Ratelimit } from "@upstash/ratelimit";
  1779→import { Redis } from "@upstash/redis";
  1780→
  1781→const redis = new Redis({
  1782→  url: process.env.UPSTASH_REDIS_REST_URL!,
  1783→  token: process.env.UPSTASH_REDIS_REST_TOKEN!,
  1784→});
  1785→
  1786→const rateLimiters = {
  1787→  submission_write: new Ratelimit({
  1788→    redis,
  1789→    limiter: Ratelimit.slidingWindow(10, "1 m"),  // 10 submissions/min
  1790→    prefix: "rl:sub:write",
  1791→  }),
  1792→  auth_register: new Ratelimit({
  1793→    redis,
  1794→    limiter: Ratelimit.slidingWindow(5, "1 m"),   // 5 registrations/min
  1795→    prefix: "rl:auth:reg",
  1796→  }),
  1797→  auth_signin: new Ratelimit({
  1798→    redis,
  1799→    limiter: Ratelimit.slidingWindow(10, "1 m"),  // 10 login attempts/min
  1800→    prefix: "rl:auth:sign",
  1801→  }),
  1802→  api_read: new Ratelimit({
  1803→    redis,
  1804→    limiter: Ratelimit.slidingWindow(100, "1 m"), // 100 reads/min
  1805→    prefix: "rl:api:read",
  1806→  }),
  1807→};
  1808→
  1809→function getClientIp(request: Request): string {
  1810→  return (
  1811→    request.headers.get("x-forwarded-for")?.split(",")[0]?.trim() ??
  1812→    request.headers.get("x-real-ip") ??
  1813→    "unknown"
  1814→  );
  1815→}
  1816→
  1817→function getRateLimiter(pathname: string, method: string) {
  1818→  if (pathname === "/api/submissions" && method === "POST") return rateLimiters.submission_write;
  1819→  if (pathname === "/api/auth/register" && method === "POST") return rateLimiters.auth_register;
  1820→  if (pathname.startsWith("/api/auth/") && method === "POST") return rateLimiters.auth_signin;
  1821→  if (pathname.startsWith("/api/") && method === "GET") return rateLimiters.api_read;
  1822→  return null;
  1823→}
  1824→
  1825→export default async function middleware(request: Request): Promise<Response | undefined> {
  1826→  const url = new URL(request.url);
  1827→  const limiter = getRateLimiter(url.pathname, request.method);
  1828→  if (!limiter) return undefined;  // pass through
  1829→
  1830→  const ip = getClientIp(request);
  1831→  const { success, limit, remaining, reset } = await limiter.limit(ip);
  1832→
  1833→  if (!success) {
  1834→    return new Response(
  1835→      JSON.stringify({ error: "Rate limit exceeded. Please try again later." }),
  1836→      {
  1837→        status: 429,
  1838→        headers: {
  1839→          "Content-Type": "application/json",
  1840→          "X-RateLimit-Limit": String(limit),
  1841→          "X-RateLimit-Remaining": "0",
  1842→          "X-RateLimit-Reset": String(reset),
  1843→          "Retry-After": String(Math.ceil((reset - Date.now()) / 1000)),
  1844→        },
  1845→      }
  1846→    );
  1847→  }
  1848→
  1849→  return undefined;  // pass through, Vercel adds rate limit headers
  1850→}
  1851→
  1852→export const config = {
  1853→  matcher: ["/api/:path*"],
  1854→};
  1855→```

  2127→### 7.5 Image Optimization Pipeline
  2128→
  2129→**Current:** Photos uploaded as-is (JPEG/PNG/WebP/HEIC up to 8 MB). No server-side optimization.
  2130→
  2131→**Phase 1 (Now) -- Client-side optimization before upload:**
  2132→
  2133→The client should compress images before base64-encoding. Add to `lib/client/imageUtils.ts`:
  2134→
  2135→```typescript
  2136→export async function compressImage(
  2137→  file: File,
  2138→  maxWidth: number = 1200,
  2139→  quality: number = 0.8,
  2140→): Promise<Blob> {
  2141→  return new Promise((resolve, reject) => {
  2142→    const img = new Image();
  2143→    const url = URL.createObjectURL(file);
  2144→    img.onload = () => {
  2145→      URL.revokeObjectURL(url);
  2146→      const canvas = document.createElement("canvas");
  2147→      let { width, height } = img;
  2148→      if (width > maxWidth) {
  2149→        height = Math.round((height * maxWidth) / width);
  2150→        width = maxWidth;
  2151→      }
  2152→      canvas.width = width;
  2153→      canvas.height = height;
  2154→      const ctx = canvas.getContext("2d");
  2155→      if (!ctx) return reject(new Error("Canvas context unavailable"));
  2156→      ctx.drawImage(img, 0, 0, width, height);
  2157→      canvas.toBlob(
  2158→        (blob) => {
  2159→          if (!blob) return reject(new Error("Compression failed"));
  2160→          resolve(blob);
  2161→        },
  2162→        "image/jpeg",
  2163→        quality,
  2164→      );
  2165→    };
  2166→    img.onerror = () => reject(new Error("Failed to load image"));
  2167→    img.src = url;
  2168→  });
  2169→}
  2170→```
  2171→
  2172→**Impact:** Reduces average upload payload from ~3-5 MB to ~200-500 KB. Critical for 2G/3G networks in Africa.
  2173→
