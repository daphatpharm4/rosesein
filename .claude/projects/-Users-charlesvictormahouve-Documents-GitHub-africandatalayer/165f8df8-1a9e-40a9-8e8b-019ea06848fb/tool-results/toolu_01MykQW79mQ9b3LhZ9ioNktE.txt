    77→### 1.2 Spoofing
    78→
    79→#### S-1: Credential Stuffing on Registration and Login
    80→
    81→**Risk:** HIGH
    82→
    83→The registration endpoint (`api/auth/register.ts`) and credential login (`api/auth/auth.ts`) have zero rate limiting. An attacker can enumerate valid accounts by observing 409 (user exists) vs 201 (created) responses on the registration endpoint, then launch credential stuffing against the login endpoint.
    84→
    85→**Current state:**
    86→- Registration: No CAPTCHA, no rate limit, leaks user existence via 409 status (`api/auth/register.ts:39`)
    87→- Login: Auth.js credential flow with no lockout, no rate limit (`api/auth/auth.ts:23-58`)
    88→- Password policy: Only minimum 8 characters (`api/auth/register.ts:31`), no complexity requirements
    89→
    90→**Attack scenario:**
    91→1. Attacker scripts POST requests to `/api/auth/register` with common email addresses
    92→2. 409 responses reveal which emails are registered
    93→3. Attacker targets confirmed emails with credential stuffing against `/api/auth/callback/credentials`
    94→4. Weak 8-character minimum means many passwords are guessable
    95→
    96→#### S-2: Admin Account Hardcoded Pattern
    97→
    98→**Risk:** CRITICAL
    99→
   100→Admin authentication uses a single shared `ADMIN_EMAIL` + `ADMIN_PASSWORD` environment variable pair (`api/auth/auth.ts:36-48`). The code explicitly supports plaintext passwords as a fallback:
   101→
   102→```typescript
   103→// api/auth/auth.ts:42-44
   104→if (adminPassword.startsWith("$2")) {
   105→  adminMatch = await bcrypt.compare(password, adminPassword);
   106→} else {
   107→  // Fallback for plain-text password (legacy)
   108→  adminMatch = password === adminPassword;
   109→}
   110→```
   111→
   112→The `.env` file committed to the repository contains `ADMIN_PASSWORD=test1234` in plaintext. Even in production, a single admin credential shared across all admin sessions creates a single point of compromise.
   113→
   114→#### S-3: Session Token Security
   115→
   116→**Risk:** MEDIUM
   117→
   118→Session management uses Auth.js JWT strategy (`api/auth/auth.ts:103`) with appropriate cookie settings:
   119→- `httpOnly: true` (prevents XSS-based cookie theft)
   120→- `sameSite: "lax"` (partial CSRF protection)
   121→- `secure: true` in production (enforced by `isSecureRequest()` in `lib/auth.ts:8-12`)
   122→
   123→However, JWT tokens have no explicit expiration configuration. Auth.js defaults to 30-day session tokens. Combined with no token rotation and no server-side session revocation capability, a stolen JWT remains valid for the full 30-day window.
   124→
   125→#### S-4: Bearer Token Acceptance Without Scope Validation
   126→
   127→**Risk:** MEDIUM
   128→
   129→The `getAuthToken` function (`lib/auth.ts:24-31`) accepts both cookie-based and Bearer token authentication. The Bearer path decodes the JWT directly without validating audience, issuer, or scope claims. Any valid JWT signed with `AUTH_SECRET` grants full access regardless of its original intended use.
   130→
   131→### 1.3 Tampering
   132→
   133→#### T-1: Offline Queue Client-Side Tampering
   134→
   135→**Risk:** HIGH
   136→
   137→The offline queue (`lib/client/offlineQueue.ts`) stores submission payloads in IndexedDB. Users with DevTools access can:
   138→1. Modify GPS coordinates in queued submissions
   139→2. Alter category or details fields
   140→3. Replace photo data with stock images
   141→4. Change the `idempotencyKey` to replay submissions
   142→
   143→The idempotency key is generated client-side (`offlineQueue.ts:110`) and sent as an option to the sync function (`offlineQueue.ts:233`), but **the server never checks it**. The POST handler in `api/submissions/index.ts` has no idempotency logic whatsoever.
   144→
   145→#### T-2: Photo Metadata Spoofing
   146→
   147→**Risk:** HIGH
   148→
   149→While the fraud detection system (`lib/server/submissionFraud.ts`) extracts and validates EXIF GPS data, EXIF metadata is trivially spoofable. An attacker can:
   150→1. Use tools like `exiftool` to set arbitrary GPS coordinates in a photo
   151→2. Set `DateTimeOriginal` to fake the capture time
   152→3. Set `Make`/`Model` to impersonate a different device
   153→
   154→The system does compare photo GPS with submission GPS and IP-based geolocation, but if the attacker controls all three (spoofed EXIF, spoofed browser geolocation, VPN for IP), all checks pass.
   155→
   156→#### T-3: PUT Endpoint Bypasses Photo Requirement
   157→
   158→**Risk:** HIGH
   159→
   160→The `PUT` handler in `api/submissions/[id].ts:68-113` creates an `ENRICH_EVENT` without requiring a photo upload:
   161→
   162→```typescript
   163→// api/submissions/[id].ts:100
   164→photoUrl: typeof body?.photoUrl === "string" ? body.photoUrl : undefined,
   165→```
   166→
   167→This accepts an arbitrary URL as the photo without validation, download, or storage. An attacker can:
   168→1. Point `photoUrl` to any external URL
   169→2. Submit enrichment events without photographic proof
   170→3. Inject URLs that could be used for tracking (e.g., unique URLs per victim to detect when admins view submissions)
   171→
   172→#### T-4: XP Manipulation via Rapid Submissions
   173→
   174→**Risk:** MEDIUM
   175→
   176→The XP award logic (`api/submissions/index.ts:677-681`) unconditionally adds 5 XP per submission with no duplicate check, no rate limit, and no cap:
   177→
   178→```typescript
   179→profile.XP = (profile.XP ?? 0) + BASE_EVENT_XP;
   180→await upsertUserProfile(auth.id, profile);
   181→```
   182→
   183→An attacker can script hundreds of submissions to farm XP for the rewards catalog and leaderboard.
   184→
   185→### 1.4 Repudiation
   186→
   187→#### R-1: Insufficient Audit Trail for Administrative Actions
   188→
   189→**Risk:** MEDIUM
   190→
   191→Admin operations (delete events via `api/submissions/[id].ts:115-144`, viewing admin_events) are not logged to any audit trail. The only log statement in the submission flow (`api/submissions/index.ts:673-675`) is gated behind `NODE_ENV !== "production"`, meaning production submissions produce no server-side log.
   192→
   193→Admin actions that should be logged but are not:
   194→- Event deletion
   195→- Admin event view access
   196→- User profile modifications
   197→- Map scope changes
   198→
   199→#### R-2: No Contributor Attribution Integrity
   200→
   201→**Risk:** MEDIUM
   202→
   203→The `userId` field on events comes from the JWT token (`api/submissions/index.ts:646`). While the JWT is server-signed, there is no mechanism to prove that the *physical person* who submitted data is the token holder. For compliance and data quality, this means:
   204→- No digital signature on individual submissions
   205→- No device attestation
   206→- No biometric verification

   229→#### I-2: Gemini API Key Baked into Client Bundle
   230→
   231→**Risk:** HIGH
   232→
   233→`vite.config.ts:20-21` injects the Gemini API key into the client-side JavaScript bundle:
   234→
   235→```typescript
   236→define: {
   237→  'process.env.API_KEY': JSON.stringify(env.GEMINI_API_KEY),
   238→  'process.env.GEMINI_API_KEY': JSON.stringify(env.GEMINI_API_KEY)
   239→},
   240→```
   241→
   242→This means anyone viewing the production JavaScript source can extract the API key. The key is then used directly in `lib/gemini.ts:4`:
   243→
   244→```typescript
   245→const ai = new GoogleGenAI({ apiKey: process.env.API_KEY });
   246→```
   247→
   248→Impact: An attacker can use the API key for their own Gemini API calls, incurring costs on ADL's Google Cloud billing account.
   249→
   250→#### I-3: Leaderboard Exposes User Identifiers
   251→
   252→**Risk:** MEDIUM
   253→
   254→The leaderboard endpoint (`api/leaderboard/index.ts:88-99`) returns `userId` for every top contributor. The `userId` is typically an email address (since email is the primary identifier). This is returned on a public, unauthenticated endpoint with cache headers set to public (`LEADERBOARD_CACHE_CONTROL = "public, s-maxage=30, stale-while-revalidate=300"`).
   255→
   256→Combined with the registration endpoint user enumeration (S-1), this gives attackers a ready-made target list.
   257→
   258→#### I-4: Password Hashes Returned in User Profile
   259→
   260→**Risk:** HIGH
   261→
   262→The `rowToUserProfile` function (`lib/server/storage/postgresStore.ts:71-86`) includes `passwordHash` in the returned `UserProfile` object:
   263→
   264→```typescript
   265→passwordHash: typeof row.password_hash === "string" ? row.password_hash : undefined,
   266→```
   267→
   268→The `GET /api/user` endpoint (`api/user/index.ts:21`) returns the full profile object:
   269→
   270→```typescript
   271→return jsonResponse(profile, { status: 200 });
   272→```
   273→
   274→This means the bcrypt hash is sent to the client in the user profile response. While bcrypt hashes are computationally expensive to crack, they should never leave the server.
   275→
   276→#### I-5: Unhandled Errors Propagate Stack Traces
   277→
   278→**Risk:** MEDIUM
   279→
   280→Several API handlers use `throw error` as the final fallback (`api/submissions/index.ts:449`, `api/submissions/[id].ts:48`, etc.). When `isStorageUnavailableError` returns false, the raw error is thrown and Vercel's default error handler may expose stack traces, internal paths, and implementation details in the response.
   281→
   282→#### I-6: Health Endpoint Exposes Database Connectivity
   283→
   284→**Risk:** LOW
   285→
   286→The health endpoint (`api/health/index.ts`) reveals whether the database is reachable. While useful for monitoring, it gives attackers reconnaissance information about infrastructure state and can be used to detect when database attacks are effective.
   287→
   288→### 1.6 Denial of Service
   289→
   290→#### D-1: No Rate Limiting on Any Endpoint
   291→
   292→**Risk:** HIGH
   293→
   294→There is zero rate limiting across the entire API surface. No middleware, no Vercel configuration, no application-level throttling. Every endpoint is vulnerable to abuse:
   295→
   296→- `POST /api/auth/register` -- mass account creation
   297→- `POST /api/auth/callback/credentials` -- brute force login
   298→- `POST /api/submissions` -- resource exhaustion (each submission triggers photo upload to Vercel Blob, EXIF parsing, IP geolocation lookup, and database write)
   299→- `GET /api/submissions` -- database query amplification (loads ALL events on every request)
   300→- `GET /api/leaderboard` -- same full-table scan pattern
   301→
   302→#### D-2: Unbounded Full-Table Scans
   303→
   304→**Risk:** HIGH
   305→
   306→The `getPointEvents` function (`lib/server/storage/postgresStore.ts:289-298`) loads every row from the `point_events` table on every GET request:
   307→
   308→```sql
   309→select id, point_id, event_type, user_id, category, latitude, longitude,
   310→       details, photo_url, created_at, source, external_id
   311→from point_events
   312→order by created_at asc
   313→```
   314→
   315→At scale (10K+ events with JSONB `details` containing fraud metadata, photo URLs, and client device info), this will:
   316→1. Exhaust the 256MB/512MB serverless function memory
   317→2. Saturate the database connection pool (max 5 connections, `lib/server/db.ts:121`)
   318→3. Timeout (10-second query timeout, `lib/server/db.ts:19`)
   319→
   320→#### D-3: Photo Upload Resource Exhaustion
   321→
   322→**Risk:** MEDIUM
   323→
   324→The submission endpoint accepts up to 8MB base64-encoded images (`MAX_IMAGE_BYTES`, `api/submissions/index.ts:56`). Since base64 encoding increases payload size by ~33%, the actual HTTP request can be ~11MB. The endpoint processes two photos per submission (primary + secondary), each requiring:
   325→1. Base64 decode to Buffer
   326→2. EXIF parsing via `exifr`
   327→3. Upload to Vercel Blob
   328→
   329→With no rate limit, an attacker can send hundreds of 11MB requests simultaneously, exhausting Vercel's concurrent execution limits and Blob storage quota.
   330→
   331→#### D-4: External IP Geolocation Dependency
   332→
   333→**Risk:** LOW
   334→
   335→The `fetchIpLocation` function (`api/submissions/index.ts:223-239`) calls `ipapi.co` for every submission. The 3-second timeout means that if ipapi.co is slow or unavailable, submissions are delayed. If ipapi.co rate-limits the server IP, all geolocation lookups fail silently (returning `null`), potentially allowing submissions without proper location validation.
   336→
   337→### 1.7 Elevation of Privilege
   338→
   339→#### E-1: Admin Role Determination via Environment Variable Email Match
   340→
   341→**Risk:** HIGH
   342→
   343→Admin status is determined solely by matching the user's email against `ADMIN_EMAIL` (`api/auth/auth.ts:36-48`, `api/auth/auth.ts:122`, `api/auth/auth.ts:180-186`). The admin flag is embedded in the JWT token:
   344→
   345→```typescript
   346→// api/auth/auth.ts:182-184
   347→if (adminEmail && email && adminEmail === email) {
   348→  (token as { isAdmin?: boolean }).isAdmin = true;
   349→}
   350→```
   351→
   352→**Problem 1:** Only one admin email is supported. There is no role hierarchy, no group-based access, no way to have multiple admins without sharing the single `ADMIN_EMAIL` credential.
   353→
   354→**Problem 2:** The JWT `isAdmin` claim is trusted by all endpoints without re-verification against the database. If `ADMIN_EMAIL` is changed (e.g., to revoke admin), existing JWTs with `isAdmin: true` remain valid until they expire (up to 30 days).
   355→
   356→#### E-2: User Profile Self-Escalation Blocked but Fragile
   357→
   358→**Risk:** MEDIUM
   359→
   360→The `PUT /api/user` endpoint (`api/user/index.ts:66-73`) does check that non-admin users cannot set `mapScope` beyond `bonamoussadi`:
   361→
   362→```typescript
   363→if (!profile.isAdmin && nextScope !== "bonamoussadi") {
   364→  return errorResponse("Only admin users can unlock map scope", 403);
   365→}
   366→```
   367→
   368→However, the `profile.isAdmin` check reads from the database, while admin status is determined by the JWT `isAdmin` claim and synced to the database in the `GET /api/user` handler (`api/user/index.ts:24-27`). If there is any race condition or inconsistency between the JWT claim and database state, escalation could occur.
   369→
   370→#### E-3: IDOR in Submission Detail View
   371→
   372→**Risk:** MEDIUM
   373→
   374→The `GET /api/submissions/[id]` endpoint (`api/submissions/[id].ts:58-63`) allows any authenticated user to view projected point data for any `pointId`:
   375→
   376→```typescript
   377→const point = points.find((item) => item.pointId === id || item.id === id);
   378→if (point) return jsonResponse(point, { status: 200 });
   379→```
   380→
   381→Only the `view=event` path checks ownership (`canViewEventDetail`). The default path returns any projected point to any authenticated user, potentially including details from other users' enrichment events aggregated into the projection.
   382→
   383→#### E-4: PUT Endpoint Missing Admin/Owner Check for Enrichment
   384→
   385→**Risk:** MEDIUM
   386→
   387→The `PUT /api/submissions/[id]` handler (`api/submissions/[id].ts:68-113`) requires authentication but does not verify that the user owns the target submission or has admin privileges. Any authenticated user can create an `ENRICH_EVENT` for any existing point, potentially overwriting legitimate data.

   428→4. Revoke and regenerate: Supabase database password, Google OAuth credentials, Vercel API token, AUTH_SECRET
   429→5. Change Supabase database password from Supabase dashboard
   430→6. Set all secrets exclusively via Vercel Environment Variables dashboard
   431→7. Audit git log for any forks or clones that may have captured the secrets

   471→#### VULN-003: Plaintext Admin Password Support
   472→
   473→| Field | Value |
   474→|-------|-------|
   475→| **Severity** | CRITICAL |
   476→| **File** | `api/auth/auth.ts:42-46` |
   477→| **CVSS** | 8.1 |
   478→| **Priority** | FIX NOW |
   479→
   480→**Description:** The admin authentication code has a fallback path that compares passwords in plaintext when `ADMIN_PASSWORD` does not start with `$2` (bcrypt prefix). The current `.env` contains `ADMIN_PASSWORD=test1234`, which triggers this plaintext path.
   481→
   482→**Proof of concept:**
   483→```bash
   484→curl -X POST https://adl.vercel.app/api/auth/callback/credentials \
   485→  -d "identifier=test@example.com&password=test1234&csrfToken=..."
   486→```
   487→
   488→**Remediation:**

   614→```json
   615→// vercel.json -- Add Vercel Firewall rules:
   616→{

   618→    {
   619→      "source": "/api/auth/(.*)",

   629→```typescript
   630→// lib/server/rateLimit.ts
   631→import { query } from "./db.js";
   632→
   633→export async function checkRateLimit(
   634→  key: string,
   635→  windowMs: number,
   636→  maxAttempts: number
   637→): Promise<{ allowed: boolean; remaining: number }> {
   638→  const windowStart = new Date(Date.now() - windowMs).toISOString();
   639→  const result = await query<{ count: string }>(
   640→    "SELECT COUNT(*) as count FROM rate_limit_events WHERE key = $1 AND created_at > $2",
   641→    [key, windowStart]
   642→  );
   643→  const count = parseInt(result.rows[0]?.count ?? "0", 10);
   644→  if (count >= maxAttempts) return { allowed: false, remaining: 0 };
   645→
   646→  await query(
   647→    "INSERT INTO rate_limit_events (key, created_at) VALUES ($1, NOW())",
   648→    [key]
   649→  );
   650→  return { allowed: true, remaining: maxAttempts - count - 1 };
   651→}
   652→```

   655→- Login: 5 attempts per email per 15 minutes
   656→- Registration: 3 accounts per IP per hour
   657→- Submissions: 10 per user per hour

   951→### 4.3 Response Procedures
   952→
   953→#### Phase 1: Detection and Triage (0-15 minutes for P1)
   954→
   955→1. **Identify** the incident source (monitoring alert, user report, security scan, external notification)
   956→2. **Classify** the priority using the matrix above
   957→3. **Assign** an Incident Commander
   958→4. **Create** an incident channel (Slack/Discord/WhatsApp group)
   959→5. **Notify** the IR team per the priority response time
   960→
   961→#### Phase 2: Containment (15 min - 4 hours for P1)
   962→
   963→1. **Isolate** affected components:
   964→   - Rotate compromised credentials immediately
   965→   - If database is compromised: change Supabase password, update `POSTGRES_URL` in Vercel
   966→   - If admin account is compromised: change `ADMIN_EMAIL` and `AUTH_SECRET` to invalidate all JWTs
   967→   - If API key is abused: revoke key in Google Cloud console
   968→2. **Preserve evidence** before making changes:
   969→   - Export Vercel function logs
   970→   - Export database audit logs from Supabase
   971→   - Screenshot relevant dashboard states
   972→   - Save HTTP access logs
   973→3. **Block** the attack vector:
   974→   - Add IP to Vercel Firewall blocklist if applicable
   975→   - Disable affected endpoint if necessary (via Vercel redirect rule)
   976→
   977→#### Phase 3: Eradication (4-24 hours for P1)
   978→
   979→1. **Root cause analysis**: Identify how the attacker gained access
   980→2. **Patch** the vulnerability (code fix, config change, or dependency update)
   981→3. **Deploy** the fix via standard CI/CD pipeline (do not skip tests)
   982→4. **Verify** the fix in staging/preview before production
   983→
   984→#### Phase 4: Recovery (24-72 hours for P1)
   985→
   986→1. **Restore** service to full operation
   987→2. **Monitor** closely for recurrence (increased logging, manual checks)
   988→3. **Communicate** resolution to affected users if applicable
   989→4. **Update** credentials rotation schedule
   990→
   991→#### Phase 5: Post-Incident Review (within 5 business days)
   992→
   993→1. **Timeline**: Minute-by-minute reconstruction of the incident
   994→2. **Impact assessment**: Data affected, users affected, financial impact
   995→3. **Root cause**: Technical and process failures that allowed the incident
   996→4. **Action items**: Specific, assignable, time-bound improvements
   997→5. **Lessons learned**: What worked well, what did not

  1025→### 4.6 Specific Playbooks
  1026→
  1027→#### Playbook A: Credential Compromise
  1028→
  1029→**Trigger:** Suspected or confirmed compromise of any credential (database password, API key, OAuth secret, admin password, AUTH_SECRET).
  1030→
  1031→**Steps:**
  1032→1. Identify which credentials are compromised
  1033→2. Rotate the compromised credentials immediately:
  1034→   - `AUTH_SECRET`: Generate new value, update in Vercel env vars. This invalidates ALL user sessions.
  1035→   - `POSTGRES_URL`: Change password in Supabase dashboard, update connection string in Vercel env vars
  1036→   - `GOOGLE_CLIENT_SECRET`: Regenerate in Google Cloud Console, update in Vercel env vars
  1037→   - `VERCEL_API_TOKEN`: Regenerate in Vercel dashboard
  1038→   - `ADMIN_PASSWORD`: Generate new bcrypt hash, update in Vercel env vars
  1039→   - `BLOB_READ_WRITE_TOKEN`: Regenerate in Vercel Blob settings
  1040→3. Audit database for unauthorized changes:
  1041→   ```sql
  1042→   -- Check for recent suspicious user profile changes
  1043→   SELECT * FROM user_profiles WHERE updated_at > NOW() - INTERVAL '24 hours' ORDER BY updated_at DESC;
  1044→   -- Check for suspicious point events
  1045→   SELECT * FROM point_events WHERE created_at > NOW() - INTERVAL '24 hours' ORDER BY created_at DESC;
  1046→   ```
  1047→4. Check Vercel deployment history for unauthorized deployments
  1048→5. Review git history for unauthorized commits
  1049→6. If `AUTH_SECRET` was compromised, all users must re-authenticate (sessions are invalidated)
  1050→
  1051→#### Playbook B: Data Breach
  1052→
  1053→**Trigger:** Confirmed unauthorized access to user PII (emails, phone numbers, passwords) or submission data.
  1054→
  1055→**Steps:**
  1056→1. Determine scope: Which tables were accessed? Which records? Which time period?
  1057→2. Contain: Revoke the access vector (rotate credentials, block IP, disable endpoint)
  1058→3. Assess PII impact:
  1059→   - User emails and phone numbers: Notify affected users
  1060→   - Password hashes: If bcrypt cost 10, assess cracking feasibility (generally low risk but notify users to change passwords)
  1061→   - Submission data: Assess if location data constitutes PII under applicable law
  1062→4. Legal assessment:
  1063→   - GDPR applicability (EU users)
  1064→   - Cameroon Data Protection Act (Law No. 2010/012)
  1065→   - Nigeria Data Protection Regulation (NDPR)
  1066→5. Notify supervisory authority within 72 hours if required
  1067→6. Notify affected users with specific remediation steps
  1068→7. Offer password reset for all affected accounts
  1069→
  1070→#### Playbook C: DDoS Attack
  1071→
  1072→**Trigger:** Sustained high traffic causing service degradation or outage.
  1073→
  1074→**Steps:**
  1075→1. Verify it is a DDoS and not a legitimate traffic spike
  1076→2. Enable Vercel's built-in DDoS protection (contact Vercel support if on Hobby plan)
  1077→3. If attack targets specific endpoint:
  1078→   - Add Vercel Firewall rules to block attacking IPs/ranges
  1079→   - Temporarily add rate limiting via Edge Middleware
  1080→   - If targeting `/api/submissions`, temporarily increase function timeout and memory
  1081→4. If attack is volumetric:
  1082→   - Enable Vercel's attack challenge mode if available
  1083→   - Contact Vercel support for CDN-level mitigation
  1084→5. Monitor database connection pool usage (`SELECT count(*) FROM pg_stat_activity`)
  1085→6. Post-attack: Implement rate limiting (VULN-007) to prevent recurrence
  1086→
  1087→#### Playbook D: Fraudulent Data Injection at Scale
  1088→
  1089→**Trigger:** Discovery of large-scale fake submissions (bot-generated points, fabricated locations, stock photos).
  1090→
  1091→**Steps:**
  1092→1. Identify the fraudulent submissions:
  1093→   ```sql
  1094→   -- Find users with abnormally high submission rates
  1095→   SELECT user_id, COUNT(*) as count, MIN(created_at) as first, MAX(created_at) as last
  1096→   FROM point_events
  1097→   GROUP BY user_id
  1098→   HAVING COUNT(*) > 50
  1099→   ORDER BY count DESC;
  1100→
  1101→   -- Find submissions from the same device ID
  1102→   SELECT details->>'clientDevice'->>'deviceId' as device_id, COUNT(*) as count
  1103→   FROM point_events
  1104→   WHERE details->'clientDevice'->>'deviceId' IS NOT NULL
  1105→   GROUP BY details->'clientDevice'->>'deviceId'
  1106→   HAVING COUNT(*) > 20;
  1107→   ```
  1108→2. Quarantine suspicious submissions (add a `status: "quarantined"` field via ENRICH_EVENT)
  1109→3. Block the offending user accounts
  1110→4. Assess data quality impact on projected points
  1111→5. If photos were uploaded, check for duplicate image hashes across submissions
  1112→6. Remove fraudulent data after confirmation
  1113→7. Recalculate XP and leaderboard rankings
  1114→8. Implement rate limiting and enhanced fraud detection to prevent recurrence
  1115→
  1116→#### Playbook E: Admin Account Compromise
  1117→
  1118→**Trigger:** Suspected unauthorized admin access.
  1119→
  1120→**Steps:**
  1121→1. IMMEDIATELY:
  1122→   - Change `ADMIN_EMAIL` to a new address in Vercel env vars
  1123→   - Rotate `AUTH_SECRET` to invalidate all sessions
  1124→   - Change `ADMIN_PASSWORD` to a new bcrypt hash
  1125→2. Audit admin actions:
  1126→   - Review all `DELETE` operations on point_events
  1127→   - Check for unauthorized user profile modifications
  1128→   - Review admin_events access logs (if logging is implemented)
  1129→3. Check for data exfiltration:
  1130→   - Admin can view all events including user identifiers
  1131→   - Admin can view fraud check metadata including IP locations
  1132→4. Restore any deleted events from database backups
  1133→5. Post-incident: Implement admin audit logging, MFA for admin accounts
