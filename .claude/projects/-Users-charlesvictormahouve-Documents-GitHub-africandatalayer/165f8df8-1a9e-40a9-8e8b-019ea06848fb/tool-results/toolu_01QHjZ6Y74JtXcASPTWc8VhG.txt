    23→## 1. Architecture Vision & Principles
    24→
    25→### 1.1 Vision Statement
    26→
    27→ADL's architecture must serve contributors on low-end Android devices over 2G/3G networks in Douala, Lagos, Nairobi, and eventually every major African city. Every architectural decision is filtered through three lenses: **does it work offline?**, **does it work on a $50 phone with 1 GB RAM?**, and **does it survive 300 ms+ round-trip latency to the nearest PoP?**
    28→
    29→### 1.2 Core Principles
    30→
    31→| # | Principle | Rationale |
    32→|---|-----------|-----------|
    33→| P1 | **Serverless-first** | Zero idle cost, automatic scaling, no ops team required at MVP stage. Current stack (Vercel Functions + Supabase) already follows this. |
    34→| P2 | **Offline-first is non-negotiable** | Contributors in Bonamoussadi and beyond face intermittent connectivity. The existing `lib/client/offlineQueue.ts` IndexedDB queue with exponential backoff already implements this; architecture must never regress it. |
    35→| P3 | **Africa-optimized edge delivery** | Vercel's edge network has PoPs in Johannesburg, Nairobi, and Cairo. Static assets and public API reads (`s-maxage=30, stale-while-revalidate=300` as in `api/submissions/index.ts`) must always be served from edge. |
    36→| P4 | **Append-only event sourcing** | `point_events` table is the source of truth, projected into `ProjectedPoint` views via `lib/server/pointProjection.ts`. Never mutate raw events; always project. This enables audit trails and replay. |
    37→| P5 | **Minimal egress, maximal caching** | African mobile data is expensive. Compress aggressively, cache at edge, serve delta updates where possible. |
    38→| P6 | **Cost proportional to usage** | Startup economics: $0/mo at zero traffic, linear scaling. No fixed-cost infrastructure until 10K+ users force it. |
    39→| P7 | **Data sovereignty by default** | User data stays in the continent. Supabase project in EU (closest available) now; migrate to Africa-region when available. |
    40→| P8 | **Progressive complexity** | Start with Vercel + Supabase. Add components only when bottlenecks are measured, never speculated. |
    41→
    42→### 1.3 Non-Functional Requirements
    43→
    44→| Metric | MVP (now) | Phase 2 (10K) | Phase 3 (100K) |
    45→|--------|-----------|---------------|----------------|
    46→| API p95 latency (Africa) | < 2s | < 800ms | < 400ms |
    47→| Offline submission queue max | 50 items | 200 items | 500 items |
    48→| Photo upload size cap | 8 MB (`MAX_SUBMISSION_IMAGE_BYTES`) | 8 MB | 5 MB (auto-compress) |
    49→| Uptime SLA | 99% | 99.5% | 99.9% |
    50→| RTO / RPO | 4h / 24h | 1h / 1h | 15min / 5min |

   191→## 3. Environments
   192→
   193→### 3.1 Environment Strategy
   194→
   195→| Environment | Branch | URL Pattern | Database | Purpose |
   196→|-------------|--------|-------------|----------|---------|
   197→| **Production** | `main` | `africandatalayer.vercel.app` | Supabase `prod` project | Live users, real data |
   198→| **Staging** | `staging` | `staging-africandatalayer.vercel.app` | Supabase `staging` project (separate) | Pre-release validation, seed data |
   199→| **Development** | `dev` | `dev-africandatalayer.vercel.app` | Supabase `dev` project (separate) | Daily development, synthetic data |
   200→| **Preview** | PR branches | `<branch>-africandatalayer.vercel.app` | Shared `staging` database (read-only mode) | PR review, visual diffing |
   201→
   202→### 3.2 Environment Parity
   203→
   204→**Current gap:** There is only one Supabase project. All environments hit the same database.
   205→
   206→**Recommended changes:**
   207→
   208→1. **Create three Supabase projects:** `adl-prod`, `adl-staging`, `adl-dev`
   209→2. **Environment variables per Vercel environment:**
   210→   - `POSTGRES_URL` -> different for each environment
   211→   - `BLOB_READ_WRITE_TOKEN` -> separate blob stores
   212→   - `AUTH_SECRET` -> unique per environment
   213→   - `EDGE_CONFIG` -> separate Edge Config stores
   214→3. **Migration parity:** Use `supabase/migrations/` directory (already has `20260219_init_adl.sql` and `20260224_phone_identifier.sql`) with Supabase CLI `db push` in CI:
   215→   ```bash
   216→   # CI pipeline per environment
   217→   supabase db push --db-url $POSTGRES_URL
   218→   ```
   219→
   220→### 3.3 Data Isolation
   221→
   222→| Rule | Implementation |
   223→|------|---------------|
   224→| Production data never in staging | Separate Supabase projects with separate connection strings |
   225→| Staging uses realistic synthetic data | Seed script `scripts/seed-edge-submissions.mjs` extended with Faker.js |
   226→| Preview deployments are read-only | `DATA_STORE_DRIVER=postgres` with a read-only Postgres role |
   227→| Secrets never shared across envs | Vercel Environment Variables scoped to Production/Preview/Development |
   228→
   229→### 3.4 Feature Flags
   230→
   231→**Current state:** Map scope is gated by `user_profiles.map_scope` column (`bonamoussadi`, `cameroon`, `global`). Admin-only features gated by `is_admin` boolean.
   232→
   233→**Recommended approach:**
   234→
   235→Use Vercel Edge Config for feature flags (already a dependency: `@vercel/edge-config`):
   236→
   237→```json
   238→{
   239→  "feature_flags": {
   240→    "phone_auth_enabled": false,
   241→    "cameroon_expansion": true,
   242→    "realtime_leaderboard": false,
   243→    "gemini_search_enabled": true,
   244→    "max_submission_image_bytes": 8388608
   245→  }
   246→}
   247→```
   248→
   249→This avoids adding another service and leverages what is already installed. Edge Config reads are sub-millisecond at the edge.

   259→| PoP Location | Target Market | Latency to Douala |
   260→|--------------|---------------|-------------------|
   261→| Johannesburg (JNB) | Southern Africa | ~120ms |
   262→| Nairobi (NBO) | East Africa | ~100ms |
   263→| Cairo (CAI) | North Africa | ~150ms |
   264→| Paris (CDG) | Francophone fallback | ~180ms |
   265→| London (LHR) | Anglophone fallback | ~170ms |
   266→
   267→**Caching strategy for African users:**
   268→
   269→```
   270→Static assets (JS/CSS/images):
   271→  Cache-Control: public, max-age=31536000, immutable
   272→  (Vite content-hashed filenames: index-QrY-Hdyj.js)
   273→
   274→Public API reads (/api/submissions GET, unauthenticated):
   275→  Cache-Control: public, s-maxage=30, stale-while-revalidate=300
   276→  (Already implemented in api/submissions/index.ts)
   277→
   278→Authenticated API reads:
   279→  Cache-Control: private, no-cache
   280→  (Must revalidate per user)
   281→
   282→Photo blobs (Vercel Blob):
   283→  Cache-Control: public, max-age=86400
   284→  (Photos are immutable once uploaded)
   285→```
   286→
   287→### 4.2 API Routing and Edge Middleware
   288→
   289→**Current:** All API routes go through Vercel Functions (Node.js runtime). No edge middleware.
   290→
   291→**Recommended additions:**
   292→
   293→1. **Rate limiting middleware** (Edge Middleware at `middleware.ts`):
   294→   ```
   295→   /api/submissions POST -> 10 requests/minute per IP
   296→   /api/auth/register POST -> 5 requests/minute per IP
   297→   /api/auth/signin POST -> 10 requests/minute per IP
   298→   ```
   299→
   300→2. **Geo-routing middleware:**
   301→   - Read `request.geo.country` from Vercel Edge
   302→   - Set `X-ADL-Country` header for downstream functions
   303→   - Block requests from sanctioned countries
   304→   - Route to nearest data source when multi-region is implemented
   305→
   306→3. **Auth pre-check at edge:**
   307→   - Validate JWT signature at edge (fast rejection of expired/malformed tokens)
   308→   - Current `lib/auth.ts` does full JWT decode in the function; move signature check to edge
   309→
   310→### 4.3 Real-Time Strategy
   311→
   312→**Current:** No real-time. All reads are polling-based.
   313→
   314→**Recommended: Supabase Realtime (WebSocket)**
   315→
   316→Supabase Realtime uses PostgreSQL's logical replication to push changes over WebSocket:
   317→
   318→```
   319→Use cases:
   320→1. Admin review queue: New submissions appear instantly
   321→2. Leaderboard: XP changes propagate in real-time
   322→3. Contributor notifications: "Your submission was approved"
   323→
   324→Channel structure:
   325→- adl:submissions:{scope} -> New point_events for a map scope
   326→- adl:leaderboard -> XP changes on user_profiles
   327→- adl:admin -> Admin-targeted events
   328→
   329→Fallback: If WebSocket unavailable (aggressive firewalls in some
   330→African networks), fall back to 30-second polling (current behavior).
   331→```
   332→
   333→### 4.4 Offline-First Sync Protocol
   334→
   335→**Current implementation is solid.** Key components:
   336→
   337→```
   338→1. ENQUEUE: User submits -> lib/client/offlineQueue.ts enqueueSubmission()
   339→   - Generates idempotency key (crypto.randomUUID)
   340→   - Stores in IndexedDB 'adl_offline_queue' store
   341→   - Status: 'pending'
   342→
   343→2. FLUSH: Online detected -> App.tsx useEffect triggers flushOfflineQueue()
   344→   - Uses requestIdleCallback (graceful on low-end devices)
   345→   - Iterates pending items, respects nextRetryAt (backoff)
   346→   - Sends via lib/client/submissionSync.ts sendSubmissionPayload()
   347→   - X-Idempotency-Key header prevents duplicates
   348→
   349→3. BACKOFF: Failed sync -> exponential backoff
   350→   - baseDelay = min(30000, 1000 * 2^retryCount)
   351→   - jitter = random * 1000
   352→   - Max 30s between retries
   353→
   354→4. PERMANENT FAILURE: Non-retryable errors -> SyncErrorRecord
   355→   - Stored in 'submission_sync_errors' IndexedDB store
   356→   - Surfaced to user via getQueueStats()
   357→```
   358→
   359→**Infrastructure-level enhancements needed:**
   360→
   361→| Enhancement | Description | Priority |
   362→|-------------|-------------|----------|
   363→| **Server-side idempotency** | Check `X-Idempotency-Key` against a Redis/Postgres dedup table before inserting `point_events`. Currently the header is sent but not checked server-side. | HIGH |
   364→| **Conflict resolution** | When two contributors submit CREATE_EVENT for the same physical location, merge into one `point_id`. Currently creates duplicates. | MEDIUM |
   365→| **Delta sync** | Return only events since `?since=<timestamp>` instead of full event list. `getPointEvents()` in `postgresStore.ts` fetches ALL events with `order by created_at asc` -- unsustainable at scale. | HIGH |
   366→| **Background sync service worker** | Register a SW that flushes the queue even when the app tab is closed. Current sync only happens when App component mounts. | MEDIUM |

   370→## 5. Data Architecture
   371→
   372→### 5.1 PostgreSQL Scaling Path
   373→
   374→**Current schema** (from `supabase/migrations/20260219_init_adl.sql`):
   375→
   376→```sql
   377→-- Two tables, both in public schema
   378→user_profiles (
   379→  id text PK,
   380→  email text UNIQUE NOT NULL,
   381→  phone text,              -- Added in 20260224 migration
   382→  name text NOT NULL,
   383→  image text DEFAULT '',
   384→  occupation text DEFAULT '',
   385→  xp integer DEFAULT 0 CHECK (xp >= 0),
   386→  password_hash text,
   387→  is_admin boolean DEFAULT false,
   388→  map_scope text DEFAULT 'bonamoussadi',
   389→  created_at timestamptz,
   390→  updated_at timestamptz
   391→)
   392→
   393→point_events (
   394→  id uuid PK,
   395→  point_id text NOT NULL,
   396→  event_type text CHECK ('CREATE_EVENT', 'ENRICH_EVENT'),
   397→  user_id text NOT NULL,
   398→  category text CHECK ('pharmacy', 'fuel_station', 'mobile_money'),
   399→  latitude double precision NOT NULL,
   400→  longitude double precision NOT NULL,
   401→  details jsonb DEFAULT '{}',
   402→  photo_url text,
   403→  created_at timestamptz NOT NULL,
   404→  source text,
   405→  external_id text
   406→)
   407→
   408→Indexes:
   409→  idx_point_events_created_at_desc
   410→  idx_point_events_point_id_created_at_desc
   411→  idx_point_events_user_id_created_at_desc
   412→  idx_point_events_external_id
   413→```
   414→
   415→**Scaling path:**
   416→
   417→| Scale | Action | Rationale |
   418→|-------|--------|-----------|
   419→| **Now (< 1K events)** | No changes needed | Current `pg.Pool({ max: 5 })` handles MVP load fine |
   420→| **1K-10K events** | Add `point_projections` materialized view | Stop projecting in application code (`projectPointsFromEvents()` scans ALL events per request) |
   421→| **10K-50K events** | Partition `point_events` by month (`created_at`) | Queries with time range filters become partition-pruned |
   422→| **50K+ events** | Supabase read replica for GET requests | Separate read/write pools. `lib/server/db.ts` gets `getReadPool()` |
   423→| **100K+ events** | PostGIS extension for spatial queries | Replace application-level `isWithinBonamoussadi()` bounding-box checks with `ST_Within()` |
   424→| **500K+ events** | Connection pooling via Supabase PgBouncer (Transaction mode) | Current `POSTGRES_PRISMA_URL` in `.env.example` suggests this is already anticipated |
   425→
   426→**Critical optimization -- materialized projection:**
   427→
   428→```sql
   429→-- Replace in-app projectPointsFromEvents() with a materialized view
   430→CREATE MATERIALIZED VIEW point_projections AS
   431→SELECT DISTINCT ON (point_id)
   432→  point_id,
   433→  category,
   434→  latitude,
   435→  longitude,
   436→  details,
   437→  photo_url,
   438→  created_at,
   439→  source,
   440→  external_id
   441→FROM point_events
   442→ORDER BY point_id, created_at DESC;
   443→
   444→-- Refresh on write (via trigger or Supabase Edge Function)
   445→CREATE UNIQUE INDEX ON point_projections (point_id);
   446→```
   447→
   448→This eliminates the O(N) full-table scan that `getPointEvents()` + `projectPointsFromEvents()` currently performs on every GET request.
   449→
   450→### 5.2 Blob Storage Lifecycle
   451→
   452→**Current:** Photos uploaded to Vercel Blob via `@vercel/blob` `put()` in `api/submissions/index.ts`:
   453→```
   454→Path pattern: submissions/{eventId}-{timestamp}.{ext}
   455→Access: public
   456→Max size: 8 MB (MAX_SUBMISSION_IMAGE_BYTES)
   457→Formats: JPEG, PNG, WebP, HEIC, HEIF
   458→```
   459→
   460→**Lifecycle policy needed:**
   461→
   462→| Age | Action | Storage |
   463→|-----|--------|---------|
   464→| 0-30 days | Original quality | Vercel Blob (hot) |
   465→| 30-180 days | Compressed to WebP 80% quality | Vercel Blob |
   466→| 180+ days | Thumbnail only (200px), original archived | Vercel Blob (thumbnail) + Cloudflare R2 (archive) |
   467→| Deleted submissions | Hard delete after 30-day grace | N/A |
   468→
   469→**Photo optimization pipeline (Phase 2):**
   470→
   471→```
   472→Upload -> Vercel Blob (original)
   473→       -> Supabase Edge Function (async trigger)
   474→          -> Generate 200px thumbnail
   475→          -> Generate 800px preview
   476→          -> Extract EXIF (already done in submissionFraud.ts)
   477→          -> Store variants in Blob
   478→          -> Update point_events.details with variant URLs
   479→```
   480→
   481→### 5.3 Event Streaming / Message Queue
   482→
   483→**Current gap:** No server-side async processing. Everything is synchronous in the request path:
   484→- Photo upload (Vercel Blob PUT)
   485→- EXIF extraction (`extractPhotoMetadata()`)
   486→- IP geolocation (`fetchIpLocation()` calls ipapi.co)
   487→- Fraud check computation (`buildSubmissionFraudCheck()`)
   488→- XP update (`upsertUserProfile()`)
   489→
   490→This makes POST `/api/submissions` slow and fragile.
   491→
   492→**Phase 2 architecture -- async processing via QStash:**
   493→
   494→```
   495→POST /api/submissions (synchronous path):
   496→  1. Validate input
   497→  2. Upload photo to Blob
   498→  3. Insert point_event with status='pending_review'
   499→  4. Return 201 immediately
   500→
   501→QStash message -> /api/internal/process-submission:
   502→  1. Extract EXIF metadata
   503→  2. Fetch IP geolocation
   504→  3. Compute fraud check
   505→  4. Update point_event.details with fraud data
   506→  5. Update XP
   507→  6. Notify admin if fraud score > threshold
   508→```
   509→
   510→**Why QStash over alternatives:**
   511→- Native Vercel integration (no infrastructure to manage)
   512→- Built-in retry with exponential backoff
   513→- HTTP-based (works with existing serverless functions)
   514→- Free tier covers MVP needs (500 messages/day)

   516→### 5.4 Data Residency and Sovereignty
   517→
   518→**African data protection landscape:**
   519→
   520→| Country | Law | Key Requirement |
   521→|---------|-----|-----------------|
   522→| **Cameroon** | Law No. 2010/012 on Cybersecurity and Cybercrime | Personal data must be processed lawfully; DPA notification required |
   523→| **Nigeria** | NDPA 2023 | Data controller must register; cross-border transfer requires adequacy |
   524→| **Kenya** | Data Protection Act 2019 | Consent-based; data transfers need adequate safeguards |
   525→| **South Africa** | POPIA | Most mature; requires data processing agreements, information officer |
   526→| **AU-wide** | Malabo Convention (2014, entered into force 2023) | Framework for harmonized data protection across AU member states |
   527→
   528→**ADL's data residency strategy:**
   529→
   530→1. **Current:** Supabase project in EU region (closest available with full feature set)
   531→2. **Phase 2:** When Supabase launches Africa regions (anticipated), migrate to Johannesburg or Nairobi
   532→3. **Phase 3:** If Supabase Africa region unavailable, self-managed PostgreSQL on AWS Africa (Cape Town `af-south-1`) or GCP Johannesburg
   533→4. **Always:** Store GPS coordinates and photos (which contain location data) under the same residency rules as PII. The `point_events.details` JSONB contains `fraudCheck.ipLocation` which is indirectly PII.
   534→
   535→**Cross-border data flow:** ADL collects data in Cameroon but serves it to API consumers potentially worldwide. Solution:
   536→- Raw PII (email, phone, GPS) stays in primary database region
   537→- Anonymized/aggregated data (point projections without `user_id`) can be replicated globally
   538→- `redactEventUserIds()` in `lib/server/submissionAccess.ts` already handles this for unauthenticated reads

   571→### 6.2 Secrets Management
   572→
   573→**Current state (.env.example):**
   574→
   575→```
   576→AUTH_SECRET=                    # Session encryption
   577→GOOGLE_CLIENT_ID=               # OAuth
   578→GOOGLE_CLIENT_SECRET=           # OAuth
   579→ADL_POSTGRES_URL=               # Database (contains password)
   580→BLOB_READ_WRITE_TOKEN=          # Vercel Blob access
   581→EDGE_CONFIG=                    # Edge Config connection
   582→VERCEL_API_TOKEN=               # Vercel API (dangerous!)
   583→ADMIN_EMAIL=                    # Hardcoded admin
   584→ADMIN_PASSWORD=                 # Hardcoded admin password!
   585→```
   586→
   587→**Security concerns:**
   588→
   589→1. **CRITICAL: `ADMIN_PASSWORD` in environment variables.** This is a plaintext admin password. Must be replaced with hashed credentials in the database (already partially done: `user_profiles.password_hash` column exists, `bcryptjs` is a dependency).
   590→
   591→2. **HIGH: Gemini API key exposed client-side.** `vite.config.ts` defines `process.env.API_KEY` which is bundled into the client JavaScript. Must be proxied through a server-side API route.
   592→
   593→3. **MEDIUM: `VERCEL_API_TOKEN` in env.** This grants full Vercel account access. Should only be in CI/CD, never in runtime functions.
   594→
   595→**Recommended secrets architecture:**
   596→
   597→| Secret | Storage | Access Pattern |
   598→|--------|---------|----------------|
   599→| `AUTH_SECRET` | Vercel Environment Variables (encrypted) | Runtime, per-environment |
   600→| `POSTGRES_URL` | Vercel Environment Variables (encrypted) | Runtime, per-environment |
   601→| `BLOB_READ_WRITE_TOKEN` | Vercel Environment Variables (encrypted) | Runtime, per-environment |
   602→| `GOOGLE_CLIENT_*` | Vercel Environment Variables (encrypted) | Runtime, shared staging+prod |
   603→| `GEMINI_API_KEY` | Vercel Environment Variables (server only) | Server-side functions only, never in `VITE_` prefix |
   604→| `VERCEL_API_TOKEN` | GitHub Actions Secrets only | CI/CD only, never runtime |
   605→| Admin credentials | `user_profiles.password_hash` (bcrypt) | Database only, no env vars |
