# ROSE-SEIN Service Delivery and Operations

## Service Delivery Principle

ROSE-SEIN is not only a software product. It is also a support operating environment for the association. The service model must therefore define how humans work through the platform, not just how screens behave.

## Operational Roles

### Association admin

- owns content publishing and institutional communication
- manages events and key announcements
- oversees service quality

### Moderator

- reviews reports
- enforces community and messaging safety rules
- handles sensitive interactions according to policy

### Volunteer

- participates only within explicitly allowed workflows
- does not inherit moderator or admin access by default

### Support operator

- handles technical and service support questions
- routes escalations

## Core Operational Flows

### Content publishing

1. draft content created
2. content validated
3. publish decision made
4. content scheduled or released

### Moderation

1. content or conversation reported
2. queue reviewed by moderator
3. action recorded
4. escalation to admin where required

### Support

1. user reaches help or support surface
2. issue categorized
3. response owner assigned
4. issue resolved or escalated

### Release communication

1. release scope approved
2. operational teams informed
3. release checklist completed
4. post-release monitoring period completed

## Support and Escalation Model

### Level 1

- FAQ
- simple technical guidance
- standard account and navigation issues

### Level 2

- moderation review
- role and access corrections
- sensitive user concerns requiring staff visibility

### Level 3

- platform incidents
- security or privacy issues
- legal or compliance escalations

## Recommended Service Expectations

Suggested internal expectations for later formalization:

- moderation queue reviewed daily
- critical safety reports reviewed urgently
- association-managed message response targets defined by business hours
- release owners explicitly assigned before production changes

## Operating Artifacts Needed

The platform should eventually maintain:

- moderator handbook
- support handbook
- release communication templates
- escalation matrix
- incident contact list

## Service Readiness Checklist

Before a major user-facing feature ships, confirm:

- support owner exists
- moderation owner exists if user-generated content is involved
- escalation path is documented
- rollback plan exists
- release communication is prepared
