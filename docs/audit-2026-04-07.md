# ROSE-SEIN — Interface Quality Audit

**Date**: 2026-04-07  
**Auditor**: Impeccable Audit (frontend-design + a11y + performance + theming + responsive)  
**Scope**: Full codebase — `app/`, `components/`, `globals.css`, `tailwind.config.ts`, `middleware.ts`  
**Standards**: WCAG 2.1 AA, HTML spec, design anti-pattern analysis

---

## Anti-Patterns Verdict

**Result: PASS on brand direction. FAIL on execution monotony.**

The design is **not AI slop**. The brand direction is clear and intentional — warm rose palette, editorial typography (when it loads), tonal hierarchy without hard borders, genuine warmth. The token system is coherent. This is not generic.

However, the implementation has **converged on a single repeated motif** across every page:

| Anti-pattern | Instances |
|---|---|
| `h-12 w-12 rounded-full bg-primary/10` icon circle before every section heading | ~35 across all pages |
| `icon + heading + description + ArrowRight` card grid | Shortcuts, soins, admin, aide, association, events — every page |
| `border-l-4 border-primary/30` or `border-l-4 border-secondary-container` as accent | `app/page.tsx:141, 173`; `article-content.tsx:37` (blockquote) |
| Nested `surface-card` inside `surface-section` | Homepage hero section, news filters |
| `bg-gradient-primary` primary CTA button | 15+ instances — appears on nearly every form submit and primary action |

These patterns are not wrong individually. At 35× repetition, they become visual wallpaper. Every page looks structurally identical. The "digital sanctuary" brand goal requires spatial variety, not a single motif tiled across 15 screens.

One specific **developer-facing content visible to end users**: `app/page.tsx:381–403` renders an "Architecture produit" section explaining Next.js, Supabase, and technical roadmap items to cancer patients and caregivers. This must be removed before production.

### ██ CRITICAL

#### C1 — Brand fonts never load

- **Location**: `app/head.tsx`, `app/layout.tsx`, `tailwind.config.ts`
- **Category**: Performance / Theming
- **Description**: `app/head.tsx` uses the Next.js Pages Router `<Head>` API (`app/head.tsx` was deprecated in Next.js 13.2 and is silently ignored in App Router). The Google Fonts `<link>` tags it contains are **never rendered**. The Tailwind config references `var(--font-plus-jakarta-sans)` and `var(--font-be-vietnam-pro)` — CSS variables that are **never defined** anywhere. Every `font-headline`, `font-body`, and `font-label` Tailwind class silently falls back to `system-ui, sans-serif`.
- **Impact**: The entire brand typography — the primary design tool per CLAUDE.md — is absent for all users. Plus Jakarta Sans headlines and Be Vietnam Pro body text are never displayed.
- **WCAG/Standard**: N/A (brand/product)
- **Recommendation**: Delete `app/head.tsx`. In `app/layout.tsx`, import both fonts via `next/font/google` with the `variable` option to emit CSS variables. Apply the resulting class strings to the `<html>` element.
- **Suggested command**: `/normalize`

#### C2 — Open redirect vulnerability

- **Location**: `app/account/page.tsx:49, 225`
- **Category**: Security
- **Description**: `const redirectTo = firstValue(params.redirectTo) ?? "/messages"` reads a URL from query params and renders it directly as `<a href={redirectTo}>Continuer</a>`. An attacker can craft `https://app.rosesein.org/account?redirectTo=https://phishing.example.com` — a logged-in user would see a trusted ROSE-SEIN page with a "Continuer" button redirecting to a malicious site.
- **Impact**: Phishing attack vector against cancer patients and caregivers who have just authenticated. Critical given the vulnerability of the user base.
- **WCAG/Standard**: OWASP A01 — Broken Access Control
- **Recommendation**: Validate `redirectTo` to only accept internal relative paths: `const safeRedirect = /^\/[^/]/.test(redirectTo ?? '') ? redirectTo : '/messages'`.
- **Suggested command**: `/harden`

---

#### C3 — Invalid HTML: `<a>` wrapping `<li>`

- **Location**: `components/notifications/notification-bell.tsx:84–90`
- **Category**: Accessibility
- **Description**: The `inner` variable is a `<li>` element. When a notification has an `href`, it renders as `<a key={id}><li>...</li></a>` inside a `<ul>`. Anchor (`<a>`) is phrasing content — it cannot contain block-level elements like `<li>`. The `<a>` must be *inside* the `<li>`, not the other way around.
- **Impact**: Invalid DOM. Browsers may discard the `<li>` from the `<ul>` context, breaking the list entirely. Screen readers misparse the list. NVDA/JAWS may skip clickable items.
- **WCAG**: 1.3.1 Info and Relationships (Level A), 4.1.1 Parsing (Level A)
- **Recommendation**: Restructure so `<li>` is the outer element and `<a>` wraps the content inside it.
- **Suggested command**: `/harden`

#### C4 — Feedback messages have no live region announcement

- **Location**: `app/account/page.tsx:71`, `app/(protected)/parametres/page.tsx:59`, `app/(protected)/parcours/page.tsx:87`, `app/(protected)/communaute/[spaceSlug]/[threadId]/page.tsx:76`, `app/(protected)/messages/[threadId]/page.tsx:93`, `app/(protected)/messages/nouveau/page.tsx:51`
- **Category**: Accessibility
- **Description**: After form submissions, feedback divs are rendered with no `role="alert"`, `role="status"`, or `aria-live`. On a full server navigation, the entire page re-renders, but assistive technologies don't re-announce static content that wasn't in a live region on page load. No live regions exist anywhere in the codebase.
- **Impact**: Screen reader users who submit appointments, notes, profile updates, messages, or threads receive zero feedback — no confirmation of success or error.
- **WCAG**: 4.1.3 Status Messages (Level AA)
- **Recommendation**: Add `role="alert"` to error feedback and `role="status"` to success feedback divs on all 6 pages.
- **Suggested command**: `/harden`

#### C5 — News search input has no accessible label

- **Location**: `components/content/news-filters.tsx:43–57`
- **Category**: Accessibility
- **Description**: `<input type="search">` has a `placeholder` but no `<label>`, `aria-label`, or `aria-labelledby`. The visible `<Search>` icon is `aria-hidden`. There is no text-based label of any kind.
- **Impact**: Screen readers announce this as "edit, blank" or equivalent. Users cannot identify the field's purpose without trial-and-error.
- **WCAG**: 1.3.1 Info and Relationships (A), 4.1.2 Name, Role, Value (A)
- **Recommendation**: Add `aria-label="Rechercher dans les actualités"` to the input, or add a visually hidden `<label htmlFor="news-search">`.
- **Suggested command**: `/harden`

---

#### C6 — Raw `<img>` tag with no lazy loading and no dimensions

- **Location**: `components/content/article-content.tsx:32–38`
- **Category**: Performance / Accessibility
- **Description**: Article images use native `<img src={block.src} alt={block.alt}>` without lazy loading, width/height attributes, or `next/image`. This causes: (1) images load eagerly regardless of viewport position, (2) no dimensions = layout shift as the image loads (CLS), (3) no CDN optimization or WebP conversion, (4) `block.alt` is passed from CMS data — if empty, screen readers encounter a decorative image that is not `aria-hidden`.
- **Impact**: Core Web Vitals regression (CLS). Performance degradation on articles with multiple images. Potential silent a11y failure if CMS data has empty alt text.
- **WCAG**: 1.1.1 Non-text Content (A)
- **Recommendation**: Replace with `next/image` or add `loading="lazy"` + `width` + `height`. Guard against empty alt text with `alt={block.alt || ""}` (empty string means decorative) and add `aria-hidden="true"` when alt is empty.
- **Suggested command**: `/optimize`

### ██ HIGH

#### H1 — No `prefers-reduced-motion` support

- **Location**: `app/globals.css`, all components — 21 animated elements identified
- **Category**: Accessibility
- **Description**: No `@media (prefers-reduced-motion: reduce)` override exists anywhere. Animated elements include: `hover:-translate-y-0.5` (links, buttons), `hover:-translate-y-1` (mood buttons), `group-hover:translate-x-1` (arrow icons), `transition-transform duration-200` (nav items, conversation cards), `active:scale-95` (FAB button).
- **Impact**: Users with vestibular disorders, epilepsy, or motion sensitivity — groups whose overlap with cancer patients is clinically significant — experience unmitigated motion on every interaction. CLAUDE.md explicitly requires reduced motion support.
- **WCAG**: 2.3.3 Animation from Interactions (Level AAA); CLAUDE.md requirement
- **Recommendation**: Add to `globals.css`: `@media (prefers-reduced-motion: reduce) { *, *::before, *::after { transition-duration: 0.01ms !important; animation-duration: 0.01ms !important; } }`. Alternatively, add `motion-reduce:transition-none motion-reduce:transform-none` to each animated element.
- **Suggested command**: `/animate`

#### H2 — ARIA tab pattern structurally invalid

- **Location**: `components/chat/message-inbox.tsx:90–115`
- **Category**: Accessibility
- **Description**: Three violations in the "Association / Privé" scope switcher:
  1. `role="tablist"` is on an outer `<div>`, but `<button role="tab">` elements are inside an intermediate `<div className="grid">` — ARIA requires tabs to be owned by the tablist with no intervening elements that change the ownership model.
  2. No `aria-controls` attribute on tab buttons linking to their panel.
  3. The `<ConversationList>` below has no `role="tabpanel"`, no `id`, and no `aria-labelledby`.
- **Impact**: Screen readers do not treat this as a tab widget. Arrow key navigation (expected by keyboard users for tabs) is not implemented. AT users see two unlabelled buttons with no relationship to the content they control.
- **WCAG**: 4.1.2 Name, Role, Value (Level A)
- **Recommendation**: Remove the intermediate `<div className="grid">` between tablist and tabs (or move `role="tablist"` to the grid div). Add `aria-controls="tab-panel-{scope}"` to each tab. Add `id="tab-panel-{scope}" role="tabpanel" aria-labelledby="tab-{scope}"` to the ConversationList wrapper.
- **Suggested command**: `/harden`

#### H3 — Form inputs missing visible focus indicators

- **Location**: `app/(protected)/parcours/page.tsx` (all inputs), `app/(protected)/parametres/page.tsx` (all inputs), `app/account/page.tsx` (all inputs), `app/(protected)/communaute/[spaceSlug]/nouveau/page.tsx` (title + body inputs)
- **Category**: Accessibility
- **Description**: These inputs use `w-full rounded-brand bg-surface-container-high px-4 py-4 text-sm text-on-surface` with no `focus:` Tailwind utilities. The global `*:focus-visible` box-shadow is `rgba(178, 37, 79, 0.15)` — 15% opacity rose. On a light background, this ring is nearly invisible and does not meet the 3:1 contrast ratio required for focus indicators. The community thread textarea and admin inputs are the exception, correctly using `focus:border-primary focus:outline-none focus:ring-[3px]`.
- **Impact**: Keyboard users cannot tell which form field is active. WCAG 2.4.7 failure across all data-entry forms in the app.
- **WCAG**: 2.4.7 Focus Visible (AA), 2.4.11 Focus Appearance (AA, WCAG 2.2)
- **Recommendation**: Standardize all inputs with `focus:outline-none focus:ring-2 focus:ring-primary/40 focus:border-primary` (matching the pattern already used in admin inputs). Or raise the global `*:focus-visible` ring to at least 40% opacity.
- **Suggested command**: `/harden`

#### H4 — No skip-to-content link

- **Location**: `app/layout.tsx`, `components/shell/app-shell.tsx`
- **Category**: Accessibility
- **Description**: No "Aller au contenu principal" skip link exists. Keyboard users must tab through all TopAppBar elements (logo, notification bell, help button, settings button = 4 interactive elements) before reaching page content on every page load.

- **Suggested command**: `/harden`

#### H6 — Notification dropdown has no keyboard dismiss

- **Location**: `components/notifications/notification-bell.tsx`
- **Category**: Accessibility / UX
- **Description**: The notification panel uses `<details>/<summary>`. Once opened, it can only be dismissed by clicking the bell again. There is no `Escape` key handler, no outside-click detection, and no `inert` or `aria-modal` management. The `w-80` (320px) fixed-width panel at `right-0` can also overflow the viewport on screens narrower than 340px.
- **Impact**: Keyboard users may become trapped cycling through all notification items before reaching other content. On mobile, the panel obscures page content indefinitely once opened.
- **WCAG**: 2.1.2 No Keyboard Trap (Level A)
- **Recommendation**: Convert to a client component with controlled open state. Add `useEffect` for outside-click detection, `keydown` listener for `Escape`, `aria-expanded` on the trigger, and `aria-haspopup="dialog"`. Constrain width with `max-w-[calc(100vw-2rem)]` for narrow screens.
- **Suggested command**: `/harden`

#### H7 — FAQ items have no expand/collapse visual indicator

- **Location**: `app/aide/page.tsx:185–198`
- **Category**: Accessibility / UX
- **Description**: FAQ `<details>` elements have `list-none` implied by Tailwind's preflight reset, which removes the browser's native disclosure triangle. No custom chevron or indicator replaces it. Users see static-looking cards with no affordance that the content is expandable.
- **Impact**: The FAQ section appears non-interactive. For cognitively fatigued users (the target population), missing affordances cause frustration.

#### M4 — `router.refresh()` on every reaction toggle causes full server rerender

- **Location**: `components/community/reaction-bar.tsx:48`
- **Category**: Performance
- **Description**: Every reaction click calls `router.refresh()` after the server action, triggering a full re-fetch and re-render of the entire server component tree for the community space or thread page. With `force-dynamic` on those pages, this means a full database round-trip on every emoji tap.
- **Impact**: Noticeable latency on reactions. Expensive at scale. Optimistic UI is already implemented correctly — the refresh is redundant for the reaction state itself.
- **Recommendation**: Remove `router.refresh()`. The optimistic update (`useOptimistic`) already reflects the new state. The server action persists the change. A refresh is only needed to update counts from *other* users' reactions, which can be achieved with a debounced refresh or a Supabase realtime subscription instead.
- **Suggested command**: `/optimize`

#### M5 — No `loading.tsx` files — blank screen during navigation

#### M7 — `force-dynamic` on all protected pages disables all caching

- **Description**: All 8 protected pages set `export const dynamic = "force-dynamic"`, opting out of all ISR, static generation, and route-level caching. #### M8 — Inconsistent focus ring strength across the codebase

- **Location**: Throughout `app/` and `components/`
- **Category**: Theming / Accessibility
- **Description**: Focus rings are applied at different opacities with no consistent system:
  - Global: `rgba(178, 37, 79, 0.15)` (15%) — nearly invisible
  - Checkboxes in parametres/account/communaute: `focus:ring-primary/30` (30%)
  - Community thread textarea + admin inputs: `focus:ring-[3px] focus:ring-primary/15` (15%)
  - MemberPicker: `focus:ring-[3px] focus:ring-primary/15` (15%)
  - Anonymous checkbox in thread: `focus:ring-primary/20` (20%)
  
  None of these opacities reliably meet the 3:1 non-text contrast ratio required for WCAG 2.4.11.
- **Impact**: Inconsistency confuses users who rely on focus indicators. No individual value is accessible.
- **Recommendation**: Standardize on `focus:ring-2 focus:ring-primary focus:ring-offset-2` (solid, no opacity modifier) everywhere. Or define a CSS custom property `--ring-color: rgb(178 37 79 / 0.5)` for a consistent accessible value.
- **Suggested command**: `/normalize`

#### M9 — `WELLNESS_TIPS` rotation bug — half of tips unreachable

- **Location**: `app/page.tsx:118`
- **Category**: Correctness
- **Description**: `WELLNESS_TIPS[new Date().getDay() % WELLNESS_TIPS.length]`. `Date.getDay()` returns 0–6 (7 values). `WELLNESS_TIPS.length` is 14. `0 % 14 = 0`, `1 % 14 = 1`, ..., `6 % 14 = 6`. Tips at indices 7–13 are mathematically unreachable. Half the content never displays.
- **Recommendation**: Use day-of-year for full rotation: `const doy = Math.floor((Date.now() - +new Date(new Date().getFullYear(), 0, 0)) / 86400000); WELLNESS_TIPS[doy % WELLNESS_TIPS.length]`.
- **Suggested command**: `/harden`

- **Description**: Every page contains a `grid gap-4 sm:grid-cols-2` (or lg:grid-cols-3) of `surface-card` components with icon-circle + heading + description + arrow link. This exact layout structure appears on 6+ distinct pages with no variation.
- **Impact**: Users cannot distinguish between sections. Navigation feels like moving between identical rooms.
- **Recommendation**: Introduce layout variation between pages: lists on some, featured/editorial layouts on others, mixed sizes. The homepage shortcuts and the soins categories could look very different from each other.
- **Suggested command**: `/bolder`

#### M15 — Primary gradient CTA overused — hierarchy lost

- **Location**: 15+ form submit buttons across all pages
- **Category**: Anti-Pattern / Theming
- **Description**: `bg-gradient-primary` (the brand gradient from `#b2254f` to `#fc5e84`) is used on nearly every form submit button, the hero CTA, the new message FAB, the new thread button, push notification buttons, and mood check-in buttons. When everything is primary, nothing is.
- **Impact**: The gradient loses emphasis. The visual hierarchy that should guide users toward the most important action on each page is absent.
- **Recommendation**: Use `bg-gradient-primary` for at most one primary action per page. Secondary actions should use `bg-surface-container-lowest shadow-ambient text-primary` (already in the codebase — use more of it). Tertiary actions should be text links.

#### M18 — No i18n mechanism despite bilingual requirement

- **Location**: Every page and component file
- **Category**: CLAUDE.md Requirement
- **Description**: CLAUDE.md requires "No hardcoded French-only strings in components" and English support. The entire codebase has hardcoded French strings in every file. No i18n library, no message catalogs, no English variants exist.

## Patterns & Systemic Issues

1. **Icon-circle before every heading** — ~35 instances. Single design decision producing visual monotony across all 15+ pages. Requires a design-level decision, not spot-fixes.

2. **Focus indicators universally underweight** — Global ring at 15% opacity; checkboxes at 30%; admin/community inputs at 15% with 3px width. No single value meets WCAG 2.4.11. Requires a system-wide fix in `globals.css`.

3. **Live region gap** — No `role="alert"`, `role="status"`, or `aria-live` exists anywhere in the codebase. Feedback messages on 6+ pages, PushChannelManager status, and ReactionBar error all fail silently for screen reader users.

4. **Font loading completely broken** — All brand typography falls back to system fonts. Affects every page, every user.

5. **No loading/error boundaries** — Zero `loading.tsx` or `error.tsx` files. Every data-fetching route either shows a blank screen during load or a raw Next.js error page on failure.

6. **Hardcoded French strings in all components** — Violates the bilingual requirement in CLAUDE.md. Every shared component (`BottomNav`, `ConversationCard`, `CommunitySpaceCard`, `ResourceCard`, etc.) has hardcoded French text.

---

## Positive Findings

1. **Design token system is coherent and well-named** — Semantic naming (`on-surface`, `surface-container-low/high/lowest/highest`, `outline-variant`) is consistently used. Token coverage is thorough.

2. **`aria-current="page"` on BottomNav** — Correctly implemented with `aria-label` on the `<nav>` wrapper. One of the best ARIA implementations in the codebase.

3. **`aria-hidden="true"` on all decorative icons** — Applied consistently across every Lucide icon usage in every file. No missed instances found.

4. **`aria-label` on icon-only buttons** — TopAppBar help and settings buttons have proper accessible names.

5. **ReactionBar uses `aria-pressed`** — The reaction toggle buttons correctly use `aria-pressed={isActive}` — exactly the right pattern for toggle buttons.

6. **Optimistic UI in ReactionBar** — `useOptimistic` for reactions is a well-implemented performance pattern.

7. **Semantic HTML throughout** — Correct use of `<article>`, `<section>`, `<nav>`, `<header>`, `<main>`, `<time>`, `<blockquote>`, `<details>/<summary>` across the codebase.

8. **Difficile Day Mode** — A thoughtful, brand-aligned accessibility feature that reduces cognitive load. Well-conceived and correctly gated.

9. **`<html lang="fr">`** — Correctly set at root layout level.

10. **SearchField component** — The one correctly-labeled search input in the codebase (`htmlFor`/`id` association, `aria-hidden` on icon). This is the pattern to replicate everywhere.

11. **Server Actions everywhere** — No client-side data exposure. All mutations use Next.js Server Actions. Good security posture.

12. **Privacy-first architecture** — RLS on Supabase tables, profile setup required before data access, staff-gated admin routes.
